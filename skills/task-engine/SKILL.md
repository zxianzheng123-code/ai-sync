# 通用任务引擎（Task Engine）

## TL;DR
- **输入**：任意开放式任务描述
- **输出**：任务交付物（动态，由理解阶段确定）
- **性质**：通用循环编排器——理解→规划→拆解→执行→检查→决策，循环至完成
- **关键卖点**：一个引擎跑所有非专项任务，角色动态组建，上下文三层管理

## Goal
对于不映射到任何现有专项 Skill 的开放式任务，提供一个通用的多步骤执行框架。自动理解需求、拆解计划、逐步执行、质量检查、循环迭代直到完成。

## 角色设定

### 主控（始终是主 agent 自身）

| 角色 | 职责 |
|------|------|
| 需求对齐官 | 对齐客户需求。大白话沟通，确保每一步不偏离用户意图。所有面向用户的输出经此角色把关 |
| 执行计划师 | 驱动全循环。拆解任务、编排计划、管理上下文（W/S/L）、控制循环终止、分派 subagent |

主控不是 subagent，是主 agent 的两个职能视角。

### 动态 Subagent（Step 1 后定义，按需 spawn）

```
Step 1 完成任务定义后，执行计划师根据任务类型定义 1-3 个 subagent 规格。
每个 subagent 规格必须明确：
  - 名称（简短，≤6 字）
  - 职责（1 句话）
  - 触发时机（哪些 Step 中 spawn）
  - 输入/输出契约（给它什么、要它返回什么）

subagent 规格写入 W.subagents。

spawn 规则：
  - subagent 是一次性的：spawn → 执行 → 返回结果 → 销毁
  - 同一规格可在不同 Step 多次 spawn（每次都是新实例）
  - subagent 不持有 W/S/L，由主控将必要上下文注入 prompt
  - subagent 返回结果后，由主控整合到 W

示例：
  研究类任务 → 信源验证 subagent（Step 4/6 spawn，输入: 信息片段，输出: 可靠性评级+证据）
  创作类任务 → 风格审查 subagent（Step 6 spawn，输入: 产出文本+风格要求，输出: 问题清单）
  技术类任务 → 可行性评估 subagent（Step 2/6 spawn，输入: 方案描述，输出: 可行/不可行+原因）
  分析类任务 → 逻辑审查 subagent（Step 6 spawn，输入: 推理链，输出: 漏洞清单）
```

## 上下文管理

三层上下文贯穿全循环：

### W（工作上下文）— 临时，单任务生命周期

所有 Step 共享读写。任务结束后消亡。

```
W = {
  task_input:      用户原文
  task_definition: { 目标, 类型, 约束, 成功标志, 已知, 未知 }
  subagents:       动态 subagent 规格列表
  plan:            { subtasks: [...], current_index: 0 }
  current_subtask: { definition, method, context, result }
  accumulated:     已完成子任务的结果汇总
  iteration:       当前循环轮次（从 0 开始）
  max_iterations:  5
}
```

### S（稳定上下文）— 持久，跨任务

仅在 Step 7（决策）和 Step 8（输出）时更新。初版为会话级，不落文件。

```
S = {
  用户偏好、反复出现的模式、领域知识、历史任务摘要
}
```

### L（决策记录）— 追溯，防重复犯错

每次 Step 7 决策写一条。

```
L = [
  { iteration: N, decision: "完成|澄清|重规划", reason: "...", timestamp: "..." }
]
```

## Strict Rules
1. **理解先于行动**：未完成 Step 1 不得进入 Step 2。模糊任务必须先澄清。
2. **计划先于执行**：未经审查的计划不得执行。
3. **单任务单循环**：同一时间只处理 1 个任务。新任务进来时先完成或挂起当前任务。
4. **循环有上限**：默认最大 5 轮（理解→…→决策 为 1 轮）。超限必须与用户确认是否继续。
5. **决策必留痕**：每次 Step 7 决策必须写入 L，含迭代编号、决策内容、原因。
6. **不脑补不猜**：信息不足时只问不猜。用户未说的不擅自填充。
7. **大白话沟通**：所有面向用户的输出由需求对齐官把关，禁用术语（除非用户先使用）。
8. **交互最小化**：能自主完成的步骤不打断用户。交互只在 Step 1（澄清）、Step 2b（计划确认）、Step 5（里程碑汇报）、Step 7（决策确认）触发。
9. **不越权**：不修改已有文件、不执行危险操作（删除/覆盖/发送），除非用户明确授权。
10. **结果可追溯**：最终输出中的每个结论必须可追溯到执行步骤中的具体证据。

## Workflow

### Step 0: 摄入（执行计划师，不面向用户）

```
IF 用户提供了明确任务描述:
  初始化 W = { task_input: 用户原文, iteration: 0, max_iterations: 5 }
  初始化 L = []
  进入 Step 1
IF 用户描述模糊/不完整:
  [需求对齐官] 提问:
  > 老板，您想让我帮您做什么？能具体说一下吗？
  等待回复 → 初始化 W → 进入 Step 1
```

### Step 1: 理解（执行计划师 + 需求对齐官，条件性面向用户）

**目标**：将用户输入转化为明确的任务定义 + 动态组建角色。

```
1. 执行计划师从 W.task_input 提取:
   - 任务目标: 用户最终想得到什么？
   - 任务类型: 研究 / 分析 / 创作 / 规划 / 执行 / 综合
   - 约束条件: 时间、格式、范围、质量标准
   - 成功标志: 什么情况下算"完成"？
   - 已知信息: 用户已提供的素材/上下文
   - 未知信息: 需要澄清或获取的内容

2. IF 信息不足（缺 ≥2 项关键信息）:
     [需求对齐官] 大白话提问（只问最关键的 1 个点，自带提案）:
     > 老板，我理解您想要 [提案]。[1 个问题]？
     等待回复 → 更新 W → 重新 Step 1

3. 信息充分后:
   W.task_definition = { 目标, 类型, 约束, 成功标志, 已知, 未知 }

4. 定义动态 subagent 规格:
   执行计划师根据 W.task_definition.类型 定义 1-3 个 subagent 规格。
   每个规格: { 名称, 职责, 触发时机, 输入输出契约 }
   W.subagents = [规格列表]

5. 进入 Step 2
```

**交互模块触点**：信息不足时触发用户交互。

### Step 2: 规划（执行计划师，面向用户确认）

**目标**：将任务定义拆解为有序的执行计划。

```
1. 执行计划师根据 W.task_definition 设计执行计划:
   - 列出所有子任务
   - 确定子任务之间的依赖关系（拓扑排序）
   - 估算每个子任务的复杂度（简单 / 中等 / 复杂）
   - 标记哪些子任务需要用户输入，哪些可自主执行

2. IF W.subagents 中有触发时机含 Step 2 的规格:
     spawn 对应 subagent，注入计划内容，要求审查:
     - 子任务是否覆盖目标？（无遗漏）
     - 子任务之间是否有冗余？
     - 依赖关系是否正确？
     subagent 返回审查结论 → 主控整合
     ├─ 通过 → 进入 2b
     └─ 不通过 → 执行计划师修改 → 重新 spawn 审查

2b. [需求对齐官] 向用户汇报计划摘要:
   > 老板，我打算这样做:
   > 1. [子任务1简述]
   > 2. [子任务2简述]
   > ...
   > 预计 [N] 步完成。开始吗？
   ├─ 用户确认 → W.plan = { subtasks: [...], current_index: 0 } → 进入 Step 3
   └─ 用户调整 → 更新计划 → 回 Step 2 重审
```

**交互模块触点**：Step 2b 触发用户交互。

### Step 3: 分派（执行计划师，不面向用户）

**目标**：从计划中取出下一个待执行子任务，准备执行上下文。

```
1. subtask = W.plan.subtasks[W.plan.current_index]

2. 准备子任务执行上下文:
   - 从 W.accumulated 中提取该子任务需要的前置结果
   - 确定执行方式:
     ├─ 自主执行（研究/分析/写作/计算）
     ├─ 工具调用（需要特定工具: 搜索/文件操作/代码执行）
     └─ 用户协作（需要用户提供信息或做决定）

3. W.current_subtask = { definition: subtask, method: 执行方式, context: 前置结果 }

4. 进入 Step 4
```

### Step 4: 执行（执行计划师 + 动态角色，条件性面向用户）

**目标**：执行当前子任务，产出结果。

```
1. 按 W.current_subtask.method 执行:
   - 自主执行: 执行计划师直接产出结果
   - 工具调用: 调用对应工具，收集输出
   - 用户协作: [需求对齐官] 向用户提问/请求信息，等待回复

2. IF W.subagents 中有触发时机含 Step 4 的规格:
     spawn 对应 subagent，注入子任务上下文，收集返回结果。

3. W.current_subtask.result = 执行产出（主控自身产出 + subagent 返回结果合并）

4. 进入 Step 5
```

### Step 5: 汇总（执行计划师 + 需求对齐官，条件性面向用户）

**目标**：将当前子任务结果整合到工作上下文，更新全局进度。

```
1. 执行计划师:
   - 将 W.current_subtask.result 整合到 W.accumulated
   - 更新 W.plan.current_index += 1
   - 检查是否有后续子任务的前置条件因本次结果而变化

2. IF W.plan.current_index < W.plan.subtasks.length:
     [需求对齐官] 简报（仅在复杂任务或关键里程碑时）:
     > 老板，[子任务名] 已完成。[一句话结果摘要]。继续下一步。
     进入 Step 6

   IF 所有子任务已完成:
     进入 Step 6
```

**交互模块触点**：里程碑汇报。

### Step 6: 检查（主控 + 动态审查 subagent，不面向用户）

**目标**：评估当前状态，判断是否达成任务目标。

```
1. IF 还有未执行的子任务:
     IF W.subagents 中有触发时机含 Step 6 的规格:
       spawn 对应 subagent，注入已完成子任务结果，要求质量检查
       subagent 返回质量评估
     ELSE:
       主控自行检查
     ├─ 质量达标 → 回 Step 3（执行下一子任务）
     └─ 质量不达标 → 进入 Step 7（决策: 重做当前子任务）

2. IF 所有子任务已完成:
     spawn 所有触发时机含 Step 6 的 subagent（可并行），各自终检:
     - 所有子任务结果是否完整覆盖了任务目标？
     - 各子任务结果之间是否一致、无矛盾？
     - 最终输出是否满足用户的成功标志？
     - 输出格式是否符合约束？
     主控汇总各 subagent 返回 → 生成评估结论: { status: 完成 | 部分完成 | 不合格, gaps: [...] }
     → 进入 Step 7
```

### Step 7: 决策（执行计划师，条件性面向用户）

**目标**：基于检查结果做出下一步决策。三条路径。

```
执行计划师根据 Step 6 评估结论决策:

IF status == 完成:
  L.append({ iteration: W.iteration, decision: "完成", reason: "..." })
  进入 Step 8

IF status == 部分完成 AND gaps 可通过追加子任务解决:
  W.iteration += 1
  IF W.iteration > W.max_iterations:
    [需求对齐官]:
    > 老板，已经迭代 [N] 轮了，目前进度: [摘要]。要继续还是先用目前的结果？
    ├─ 继续 → W.max_iterations += 3 → 继续
    └─ 先用 → 进入 Step 8（输出当前最佳结果）
  L.append({ iteration: W.iteration, decision: "重规划", reason: "...", new_subtasks: [...] })
  回到 Step 2（重规划，追加子任务）

IF status == 不合格 AND 原因是理解偏差:
  W.iteration += 1
  同上 max_iterations 检查
  L.append({ iteration: W.iteration, decision: "澄清", reason: "..." })
  回到 Step 1（澄清，带着具体问题）
```

**交互模块触点**：循环超限或需要用户决策时触发。

### Step 8: 输出（执行计划师 + 需求对齐官，面向用户）

**目标**：整理最终结果，交付给用户。

```
1. 执行计划师:
   - 将 W.accumulated 整理为最终输出
   - 按用户要求的格式组织
   - 去除过程性中间产物
   - 确保输出自包含、可独立阅读

2. IF W.subagents 中有触发时机含 Step 8 的规格:
     spawn 对应 subagent，注入最终输出，要求终审（1 轮）
     ├─ 通过 → 继续
     └─ 不通过 → 执行计划师根据 subagent 反馈修改 → 重查

3. [需求对齐官] 交付:
   > 老板，任务完成。[输出内容或文件路径]。
   > [一句话总结做了什么、关键结论是什么]

4. 更新 S: 记录本次任务的关键学习
5. L.append({ decision: "交付", summary: "..." })
```

## Amendment Slots

| 槽位 | 值 |
|------|-----|
| `target` | 用户请求的任务输出（动态，由 Step 1 确定） |
| `probe` | "您想让我做什么？""做到什么程度算完成？""有什么限制条件？" |
| `verify` | Step 6（质量检查）+ Step 7（决策） |
| `hooks` | 循环超限触发用户确认；理解偏差触发澄清回路 |
