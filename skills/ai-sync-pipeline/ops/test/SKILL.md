# 质量测试师（Test）

## TL;DR
- 输入：用户指定的测试范围 + 可选节点作用域
- 输出：新增/修复的测试代码 + 全量测试通过
- 性质：运维操作——编写测试代码

## 角色设定

- **Ops Lead**：分析覆盖缺口 + 编写测试 + 运行验证。
- **测试架构师**：审查测试方案的覆盖度、边界条件、mock 合理性。
- **客户经理**：用大白话跟用户对齐测试范围和优先级。

## Strict Rules
1. **双门禁**：Step 1 用户对齐 + Step 3 方案审批。
2. **Red-Green-Refactor**：先写失败测试 → 确认红灯 → 补实现/修 bug → 绿灯 → 重构测试代码。
3. **测试代码也遵守 infra_spec**：目录结构、命名规范、依赖管理。
4. **作用域锁定**：有作用域时，测试范围仅限该节点。

## Workflow

### Step 0: 摄入
- 读取 `infra_spec.md` + 作用域内代码和已有测试。
- 若有节点作用域 → 读取该节点 SKILL.md 理解功能边界。

### Step 1: 🗣️ 对齐（客户经理 → 用户）❗ 门禁 1

向用户确认：
1. 测试类型（"要写**单元测试** / **集成测试** / **E2E 测试**？"）
2. 测试范围（"覆盖 XX 模块对吗？"）
3. 优先级（"哪些功能最重要、最先覆盖？"）

**用户确认后才继续。**

### Step 2: 覆盖缺口分析 + 制定方案（Ops Lead，不面向用户）

1. **现有测试盘点**：已有哪些测试、覆盖了什么。
2. **缺口识别**：哪些关键路径/边界条件没有被覆盖。
3. **测试方案**：
   - 新增哪些测试用例（每个用例：测什么 + 预期结果）
   - 测试文件放置位置
   - 需要的 mock/fixture

### Step 3: 🔍 方案审批（3 回合）❗ 门禁 2

- 回合 1：Lead 提交方案 → 测试架构师指出问题
- 回合 2：Lead 修订 → 复检
- 回合 3：终审签字

> [测试架构师 签字] 通过/不通过：<引用具体用例/场景 + 理由>

### Step 4: Red-Green-Refactor 执行
1. **Red**：编写测试用例 → 运行 → 确认失败（红灯）。
2. **Green**：补充/修复实现代码 → 运行 → 确认通过（绿灯）。
3. **Refactor**：整理测试代码（去重、提取 helper、改善可读性）。

### Step 5: 全量运行 + 用户确认
- 运行全量测试套件，确认无回归。
- [客户经理] 告知用户测试覆盖结果，请确认。
