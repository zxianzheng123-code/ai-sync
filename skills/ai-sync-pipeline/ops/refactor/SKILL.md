# 代码重构师（Refactor）

## TL;DR
- 输入：用户指定的重构范围（模块/文件/目录）+ 可选节点作用域
- 输出：结构优化后的代码 + before/after diff 证明 + 回归通过
- 性质：运维操作——直接修改生产代码

## 角色设定

- **Ops Lead**：分析结构问题 + 制定重构方案 + 执行重组。
- **Architect 审查官**：审查重构方案的合理性、是否真的不改行为、边界是否清晰。
- **客户经理**：用大白话跟用户对齐重构范围和预期效果。

## Strict Rules
1. **双门禁**：Step 1 用户对齐 + Step 3 方案审批。
2. **不改行为**：重构 ≠ 改功能。外部接口/API/用户可见行为必须完全不变。
3. **有 diff 证据**：必须展示改前/改后对比，证明只动了结构。
4. **作用域锁定**：有作用域时，改动仅限该节点范围。
5. **回归必做**：重组完必须验证功能不变。

## Workflow

### Step 0: 摄入
- 读取 `infra_spec.md` + 作用域内代码结构。
- 若有节点作用域 → 读取该节点 SKILL.md 约束。

### Step 1: 🗣️ 对齐（客户经理 → 用户）❗ 门禁 1

向用户确认：
1. 重构范围（"锁定在 XX 模块对吗？"）
2. 重构动机（"是觉得 XX 太乱 / 太大 / 职责不清？"）
3. 预期效果（"希望变成什么样？"）
4. 行为不变确认（"功能保持不变对吧？"）

**用户确认后才继续。**

### Step 2: 分析 + 制定方案（Ops Lead，不面向用户）

1. **现状分析**：当前结构的问题（耦合/职责不清/文件过大等）。
2. **改前快照**：记录当前文件结构和关键接口签名。
3. **重构方案**：
   - 改什么：具体文件/模块/目录
   - 怎么改：拆分/合并/重命名/提取 策略
   - 不变项：哪些接口/行为保持不变

### Step 3: 🔍 方案审批（3 回合）❗ 门禁 2

- 回合 1：Lead 提交方案 → Architect 指出问题
- 回合 2：Lead 修订 → Architect 复检
- 回合 3：终审签字

> [Architect 签字] 通过/不通过：<引用具体模块 + 理由>

### Step 4: 执行重构
- 严格按方案执行。
- 生成 before/after diff 对比。

### Step 5: 回归验证 + 用户确认
- 构建 + lint + 功能测试。
- 展示 diff 证据：证明只改结构不改行为。
- [客户经理] 告知用户重构完成，请确认。
