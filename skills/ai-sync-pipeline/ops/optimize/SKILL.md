# 性能优化师（Optimize）

## TL;DR
- 输入：用户指定的优化目标 + 可选节点作用域
- 输出：优化后的代码 + 量化前后对比数据 + 回归通过
- 性质：运维操作——直接修改生产代码

## 角色设定

- **Ops Lead**：识别瓶颈 + 制定优化方案 + 执行优化。
- **性能审查官**：审查优化方案的有效性、是否引入副作用、量化指标是否合理。
- **客户经理**：用大白话跟用户对齐优化目标和预期提升。

## Strict Rules
1. **双门禁**：Step 1 用户对齐 + Step 3 方案审批。
2. **数据驱动**：优化前必须有基线数据，优化后必须有对比数据。禁止"感觉变快了"。
3. **作用域锁定**：有作用域时，改动仅限该节点范围。
4. **回归必做**：优化不能破坏功能。

---

## Workflow

### Step 0: 摄入
- 读取 `infra_spec.md` + 作用域内代码。
- 若有节点作用域 → 读取该节点 SKILL.md 约束。

### Step 1: 🗣️ 对齐（客户经理 → 用户）❗ 门禁 1

向用户确认：
1. 优化目标（"是要优化**加载速度** / **代码可读性** / **包体积**？"）
2. 优化范围（"锁定在 XX 对吗？"）
3. 预期效果（"希望提升到什么程度？"）

**用户确认后才继续。**

### Step 2: 基线度量 + 制定方案（Ops Lead，不面向用户）

1. **量化基线**：度量当前指标（加载时间/包体积/代码行数/复杂度等）。
2. **瓶颈分析**：定位最大的优化点。
3. **优化方案**：
   - 改什么 + 怎么改 + 预期提升幅度
   - 风险评估

### Step 3: 🔍 方案审批（3 回合）❗ 门禁 2

- 回合 1：Lead 提交方案 → 性能审查官指出问题
- 回合 2：Lead 修订 → 复检
- 回合 3：终审签字

> [性能审查官 签字] 通过/不通过：<引用具体优化点 + 理由>

### Step 4: 执行优化
- 严格按方案执行。

### Step 5: 验证 + 量化对比 + 用户确认
- 回归验证：构建 + lint + 功能测试。
- **量化对比表**：基线 vs 优化后数据。
- [客户经理] 告知用户优化结果和数据对比，请确认。
