# 优化技能

基于 autoresearch 方法论，通过自动化实验循环优化一个技能的 SKILL.md。

核心原理：跑基线 → 每次只改一处 → 二元评分 → 分数提高就保留，否则回滚 → 循环。

## 上下文

- 目标技能路径参数：$ARGUMENTS
- 评估指南：!`cat scripts/autoresearch/references/eval-guide.md`

## 执行流程

### 准入检查

1. 如果 $ARGUMENTS 非空 → 直接读取该路径的 SKILL.md
2. 如果 $ARGUMENTS 为空 → 扫描 `技能/` 下所有含 SKILL.md 的目录，列出清单让用户选择
3. 路径无效或 SKILL.md 不存在 → 报错退出

读取目标技能的 SKILL.md 后，问用户以下问题，**全部确认后才能继续**：

1. **测试输入** — 3-5 个不同的提示/场景，用来测试技能（要覆盖不同用例，避免过拟合）
2. **二元评估标准** — 3-6 条 yes/no 判断，定义"好产出"长什么样
3. **每轮实验次数** — 每次变异跑几遍？默认 5
4. **预算上限** — 最多跑几轮实验？默认不封顶（跑到你喊停）

如果用户定义不出二元评估标准（只能说"看感觉"），直接告诉用户：

> 这个技能目前不适合跑自动优化。建议先手动用几次，积累对"好/坏"的具体判断，再来。

不要硬凑评估标准往下走。这是唯一的硬门槛。

### Step 1：理解技能

1. 完整阅读目标 SKILL.md
2. 阅读 references/ 下引用的所有文件
3. 理解技能的核心任务、流程步骤、输出格式
4. 记录现有的质量检查和反模式规则

### Step 2：构建评估套件

把用户的评估标准格式化：

```
EVAL [编号]: [短名称]
问题: [yes/no 问题]
通过: [什么算 yes — 一句话，具体]
失败: [什么算 no — 一句话，具体]
```

规则：
- 只要二元判断，不要打分制
- 具体到两个不同的人看同一个产出能得出一样的结论
- 3-6 条，多了技能会开始"应试"而不是真正变好

满分 = 评估条数 × 每轮次数。

### Step 3：建立基线（实验 #0）

1. 问用户给优化版本起个名字（如 v2、optimized）
2. 在技能目录下创建 `autoresearch/` 工作目录
3. 复制 SKILL.md 到工作目录作为 `[用户起的名字].md` — 所有改动只在这个副本上
4. 保存 `SKILL.md.baseline` 作为回滚基准
5. 创建 results.tsv、results.json、dashboard.html，打开仪表盘
6. 用副本跑 N 次，逐条评分，记录基线分数

**results.tsv 格式（tab 分隔）：**

```
experiment	score	max_score	pass_rate	status	description
0	14	20	70.0%	baseline	原始技能 — 未做任何修改
```

**results.json 格式：**

```json
{
  "skill_name": "[技能名]",
  "status": "running",
  "current_experiment": 0,
  "baseline_score": 70.0,
  "best_score": 70.0,
  "experiments": [],
  "eval_breakdown": []
}
```

基线跑完后，向用户确认分数。如果已经 ≥ 90%，问是否还要继续。

### Step 4：生成实时仪表盘

在 `autoresearch/dashboard.html` 创建自动刷新的浏览器仪表盘：

- 每 10 秒读取 results.json 刷新
- 折线图显示分数趋势（X 轴实验编号，Y 轴通过率）
- 每轮实验用色条标记：绿 = 保留，红 = 丢弃，蓝 = 基线
- 表格列出所有实验：编号、得分、通过率、状态、描述
- 按评估项拆分：哪条评估通过率最高/最低
- 当前状态："正在运行实验 [N]…" 或 "已完成"

单文件，内联 CSS + JS，用 CDN 加载 Chart.js。创建后立即用 `open dashboard.html` 打开。

每轮实验结束后同步更新 results.json。全部完成后把 status 改为 `"complete"`。

### Step 5：实验循环

启动后自动运行，不再逐轮问用户。

**每轮：**

1. **分析失败** — 哪条评估挂最多？实际产出哪里不对？
2. **形成假设** — 只改一个地方。好的改动：
   - 针对最常见失败加一条明确指令
   - 把模糊措辞改具体
   - 加反模式（"不要做 X"）
   - 把被埋没的关键指令提到前面
   - 加一个正确示例
   - 删掉导致过度优化某一项的指令
3. **改副本** — 编辑工作目录里的副本，绝不碰原始 SKILL.md
4. **跑实验** — 同样的测试输入，N 次
5. **评分** — 逐条评估，算总分
6. **决定** — 分数涨了 → 保留；没涨或降了 → 回滚副本到上一版
7. **记录** — 更新 results.tsv、results.json、changelog.md
8. **继续** — 回到第 1 步

**停止条件：**
- 用户手动停止
- 达到预算上限
- 连续 3 轮 ≥ 95% 通过率

**changelog.md 每轮格式：**

```markdown
## 实验 [N] — [保留/丢弃]

**得分：** [X]/[满分] ([百分比]%)
**改动：** [一句话描述改了什么]
**理由：** [为什么预期这个改动有帮助]
**结果：** [哪条评估提升/下降了]
**仍存在的失败：** [还有什么没解决]
```

### Step 6：交付

循环结束后输出：

1. 基线分数 → 最终分数（提升百分比）
2. 总实验轮数，保留/丢弃各多少
3. 帮助最大的 3 个改动（从 changelog 提取）
4. 仍然存在的失败模式（如有）
5. 改进后的副本位置
6. results.tsv 和 changelog.md 位置

**绝不自动覆盖原始 SKILL.md。** 用户自己决定是否采用。

## 产出文件结构

```
技能/XX/autoresearch/
├── dashboard.html       ← 浏览器实时仪表盘
├── results.json         ← 仪表盘数据源
├── results.tsv          ← 每轮实验得分记录
├── changelog.md         ← 每轮变异详细日志
├── SKILL.md.baseline    ← 优化前的原始备份
└── [用户起的名字].md     ← 优化后的副本
```
