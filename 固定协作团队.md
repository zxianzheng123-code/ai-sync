# 固定协作团队

## 定位

本文件定义 `ai-sync` 仓库的默认 agent team。  
目标不是把每次任务都拆成很多 agent，而是给 `Codex` 和 `Claude Code` 一套稳定、可复用、可裁剪，而且会被真实执行的 subagent 协作分工。

默认原则：

- 主控始终只有一个，负责直接面向用户。
- 其他 agent 只做子任务，不直接和用户对话。
- 小任务可以不启用 team，直接由主控完成。
- 只有当任务满足“跨文件、跨层级、可并行、可验收”时，才启用 team。
- 一旦启用固定 team，就必须实际派发 subagents；固定 team 不是心智模型，不是口头分角色。

## 与角色库的关系

本仓库有两套角色体系，各管各的：

- **固定协作团队**（本文件）= 管仓库维护的分工角色，负责"ai-sync 仓库本身怎么改"
- **角色库**（`协议/角色/` 目录）= 管技能执行的行为角色，负责"技能运行时谁做什么"

两者不冲突、不重叠，可以同时启用。快速判断：

- 改仓库结构、改文件、检查一致性 → 用固定协作团队
- 执行技能任务（起草、审查、翻译）→ 用角色库

详见 `协议/角色协议.md` 第五节。

## 默认 4 角

### 1. 总控 Lead

负责：

- 理解用户目标，判断任务属于哪一层。
- 决定是否需要并行和派子任务。
- 给每个子任务定义输入、输出、边界和验收标准。
- 最终整合结果，并面向用户汇报。

不负责：

- 不把未验收的子结果直接交给用户。

### 2. 规则与目录官

负责：

- 审查改动是否符合 `README.md`、`AGENTS.md`、`CLAUDE.md`、`使用指引.md` 的边界。
- 判断内容该放根目录、`技能/`、`对外规则库/` 还是 `临时资料/`。
- 防止把一次性资料混入正式资产。

适用任务：

- 仓库结构调整
- 新增或移动文档
- 协作规则更新

### 3. 技能与协议官

负责：

- 审查技能、共享协议、引用矩阵之间是否一致。
- 新建或修改 `SKILL.md` 时，检查输入输出、暗号、上下游、协议引用是否失配。
- 判断是否需要同步 `正哥技能召唤书.md`、`技能/CHANGELOG.md`、`协议/协议引用矩阵.md`。

适用任务：

- 新建技能
- 修改技能边界
- 修改共享协议
- 技能体系重构

### 4. 验证与审计官

负责：

- 做路径、引用、结构、变更范围检查。
- 回读关键文件，确认没有串位和漏同步。
- 汇总 `git status --short` 和必要 diff，给总控做最终验收依据。

适用任务：

- 任何正式资产改动后的收尾

## 可选扩展角色

### 代码与脚本官

只在任务明确涉及脚本、自动化或实现逻辑时启用。

负责：

- 改脚本或代码片段
- 跑最小必要验证
- 反馈实现风险和未覆盖点

默认不开启。这个仓库不是代码优先仓，避免把所有任务都误判成开发任务。

## 启用规则

### 单人直做

满足以下任一情况时，不拆 team：

- 单文件微调
- 纯措辞修改
- 路径和边界都很清楚
- 不涉及跨文件同步

### 启用固定 team

满足以下任一情况时，必须启用真实 subagents：

- 需要同时改 `AGENTS.md` / `CLAUDE.md` / `README.md` / `使用指引.md`
- 技能、协议、召唤书、变更日志之间要联动
- 任务拆成 2 个以上可以独立验收的子块
- 需要一边改内容，一边做结构或引用核查

启用后必须遵守：

- 除总控外，至少派发 `2` 个独立 subagents，不能只开 `1` 个就称为 team。
- 每个 subagent 都要有明确的输入、输出、边界和验收标准。
- 禁止伪 team：不允许主控在单线程里自称“规则官 / 协议官 / 审计官已完成”，但实际上没有真实派发。
- 子任务之间如果写入范围重叠，总控要先拆边界，再派发。

## 默认分工模板

```text
总控 Lead
  ├─ 规则与目录官：判定放置位置、检查仓库边界
  ├─ 技能与协议官：检查技能体系、协议引用、同步文件
  └─ 验证与审计官：做路径验证、内容回读、git 变更确认
```

如果任务涉及脚本或实现，再额外挂：

```text
总控 Lead
  ├─ 代码与脚本官：负责实现
  └─ 验证与审计官：负责验收
```

## 平台约束

- `Codex` 和 `Claude Code` 共用这套 team 角色定义。
- 平台不同，工具可以不同，但角色职责不变。
- `Codex` 启用固定 team 时，必须使用 agent / sub-agent 功能做真实派发，并遵守本仓库 `AGENTS.md` 和 `协议/工具链协议.md`。
- `Claude Code` 启用固定 team 时，必须使用 subagent 机制做真实派发，同样遵守这里的角色边界和单一真相源原则。
- 两边都不允许只写角色名、不做真实分派。

## 交付要求

每次启用 team，最终都要由总控收束成三件事：

1. 改了什么
2. 为什么这么分工
3. 还有什么没闭环
