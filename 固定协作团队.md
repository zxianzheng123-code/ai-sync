# AI 团队架构

## 定位

本文件定义 `ai-sync` 仓库的团队架构和协作机制。

核心理念：**技能 = 工作流程，不是角色。** 一个技能描述的是"一件事怎么干完"，而不是"一个人会干什么"。流程里可以有多个工作人员协作。

-----

## 团队角色

### 项目工程师（总指挥）

- 身份：所有角色的创造者和管理者，一心一意为用户打造一流团队
- 职责：
  - 创建和管理所有下属角色
  - 维护协议层（共享协议只有他能改）
  - 维护项目运维脚本（`scripts/`）
  - 团队架构调整、角色增删
  - 管理人设库（`技能/人设库/`）
  - 维护护栏钩子（`hooks/`）

### 档案师（资产管家）

- 职责：
  - 维护**专业知识**（`技能/人设库/XX/专业知识/`）—— 人设自带的能力底子
  - 维护**作业指引**（技能目录下 `作业指引/`）—— 各工作流程专属的操作指导
  - 维护**输出模板**（技能目录下 `资源/`）—— 各工作流程产出的固定骨架

### 工作编排师（流程总监）

- 职责：
  - 编排所有工作流程（SKILL.md 的 Workflow 部分）
  - 维护**业务脚本**（技能目录下 `脚本/`）—— 工作流程中需要执行的自动化脚本
  - 维护**验证脚本**（技能目录下 `脚本/验证/`）—— 产出质量的自动化检查
  - 分配工作人员到各流程中

### 工作人员（执行者）

- 身份：真正干活的"人"，是人设不是技能
- 存放：集中存放在 `技能/人设库/`，各工作流程按需引用
- 特点：
  - 一个人设可以被多个工作流程引用
  - 人设定义的是**通用能力基线**，流程专属的行为差异写在 SKILL.md 的步骤说明中

-----

## 资源分类与归属

| 资源类型 | 是什么 | 谁维护 | 放在哪 |
|---|---|---|---|
| 工作流程 | 技能本体，描述一件事从头到尾怎么干 | 工作编排师 | `SKILL.md` |
| 工作人员（人设） | 参与流程的角色定义，可跨流程复用 | 项目工程师 | `技能/人设库/` |
| 专业知识 | 人设自带的能力底子，跟着人走 | 档案师 | `技能/人设库/XX/专业知识/` |
| 作业指引 | 流程专属的操作指导，跟着流程走 | 档案师 | 技能目录下 `作业指引/` |
| 输出模板 | 工作流程产出的固定骨架 | 档案师 | 技能目录下 `资源/` |
| 客户反馈报告 | 踩坑点、常见错误、用户反馈 | 客服工作自动积累 + 用户主动要求 | 仓库根目录 `客户反馈报告/` |
| 业务脚本 | 工作流程中的自动化脚本 | 工作编排师 | 技能目录下 `脚本/` |
| 验证脚本 | 检验产出是否达标的自动化检查 | 工作编排师 | 技能目录下 `脚本/验证/` |
| 护栏钩子 | 执行前/后自动拦截危险操作或强制检查 | 项目工程师 | 仓库根目录 `hooks/` |
| 项目运维脚本 | 仓库级维护脚本 | 项目工程师 | `scripts/` |

## 人设库

集中存放在 `技能/人设库/`，每个人设一个目录：

```
技能/人设库/
├── 需求分析员/
│   ├── 人设.md
│   └── 专业知识/
├── 产品架构师/
│   ├── 人设.md
│   └── 专业知识/
├── 全栈开发/
│   ├── 人设.md
│   └── 专业知识/
├── UI设计/
│   ├── 人设.md
│   └── 专业知识/
├── 测试工程师/
│   ├── 人设.md
│   └── 专业知识/
├── 部署运维/
│   ├── 人设.md
│   └── 专业知识/
└── ...
```

### 人设复用规则

人设定义的是**通用能力基线**——这个角色的身份、擅长什么、底线是什么。

当同一个人设在不同流程中行为有差异时，差异部分**写在 SKILL.md 的步骤说明里**，不改人设本身。

各工作流程的 SKILL.md 中按需引用：

```
## 工作人员
- 需求分析员（→ 技能/人设库/需求分析员/人设.md）
- 产品架构师（→ 技能/人设库/产品架构师/人设.md）
```

## Master Agent 与 SubAgent 机制

每个工作流程必须采用 **Master Agent + SubAgent** 的执行模式：

### Master Agent（主控）

- 就是 SKILL.md 本身的执行者
- 职责：调度各步骤、启动 SubAgent、收集产出、判断是否达标、推进到下一步
- **不亲自干活**，只做编排和验收

### SubAgent（子代理）

- 每个工作人员在执行具体步骤时，以 SubAgent 身份启动
- SubAgent 必须**显式启动**，不能由 Master Agent 自己"扮演"工作人员
- 启动时传入：人设 + 专业知识 + 该步骤的目标和成功标准

### 启动指令格式

SKILL.md 中每个步骤必须写明：

```
### Step N：[步骤名]
- 执行者：[工作人员名]（SubAgent）
- 目标：[这一步要完成什么]
- 成功标准：[什么情况算完成，用可验证的条件]
- 输入：[从上一步接收什么]
- 输出：[交给下一步什么]
```

Master Agent 在 SubAgent 返回结果后，对照成功标准检查：

- 达标 → 进入下一步
- 不达标 → 退回 SubAgent 重做，附上不达标原因

## 加载上下文机制

每个技能配一个 `脚本/加载上下文.sh`，在 Workflow 的 Step 0 强制执行。脚本把所有引用内容一次性拼接输出给 AI，确保不会漏读。

### 注意事项

- 加载脚本由**工作编排师**维护，每次技能引用的人设或资料变更时同步更新
- 如果某个文件不存在，脚本不会报错，只是跳过
- 上下文窗口有限，如果一个技能引用的内容太多导致溢出，需要精简专业知识或作业指引

-----

## 验证机制

每个工作流程可以配验证脚本，放在 `脚本/验证/` 下。验证在**交付用户之前**自动执行。

验证分两种：

- **脚本验证**：能用代码检查的就用代码
- **自查清单验证**：不能用代码检查的用清单（`作业指引/交付自查清单.md`）

SKILL.md 的 Workflow 中，交付步骤前必须有验证步骤。

## 护栏钩子

放在仓库根目录 `hooks/`，由项目工程师维护。钩子分两类：

### 全局钩子（始终生效）

- **危险操作拦截**：执行 bash 命令前自动检查，遇到危险操作时拦截并要求用户确认
- **删除确认**：任何删除文件/目录操作，必须向用户确认后执行

### 按需钩子（用户主动激活）

- **严格模式**：拦截所有对指定目录之外的写操作
- **生产环境护栏**：操作生产环境时自动增加二次确认

-----

## 客服工作（用户唯一入口）

客服工作是用户的唯一入口，职责：

1. **接待**：接收用户的原始需求
2. **成题**：判断需求是否清晰，不清晰就先问
3. **路由**：把需求分配到正确的工作流程
4. **计划模式**：当现有工作流程都无法覆盖用户需求时，进入临时编排
   - 从人设库中挑选合适的工作人员
   - 临时定义步骤、交付物、检查点
   - 执行完毕后询问用户：这个流程要不要沉淀为正式技能？
5. **客户反馈积累**：自动记录工作流程执行中的问题 + 用户主动反馈

-----

## SKILL.md description 写法规范

description 字段**不是功能说明，而是触发条件**。

```
# ❌ 错误写法
description: 这个技能帮你规划产品需求

# ✅ 正确写法
description: 当用户想做一个新工具/新功能，但还没想清楚要做什么、做成什么样时，使用此技能
```

规则：

- 用"当……时"的句式
- 描述用户的**状态和意图**，而不是技能的能力
- 如果两个技能的 description 容易混淆，要写清楚边界

## 整体架构一览

```
用户
 │
 ▼
客服工作（唯一入口）
 │  接待 → 成题 → 路由
 │
 ├─ 匹配到现有流程 ──→ 具体工作流程（技能）
 │                       │  Master Agent 启动
 │                       │  Step 0：加载脚本强喂所有引用内容
 │                       │  Step 1~N：逐步启动 SubAgent 执行
 │                       │    每步有目标 + 成功标准
 │                       │    不达标 → 退回重做
 │                       │  交付前：验证脚本 + 自查清单
 │                       ▼
 │                      产出交付给用户
 │
 └─ 没有匹配的流程 ──→ 计划模式（临时编排）
                        │  从人设库挑人
                        │  临时定义步骤和交付物
                        │  执行完毕
                        ▼
                       问用户：要沉淀为正式技能吗？
                        ├─ 要 → 交给编排师正式建技能
                        └─ 不要 → 一次性任务结束

全程生效：护栏钩子（危险操作拦截、删除确认）
全程生效：客户反馈自动积累
```

管理层：

- **项目工程师**：管协议、管人设库、管架构、管护栏钩子
- **档案师**：管专业知识、管作业指引、管输出模板
- **工作编排师**：管工作流程、管业务脚本、管验证脚本

-----

## 平台约束

- `Codex` 和 `Claude Code` 共用这套团队架构。
- 平台不同，工具可以不同，但角色职责不变。
- 两边都必须使用 subagent 机制做真实派发，不允许只写角色名、不做真实分派。
