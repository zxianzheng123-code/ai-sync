# 严格模式

## 类型

按需（用户手动激活/停用）

## 触发时机

激活后，在任何**写操作**（创建、修改、删除、移动、重命名文件或目录）执行**之前**触发。

## 激活方式

用户通过以下方式激活：

```
启用严格模式，白名单目录：src/components/, tests/
```

停用：

```
关闭严格模式
```

## 拦截规则

### 核心逻辑

维护一个**目录白名单**。所有写操作的目标路径必须落在白名单目录内，否则拦截。

### 检测的写操作类型

- 文件创建（新建文件、`touch`）
- 文件修改（编辑、`sed`、`awk` 写入、重定向 `>`/`>>`）
- 文件删除（`rm`、`unlink`）
- 文件移动/重命名（`mv`、`git mv`）
- 目录创建/删除（`mkdir`、`rmdir`）
- Git 操作（`git checkout -- <file>`、`git restore`、`git stash pop` 等会改变工作区的命令）

### 路径匹配

- 白名单使用**前缀匹配**：`src/components/` 包含其下所有子目录和文件。
- 支持绝对路径和相对路径（相对于项目根目录）。
- 符号链接解析后再匹配，防止通过软链接绕过限制。

## 处理方式

1. **阻断执行**：目标路径不在白名单内时，拒绝执行。
2. **明确提示**：告知用户被拦截的操作、目标路径，以及当前白名单。
3. **支持临时放行**：用户可对单次操作授权放行，不改变白名单。
4. **支持动态调整**：用户可随时追加或移除白名单目录。

## 示例

假设白名单为 `src/components/`、`tests/`：

```bash
# 允许 — 目标在白名单内
vim src/components/Button.tsx        # 通过
touch tests/unit/new-test.spec.ts    # 通过

# 拦截 — 目标不在白名单内
vim src/utils/helpers.ts             # 拦截
rm package-lock.json                 # 拦截
echo "debug" >> .env                 # 拦截
git checkout -- src/index.ts         # 拦截
```

拦截提示示例：

```
[严格模式] 操作被拦截
  操作：修改文件
  路径：src/utils/helpers.ts
  原因：该路径不在白名单内
  当前白名单：
    - src/components/
    - tests/

  输入"放行"可临时允许此次操作，或使用"白名单追加 src/utils/"永久添加。
```
