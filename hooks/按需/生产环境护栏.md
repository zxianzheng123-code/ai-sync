# 生产环境护栏

## 类型

按需（用户手动激活）

## 触发时机

激活后，在任何可能影响**生产环境**的操作执行**之前**触发。

## 激活方式

用户通过以下方式激活：

```
启用生产环境护栏
```

停用：

```
关闭生产环境护栏
```

## 拦截规则

### 生产环境 URL 检测

匹配包含以下特征的 URL：

- `prod`、`production` 关键词（如 `api.production.example.com`）
- 已知的生产域名（如不含 `staging`、`dev`、`test`、`local` 的正式域名）
- 端口 443/80 上的非开发地址

### 生产分支检测

匹配以下 Git 分支的推送、合并、部署操作：

- `main`、`master`
- `release/*`、`production`、`prod`
- 任何配置为保护分支的分支名

### 生产配置文件检测

匹配对以下文件的修改：

- `.env.production`、`.env.prod`
- `config/production.*`、`config/prod.*`
- `docker-compose.prod.yml`、`docker-compose.production.yml`
- `k8s/production/`、`deploy/prod/`
- `Dockerfile`（当内容涉及生产构建阶段）
- CI/CD 配置中的生产部署步骤

### 生产命令检测

- `deploy`、`publish` 命令中包含 `prod`/`production` 参数
- `kubectl apply` 指向生产命名空间
- `terraform apply` 未指定 `-target`（全量应用）
- `ssh` 连接到生产服务器

## 处理方式

1. **醒目警告**：使用高优先级提示，明确标注"生产环境操作"。
2. **展示影响范围**：列出操作将影响的生产资源（URL、分支、配置文件等）。
3. **双重确认**：要求用户输入确认短语（如"确认操作生产环境"），而非简单的 y/n。
4. **冷却期**：确认后等待 3 秒再执行，给用户最后的反悔机会。
5. **记录审计日志**：完整记录操作人、时间、命令、确认过程。

## 示例

以下操作会触发护栏：

```bash
# 推送到生产分支
git push origin main
git merge feature/new-api --into production

# 部署到生产
npm run deploy -- --env production
kubectl apply -f k8s/production/deployment.yaml

# 修改生产配置
vim .env.production
sed -i 's/old/new/' config/prod.json

# SSH 到生产服务器
ssh user@prod-server-01.example.com

# 数据库操作指向生产
psql -h db.production.example.com -d maindb
```

确认对话示例：

```
⚠️ [生产环境护栏] 检测到生产环境操作

  操作：git push origin main
  影响：将代码推送到生产分支 main

  此操作将直接影响生产环境，请确认：
  输入"确认操作生产环境"以继续，或输入任意其他内容取消。

  > 确认操作生产环境

  [等待 3 秒冷却期...]
  执行中...
```
