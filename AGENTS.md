# ai-sync AGENTS

## 仓库定位

本仓库是 `正哥` 的个人 AI 工具库，主要沉淀可复用的 `技能`、`共享协议`、`MCP 接入资产`、`CLI 工具资产`、`对外规则` 和少量仓库级方法文档。

它不是普通代码仓。默认先判断这次任务是在维护资产体系，还是只是在整理临时资料；不要先入为主按“装依赖、跑构建、改业务代码”的方式开工。

## 与 Claude 协作

本仓库采用 `AGENTS.md` 作为项目级规则主文件。

- `Codex` 直接读取本文件执行。
- `Claude Code` 通过根目录 `CLAUDE.md` 跳转到本文件，不维护第二份独立规则。
- 涉及仓库长期规则时，优先改这里，不要在 `AGENTS.md` 和 `CLAUDE.md` 各写一套。
- 如果某条规则只适用于某个子目录，应在对应子目录补局部规则文件，而不是继续把根文件写成长总表。

固定 agent team 见 `固定协作团队.md`。涉及多步骤、跨文件、可并行的任务时，必须按该文档启用真实 subagents 分工，不允许只在主线程口头扮演多个角色。

## 开工顺序

进入本仓库后，默认按这个顺序建立上下文：

1. 读 `context/project-context.md`，了解当前做到哪、下一步干什么、有什么注意点。
2. 读 `README.md`，确认目录职责和维护规则。
3. 读 `正哥技能召唤书.md`，确认对外入口和技能注册口径。
4. 读 `使用指引.md`，确认接入方式和调用方式。
5. 如果任务涉及 MCP 或外部工具，继续读 `MCP/MCP接入总表.md`，确认项目清单和模板入口；全量工具清单见 `工具注册表.md`。
6. 改动前先看 `git status --short`，避免覆盖用户正在进行的调整。

如果任务明确只涉及某个技能或某个协议，再进入对应目录继续读，不要一上来全仓漫游。

## 目录边界

- `README.md`：仓库定位、目录结构、维护规则。
- `正哥技能召唤书.md`：对外入口，只放技能列表、暗号、路径、协议索引。
- `使用指引.md`：外部项目如何接入本仓库。
- `工具注册表.md`：全量工具（MCP + CLI + Plugin）的单一真相源。
- `项目工具清单/`：每项目一份的工具启用清单，同时包含 MCP 和 CLI。
- `MCP/`：MCP 接入总表、配置模板、密钥模板和 MCP 变更记录。
- `CLI/`：CLI 工具变更记录。
- `协议/`：共享协议和角色库（所有技能的底层规范）。
- `技能/`：正式技能资产。
- `对外规则库/`：可复用的通用规则；没有用户明确要求，不要修改。
- `临时资料/`：临时参考资料；没有用户明确要求，不要把这里的内容当成正式规则来源。

## 改动原则

- 先判断改动对象属于“对外入口 / 仓库维护 / 技能资产 / 临时资料”哪一层，再动手。
- 一次性项目资料不要塞进 `技能/`。
- 技能目录默认使用中文命名；固定文件名保留 `SKILL.md`、`README.md` 这类标准名。
- 新建技能前，先确认现有技能不能覆盖，避免重复造轮子。
- 删除任何文件或目录前，先得到用户确认。
- 不要因为这个仓库里也有脚本或代码片段，就假设所有任务都需要运行构建或安装依赖。
- `工具注册表.md`、`项目工具清单/`、`MCP/` 内的 `tool_id`、`project_id`、模板文件名默认使用 ASCII `kebab-case`；不要在项目清单里发明注册表之外的新值。

## 技能与协议变更规则

### 新建技能

- 目录放在 `技能/<技能名>/`
- 必须包含 `SKILL.md`
- `SKILL.md` 至少覆盖：`YAML frontmatter`、`TL;DR`、`角色设定`、`Strict Rules`、`Workflow`、`Initialization`
- 同步更新 `正哥技能召唤书.md`
- 同步更新 `技能/CHANGELOG.md`

### 修改技能

- 微调文案或结构时，确认不破坏现有输入输出口径。
- 如果改了技能职责、暗号、路径、输入输出或上下游关系，要同步检查：
  - `README.md`
  - `正哥技能召唤书.md`
  - `技能/CHANGELOG.md`
  - `协议/协议引用矩阵.md`

### 修改共享协议

- 先确认哪些技能引用了该协议。
- 改完后，检查引用技能是否仍然成立，避免协议升级后下游失配。
- 如果协议边界变了，补同步 `README.md`、`正哥技能召唤书.md` 和 `协议/协议引用矩阵.md`。

## 工具变更规则

### 新建或删除工具（MCP / CLI / Plugin）

- 在 `工具注册表.md` 补或删 canonical 条目
- MCP 类型：同步 `MCP/MCP接入总表.md`、`MCP/CHANGELOG.md`；如有项目启用差异，更新 `项目工具清单/*.md`
- CLI 类型：同步 `CLI/CHANGELOG.md`；如有项目启用差异，更新 `项目工具清单/*.md`
- Plugin 类型：只更新 `工具注册表.md` 和 `MCP/CHANGELOG.md`

### 修改工具模板或边界

- 先确认改动落在注册表、项目清单，还是配置模板
- `项目工具清单/*.md` 中的 `tool_id`、`version`、`模板文件`、`密钥键` 必须来自 `工具注册表.md`
- 同步检查 `README.md`、`使用指引.md` 和相关项目清单
- 工具变更不得写入 `技能/CHANGELOG.md`

## 默认验证

本仓库优先做“结构和引用验证”，不是默认跑构建：

1. 结构检查：确认新增或修改的路径真实存在。
2. 内容回读：回读关键文件，确认分工和口径没有串位。
3. 引用检查：确认 `正哥技能召唤书.md` 中的技能路径有效，必要时检查下游引用。
4. 涉及工具时，确认 `工具注册表.md`、`MCP接入总表.md`、`项目工具清单/` 和模板路径一致。
5. 变更确认：查看 `git status --short` 和必要的 `git diff -- <file>`。

如果任务确实涉及脚本、自动化或代码，再补对应的命令验证；不要机械地给每次文档改动都附加无意义构建。

## 任务输入建议

用户给任务时，优先按这五项理解和补齐上下文：

- `目标`
- `上下文`
- `约束`
- `完成标准`
- `验证方式`

如果任务复杂或边界模糊，先做分析和计划，再改文件。
