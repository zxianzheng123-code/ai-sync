# ai-sync AGENTS

## 仓库定位

本仓库是 `正哥` 的个人 AI 工具库，主要沉淀可复用的 `工作流程（技能）`、`人设库`、`MCP 接入资产`、`CLI 工具资产`、`对外规则` 和仓库级方法文档。

它不是普通代码仓。默认先判断这次任务是在维护资产体系，还是只是在整理临时资料；不要先入为主按"装依赖、跑构建、改业务代码"的方式开工。

## 核心理念

**技能 = 工作流程，不是角色。** 一个技能描述的是"一件事怎么干完"，而不是"一个人会干什么"。流程里可以有多个工作人员（人设）协作。

## 与 Claude 协作

本仓库采用 `AGENTS.md` 作为项目级规则主文件。

- `Codex` 直接读取本文件执行。
- `Claude Code` 通过根目录 `CLAUDE.md` 跳转到本文件，不维护第二份独立规则。
- 涉及仓库长期规则时，优先改这里。

团队架构见 `固定协作团队.md`。涉及多步骤、跨文件、可并行的任务时，必须按该文档启用 Master Agent + SubAgent 真实分派，不允许只在主线程口头扮演多个角色。

## 开工顺序

进入本仓库后，默认按这个顺序建立上下文：

1. 读 `README.md`，确认目录职责和维护规则。
2. 读 `正哥技能召唤书.md`，确认对外入口和技能注册口径。
3. 读 `固定协作团队.md`，了解团队架构和协作机制。
4. 读 `使用指引.md`，确认接入方式和调用方式。
5. 如果任务涉及 MCP 或外部工具，继续读 `MCP/MCP接入总表.md`；全量工具清单见 `工具注册表.md`。
6. 改动前先看 `git status --short`，避免覆盖用户正在进行的调整。

如果任务明确只涉及某个技能，再进入对应目录继续读，不要一上来全仓漫游。

## 目录边界

- `README.md`：仓库定位、目录结构、维护规则。
- `正哥技能召唤书.md`：对外入口，只放技能列表、暗号、路径。
- `固定协作团队.md`：团队架构、角色定义、协作机制。
- `使用指引.md`：外部项目如何接入本仓库。
- `工具注册表.md`：全量工具（MCP + CLI + Plugin）的单一真相源。
- `项目工具清单/`：每项目一份的工具启用清单。
- `MCP/`：MCP 接入总表、配置模板、密钥模板和变更记录。
- `CLI/`：CLI 工具变更记录。
- `协议/`：共享协议（已标记迁移状态，原始内容保留供参考）。
- `技能/`：正式技能资产（工作流程）。
- `技能/人设库/`：集中管理的工作人员人设和专业知识。
- `hooks/`：护栏钩子（全局 + 按需）。
- `客户反馈报告/`：各工作流程的客户反馈积累。
- `对外规则库/`：可复用的通用规则；没有用户明确要求，不要修改。
- `临时资料/`：临时参考资料；不要把这里的内容当成正式规则来源。

## 临时资料 → Obsidian Canvas 逻辑图

当有新的临时资料添加到 `临时资料/` 时，需要同步生成对应的 Obsidian Canvas（`.canvas`）逻辑图：

1. **触发条件**：用户将参考资料放入 `临时资料/` 后，主动要求或 Agent 判断该资料适合可视化时。
2. **产出位置**：逻辑图保存在 `临时资料/` 同级目录，文件名与资料主题一致，扩展名为 `.canvas`。
3. **内容要求**：
   - 提炼资料的核心概念、第一性原理、关键环节之间的逻辑关系。
   - 卡片按层级从上到下排列，同层卡片横向展开。
   - 用颜色区分层级（如：核心价值=紫色、原理=红色、机制=绿色、流程=青色、模式=橙色、补充=黄色）。
   - 连线采用链式串联，避免从同一节点发射多条线导致视觉拥挤。
   - 卡片间保持足够间距（水平 ≥ 80px，垂直 ≥ 100px）。
4. **格式要求**：`.canvas` 文件是标准 JSON，必须用 `json.dump` 或等效方式生成以确保格式合法；文本中不得包含未转义的双引号。
5. **来源链接**：如资料有外部来源（URL / 仓库），在画布上添加 `"type": "link"` 节点。

## 改动原则

- 先判断改动对象属于"对外入口 / 仓库维护 / 技能资产 / 临时资料"哪一层，再动手。
- 一次性项目资料不要塞进 `技能/`。
- 技能目录默认使用中文命名；固定文件名保留 `SKILL.md`、`README.md` 这类标准名。
- 新建技能前，先确认现有技能不能覆盖，避免重复造轮子。
- 删除任何文件或目录前，先得到用户确认。
- `工具注册表.md`、`项目工具清单/`、`MCP/` 内的 `tool_id`、`project_id`、模板文件名默认使用 ASCII `kebab-case`。

## Mock 数据使用规范

- **严禁硬编码**：所有 Mock 数据不得直接写死在代码中。
- **动态生成优先**：通过函数、工厂方法或 Faker 等库动态生成 Mock 数据。
- **配置文件读取**：固定结构的 Mock 数据放在独立配置文件（JSON / YAML）中，代码通过读取配置获得。
- **专用 Mock 服务**：涉及 API 模拟时，使用 Mock Server 或拦截层，不在业务代码里内联假数据。
- **真实性要求**：Mock 数据必须模拟真实数据结构和响应格式，覆盖正常、边界和异常场景。
- **可维护性**：Mock 数据源集中管理，便于随业务需求变化统一调整和更新。

## SKILL.md 规范

### 文件格式

每个 SKILL.md 必须包含：

```yaml
---
description: 当……时，使用此技能（触发条件，用"当...时"句式）
trigger: 暗号
version: 2.0
---
```

### 必须章节

```markdown
## 执行模式
Master Agent 负责调度和验收，不亲自干活。每个步骤启动 SubAgent 执行。

## 工作人员
- XX（→ 技能/人设库/XX/人设.md）

## Workflow

### Step 0：加载上下文（Master Agent）
执行 脚本/加载上下文.sh

### Step 1：[步骤名]
- 执行者：[工作人员名]（SubAgent）
- 目标：
- 成功标准：
- 输入：
- 输出：

### Step N-1：验证
- 如有 脚本/验证/ 下的脚本 → 自动执行
- 如有 作业指引/交付自查清单.md → 逐项检查
- 全部通过才进入交付

### Step N：交付审核（Master Agent）
输出成品，等用户确认
```

### description 写法

- 用"当……时"的句式
- 描述用户的**状态和意图**，而不是技能的能力
- 两个技能 description 容易混淆时，写清楚边界

### 步骤执行规范

以下规则替代原 `流程执行协议` 和 `执行闭环协议`：

1. **步骤不可跳过**：按编号顺序执行，SKILL.md 授权的条件跳转除外
2. **步骤不可合并**：每个 Step 是独立执行单元，分开做、分开汇报
3. **步骤产出不可省略**：每个步骤定义了输出就必须产出
4. **不达标退回**：Master Agent 对照成功标准检查，不通过退回 SubAgent 重做
5. **用户确认不可替代**：标注"等用户确认"的步骤，必须等用户回复
6. **步骤完成标记**：每完成一步输出 `[Step X 完成] 步骤名 — 摘要`
7. **最小改动原则**：只改必须改的，不顺手加功能、不顺手重构

### SubAgent 分派规范

以下规则替代原 `工具链协议`：

1. SubAgent 必须**显式启动**，传入人设 + 专业知识 + 目标 + 成功标准
2. SubAgent 不直接面向用户对话，所有沟通由 Master Agent 统一转达
3. SubAgent 之间不直接通信
4. SubAgent 完成后按标准格式回报：执行状态、产出、摘要、遗留问题

### 技能目录标准结构

```
技能/XX/
├── SKILL.md                ← 工作流程
├── 作业指引/               ← 流程专属操作指导
├── 资源/                   ← 输出模板
└── 脚本/
    ├── 加载上下文.sh       ← Step 0 加载所有引用内容
    └── 验证/               ← 产出质量检查脚本
```

## 技能与协议变更规则

### 新建技能

- 目录放在 `技能/<技能名>/`
- 必须包含 `SKILL.md`，遵循上述格式规范
- 必须包含 `脚本/加载上下文.sh`
- 同步更新 `正哥技能召唤书.md`
- 同步更新 `技能/CHANGELOG.md`

### 修改技能

- 微调文案或结构时，确认不破坏现有输入输出口径。
- 如果改了技能职责、暗号、路径、输入输出或上下游关系，要同步检查：
  - `README.md`
  - `正哥技能召唤书.md`
  - `技能/CHANGELOG.md`

### 修改人设

- 人设只定义通用能力基线
- 流程专属的行为差异写在 SKILL.md 步骤说明里
- 修改人设后，检查引用该人设的所有技能是否仍然成立

## 工具变更规则

### 新建或删除工具（MCP / CLI / Plugin）

- 在 `工具注册表.md` 补或删 canonical 条目
- MCP 类型：同步 `MCP/MCP接入总表.md`、`MCP/CHANGELOG.md`
- CLI 类型：同步 `CLI/CHANGELOG.md`
- 如有项目启用差异，更新 `项目工具清单/*.md`

### 修改工具模板或边界

- 先确认改动落在注册表、项目清单，还是配置模板
- `项目工具清单/*.md` 中的值必须来自 `工具注册表.md`
- 同步检查 `README.md`、`使用指引.md` 和相关项目清单
- 工具变更不得写入 `技能/CHANGELOG.md`

## 默认验证

本仓库优先做"结构和引用验证"，不是默认跑构建：

1. 结构检查：确认新增或修改的路径真实存在。
2. 内容回读：回读关键文件，确认分工和口径没有串位。
3. 引用检查：确认 `正哥技能召唤书.md` 中的技能路径有效。
4. 涉及工具时，确认 `工具注册表.md`、`MCP接入总表.md`、`项目工具清单/` 和模板路径一致。
5. 变更确认：查看 `git status --short` 和必要的 `git diff -- <file>`。

## 任务输入建议

用户给任务时，优先按这五项理解和补齐上下文：

- `目标`
- `上下文`
- `约束`
- `完成标准`
- `验证方式`

如果任务复杂或边界模糊，先做分析和计划，再改文件。
