# 通用文档写作规则

## 1. 逻辑规则

| 规则 | 要求 | 不合格表现 |
| --- | --- | --- |
| 逻辑要求 A | 分类依据唯一；同一组内容只能按一种标准分类 | 页面层、字段层、流程层被放进同一组 |
| 逻辑要求 B | 前提先于结论；结论必须能由前文推出 | 先写结论，再回头补前提 |
| 逻辑要求 C | 定义先于使用；新概念、新对象、新页面、新字段，必须先定义再展开 | 下文首次出现上文未声明内容 |
| 逻辑要求 D | 总规则先于分规则，主规则先于例外规则 | 先写例外，再写主规则 |
| 逻辑要求 E | 规则必须挂靠到明确对象；不能出现“无挂靠规则” | 只写一条规则，不说明它作用于哪个对象 |
| 逻辑要求 F | 同一结论只保留一套口径；同一问题只保留一个真源位置 | 不同章节各写一套说法 |
| 逻辑要求 G | 输入输出闭环；写上游时要能落到下游，写下游时要能追到上游 | 只写消费，不写产出；只写结果，不写来源 |
| 逻辑要求 H | 例外必须说明它偏离的是哪条主规则 | 只写“例外如此”，不写它例外于什么 |

## 2. 层级规则

### 2.1 纵向排序规则

| 排序  | 纵向层   | 边界                 | 负责内容                      | 下层才能展开的内容        |
| --- | ----- | ------------------ | ------------------------- | ---------------- |
| 1   | 文档定义层 | 只定义文档定位与范围，不展开业务明细 | 这份文档管什么、不管什么、给谁看、用来解决什么问题 | 分类、对象、流程         |
| 2   | 文档导览层 | 只给总览，不展开细节规则       | 章节树、模块树、总图、总表入口           | 一级章节内容           |
| 3   | 正文主层  | 只承接正式口径，不写程序映射     | 一级分类、一级对象、一级流程、一级边界       | 二级对象、字段组、页面组、状态组 |
| 4   | 正文展开层 | 只展开上层已定义内容，不引入新主线  | 字段明细、页面明细、流程步骤、规则主体       | 例外、补充、映射         |
| 5   | 规则例外层 | 只处理偏离主规则的情况        | 例外条件、特殊场景、限制条件            | 附件、映射            |
| 6   | 附件映射层 | 只承接正文不适合展开的大表或程序对接 | 明细表、执行包、程序字段映射表、接口映射表     | /                |

### 2.2 纵向层级展开矩阵

| 排序 | 层级（边界 / 负责内容） | 子层级（边界 / 负责内容） | 子子层级（边界 / 负责内容） |
| --- | --- | --- | --- |
| 1 | 文档定义层：定义文档定位、范围、目标；负责内容：文档管什么、不管什么 | 文档导览层：只做总览，不展开明细；负责内容：章节树、模块树、关系总图 | 正文主层：只写正式口径；负责内容：主结构、主分类、主流程 |
| 2 | 正文主层：只承接正式内容；负责内容：本文主结论 | 一级章节层：承接正文一级分类；负责内容：一级模块、一级对象、一级流程 | 二级章节层：承接一级章节展开；负责内容：二级对象、字段组、页面组、状态组 |
| 3 | 一级章节层：先定义对象和边界；负责内容：对象名称、对象归属、对象范围 | 二级章节层：先展开结构和内容；负责内容：字段、页面、状态、资料 | 三级章节层：先展开规则和例外；负责内容：主规则、判断规则、例外条件 |
| 4 | 正文主层：只保留正式口径；负责内容：正式结论 | 附件层：承接正文不适合展开的大表；负责内容：明细表、执行包、补充清单 | 映射层：承接程序对接内容；负责内容：程序字段映射表、接口映射表 |

### 2.3 横向排序规则

| 排序  | 横向块 | 边界             | 负责内容              | 备注      |
| --- | --- | -------------- | ----------------- | ------- |
| 1   | 名称块 | 只回答“这是什么”      | 名称、标题、对象名、模块名     | 不写规则    |
| 2   | 边界块 | 只回答“写到哪里为止”    | 范围、排除项、归属界线       | 不写明细    |
| 3   | 分类块 | 只回答“怎么分”       | 类别、层次、分支、模块树      | 不写例外    |
| 4   | 对象块 | 只回答“具体有哪些东西”   | 对象、档案、字段组、页面组、状态组 | 不写程序映射  |
| 5   | 明细块 | 只回答“对象具体长什么样”  | 字段、列、取数项、页面项、流程步骤 | 不抢规则位   |
| 6   | 规则块 | 只回答“怎么判断、怎么运作” | 主规则、判断条件、操作规则     | 不抢例外位   |
| 7   | 例外块 | 只回答“哪里不按主规则来”  | 例外、特殊场景、限制条件      | 必须挂靠主规则 |
| 8   | 补充块 | 只在必要时补充        | 输入输出、引用关系、附件入口    | 能省则省    |

### 2.4 横向信息排列矩阵

| 排序 | 横向项 | 边界 | 负责内容 |
| --- | --- | --- | --- |
| 1 | 名称 | 只回答“这一层叫什么” | 统一命名、统一口径 |
| 2 | 横向界限 | 只回答“这一层不写什么” | 边界、范围、禁止越层 |
| 3 | 负责内容 | 只回答“这一层承接什么” | 本层正式内容 |
| 4 | 输入输出 | 只在必要时补充 | 上游依赖、下游产出 |

## 3. 信息密度规则

| 规则 | 要求 | 不合格表现 |
| --- | --- | --- |
| 信息密度要求 A | 标题、表头、列名、正文都要承载信息 | 标题和表头只有装饰作用 |
| 信息密度要求 B | 一张表只回答一类问题；不要一张表同时承接两层以上逻辑 | 同一张表里同时混对象归属、页面关系、流程状态 |
| 信息密度要求 C | 优先使用总表或少量大表，不拆成过多碎小节 | 1.1.1、1.1.2 式碎展开太多 |
| 信息密度要求 D | 删除废话、重复话、解释性空话 | 靠说明文兜底而不是靠结构说话 |
| 信息密度要求 E | 关键条件必须显式写出 | 为了简短省掉必要条件 |
| 信息密度要求 F | 例子只保留最小必要量；例子要服务理解，不要变成正文主体 | 连续堆很多例子，反而冲淡规则 |
| 信息密度要求 G | 引用已有口径时，优先引用，不重复改写 | 同一规则在三处用不同话重写一遍 |
| 信息密度要求 H | 一段话、一张表应能直接支持下一步动作 | 读完还需要再开会解释 |

## 4. 内容编写规则

| 类型 | 规则 | 要求 |
| --- | --- | --- |
| 口径 | 正文口径规则 | 正文一律使用中文业务口径 |
| 字段 | 固定字段规则 | 固定字段用字段块包裹 |
| 程序字段 | 程序字段规则 | 英文程序字段不得出现在文档正文、正文表格、正文流程中 |
| 程序字段 | 映射规则 | 如需对接程序实现，必须单独维护《程序字段映射表》 |
| 程序字段 | 引用规则 | 正文末尾只引用《程序字段映射表》，不在正文中展开程序字段 |
| 流程 | 流程主线规则 | 如果文档涉及流程，优先写线性主线 |
| 流程 | 流程阶段规则 | 如果文档涉及阶段，先写正式阶段，再写阶段内规则，再写阶段例外 |
| 例外 | 标注规则 | 确认点、例外、高风险点必须显式标明 |
| 阶段 | mock 规则 | 如果某节只适用于 mock / 原型阶段，必须明确标注，不得与正式口径混写 |
| 正文与附件 | 附件规则 | 附件只承接正文不适合展开的大表、补充清单、映射清单，不替代正文主结论 |
| 页面与数据 | 输入输出规则 | 涉及页面或模块时，要能看出上游产出、下游消费；只在必要位置写，不到处重复 |

## 5. 禁止或尽量不写

| 类型 | 规则 | 说明 |
| --- | --- | --- |
| 禁止 | 在正文中直接混写英文程序字段 | 程序字段只允许出现在单独映射表 |
| 禁止 | 上层先出现下层信息，下层跳出上层定义 | 会直接破坏层级 |
| 禁止 | 把页面层、数据层、字段层、流程层揉在同一层表达 | 会直接破坏分类依据 |
| 禁止 | 一份文档里并存两套口径、两套分类标准或两套流程说法 | 会直接破坏唯一口径 |
| 禁止 | 没有主规则就先写例外 | 例外会失去挂靠对象 |
| 禁止 | 规则没有挂靠对象 | 会让读者不知道规则作用于谁 |
| 尽量不写 | 解释性兜底文字 | 能靠结构讲清楚，就不要靠说明补救 |
| 尽量不写 | 过多碎小节 | 能合并成总表或少量大表，就不要拆散 |
| 尽量不写 | 半成品表格、占位词、空列、`、、、`、`/` 滥用 | 会降低正式感和可执行性 |
| 尽量不写 | 模糊词，如“相关”“适当”“等等”“视情况而定” | 会削弱可执行性 |
| 尽量不写 | 为追求简短而省略关键条件 | 会把高密度写成低可用性 |
| 尽量不写 | 同一件事先写正文，后面又换一个说法重写一遍 | 会制造伪增量和二次歧义 |
