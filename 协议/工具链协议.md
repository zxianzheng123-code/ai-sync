# 工具链协议（已迁移）

> 定义所有技能在执行过程中使用后台助手、进度文件、技能调度等基础设施时必须遵守的规范。核心问题：后台任务怎么派、进度怎么追踪、技能之间怎么衔接。各 SKILL.md 引用本协议。

## 一、后台助手分派契约

后台工作（不需要你参与的）可以派后台助手执行。

**身份定义**：
- **分配者** = 总控角色（造工具总控、循环定时任务总控等使用路由调度员角色的技能）
- **执行者** = 被派发的下游技能（需求规划师、全栈编码师等）

```
分派时必须传入：
  ① 任务描述：明确说清楚要干什么
  ② 必要输入：前面做好的东西、参考文档、约束条件
  ③ 输出要求：产出什么格式、写到哪里
  ④ 边界约束：只读还是可写、不能碰什么

后台助手完成后，按以下标准格式回写到进度文件对应阶段的"执行结果"区域：
  · 执行状态：已完成 / 需返工
  · 产出文件：实际产出的文件路径列表
  · 执行摘要：1-3 句话概括做了什么、关键决策
  · 遗留问题：未解决的问题，无则写"无"
  · 完成日期：实际完成日期

总控读取结果，验收后向你汇报

禁止：
  · 后台助手直接面向你对话（所有沟通由总控统一转达）
  · 后台助手之间直接通信
  · 后台助手修改未授权的文件
```

## 二、进度文件管理

长流程任务使用进度文件跟踪状态：

```
进度文件职责：
  · 记录当前走到哪个阶段
  · 每个阶段的任务说明、验收标准（由分配方编制）
  · 每个阶段的执行结果（由执行方回填）
  · 每个阶段的验收结论（由分配方回填）
  · 产出物清单和路径

进度文件位置：项目目录下 design/progress.md

进度文件标准格式：
  文件头（必填）：
    · 项目名称：项目/任务名
    · 创建日期：首次创建日期
    · 当前阶段：编号 + 阶段名（如 "②产品设计"）
    · 总控状态：规划中 / 执行中 / 已完成

  每个阶段必须包含：
    分配者填写 ──
    · 任务描述：本阶段做什么
    · 输入物：前面做好的东西的路径
    · 预期产出：本阶段应输出什么
    · 验收标准：逐条列出

    执行者填写 ──
    · 执行状态：未开始 / 进行中 / 已完成 / 需返工
    · 产出文件：实际产出的文件路径列表
    · 执行摘要：做了什么、关键决策
    · 遗留问题：未解决的问题，无则写"无"
    · 完成日期：实际完成日期

    分配者回填 ──
    · 验证结论：通过 / 需返工（附原因）

阶段锁定门禁：
  · 前一阶段的执行状态必须为"已完成"且验收结论为"通过"，才能启动下一阶段
  · 未锁定时启动下一阶段 = 违规

更新时机：
  · 执行者完成后 → 立即将执行结果写入进度文件
  · 分配者验收后 → 立即将验收结论写入进度文件

跨对话恢复：
  · 每次对话开头读取进度文件
  · 从最后一个"已完成 + 通过"的阶段的下一阶段开始
  · 向你汇报当前状态
  · 对时效敏感的信息重新确认
  · 你确认后继续
```

## 三、技能调度

造工具总控调度下游技能时的规范：

```
调度流程：
  1. 读取当前阶段对应的 SKILL.md
  2. 跟你对话的步骤 → 总控直接按 SKILL.md 执行
  3. 后台工作步骤 → 派后台助手执行
  4. 阶段完成 → 验收 → 汇报 → 你确认 → 下一阶段

上下文传递：
  · 前一阶段产出物路径写入进度文件
  · 下游技能启动时从进度文件读取前面做好的东西
  · 不依赖对话记忆传递信息，一切以文件为准
```
