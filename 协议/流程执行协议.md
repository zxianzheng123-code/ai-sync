# 流程执行协议（已迁移）

> 定义所有技能在执行 Workflow 时必须遵守的步骤执行规则。
> 本协议只管"步骤怎么走"；不管"每步做什么内容"（那是各 SKILL.md 自己定的），也不管"怎么汇报和验收"（那是执行闭环协议管的）。

## 一、步骤不可跳过

SKILL.md 的 Workflow 中每个 Step/阶段定义了执行顺序。

- 必须从 Initialization 指定的起始步骤开始，按编号顺序执行。
- 当前步骤未完成，不得进入下一步骤。
- 例外一：SKILL.md 本身写了条件跳转（如 `IF ... → 跳过 Step X`），按 SKILL.md 走。
- 例外二：SKILL.md 本身写了循环回退（如 `回到 Step X`、`回到 Phase B`），按 SKILL.md 走，不算跳步。
- **跳步的定义**：未经 SKILL.md 授权的步骤跳过。SKILL.md 中写明的条件分支和循环回退都是合法路径。
- 不得自行发明跳转理由。

## 二、步骤不可合并

每个 Step/阶段是独立执行单元。

- 不得将两个或多个步骤合并为一步完成。
- 即使两个步骤的内容看起来相似或可以一起做，也必须分开做、分开汇报。
- "效率更高"不是合并步骤的合法理由。

## 三、步骤产出不可省略

每个步骤如果在 SKILL.md 中定义了输出格式或产出物：

- 必须完整产出，不得省略。
- 内部步骤（标注"内部"）也必须产出内部结论，在步骤标记中可见。
- "面向客户"步骤的产出必须展示给用户。

## 四、审查不可走形式

审查环节（引用通用审查协议的交叉检查）不得走形式。

- 通用审查协议中"必须找出问题"的规则在此重申：每轮每位专家至少指出 1 个具体问题，若确实无问题必须列出完整检查项清单。
- "显然通过"不是跳过审查的合法理由。
- "内容比较简单所以不需要审查"不是跳过审查的合法理由。

## 五、用户确认不可替代

步骤中标注了"面向客户"的确认环节：

- 必须等用户明确回复后才能继续。
- 不得因为"用户已经提供了信息"就自行判定某个需要用户确认的步骤已通过。
- 不得用"根据上下文可以推断用户的意图"跳过用户确认。

## 六、步骤执行标记

每完成一个步骤，必须在输出中包含一个步骤完成标记。

格式：

```
[Step X 完成] 步骤名 — 一句话结论/产出摘要
```

示例：

```
[Step 1 完成] 锁定问题 — 确认问题是"三家面料供应商选哪家"
[Step 2 完成] 边界门禁 — 确认属于判断类型，继续执行
```

## 七、执行前声明

每次开始执行一个技能的 Workflow 时，必须先输出执行声明：

```
本次执行遵守流程执行协议。步骤总数：[N]。起始步骤：Step [X]。
```

这个声明必须出现在第一个步骤的输出之前。

## 八、用户回放权

用户随时可以说"回放流程"，AI 必须立即列出当前已完成的所有步骤标记，供用户核对执行轨迹。

## 九、边界说明

- 本协议不覆盖步骤内部的具体做法——那是 SKILL.md 自己管的。
- 本协议不覆盖推进节奏和汇报格式——那是执行闭环协议管的。
- 本协议不覆盖审查的具体规则（轮数、全员确认等）——那是通用审查协议管的。本协议只要求"不得跳过审查"和"审查不得走形式"。
- 如果 SKILL.md 的 Workflow 中某个步骤包含条件分支（IF/ELSE）或循环回退，按 SKILL.md 逻辑走不算"跳步"，这是 SKILL.md 授权的路径选择。

## 十、与执行闭环协议的配合

两个协议各管各的：

- **本协议**管步骤的**执行顺序**：先后、跳转、标记
- **执行闭环协议**管步骤的**推进节奏**：汇报、验收、复盘

配合方式：

1. 每完成一个 Step，先打标记（本协议第六条）
2. 再按需汇报（执行闭环协议第三条）
3. 不是每个 Step 都需要验收——只有"面向客户"步骤和阶段切换时才触发执行闭环协议的验收流程
