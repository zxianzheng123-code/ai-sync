# 协议引用矩阵

> 本文件只统计 **现役技能**，且为现役技能清单的**唯一权威来源**。其他文档（依赖图、README 等）引用本矩阵的技能清单，不另行维护。

## 协议图例

| 协议 | 缩写 | 管什么 |
| --- | --- | --- |
| `角色协议` | `R` | 身份、底线、角色库索引（具体角色见 `角色/` 目录） |
| `总转化协议` | `C` | 输入如何转成答案、判断、方法、教材、工具这些最终成果 |
| `成题协议` | `F` | 题目有没有说清、缺口如何补、什么时候必须先拆 |

| `证据协议` | `E` | 信息检索、来源分级、交叉验证、时效和引用 |
| `判断协议` | `J` | 对比、诊断、取舍、单一推荐 |
| `方法协议` | `Met` | SOP、规则、流程、AI 执行规则这一类方法成果 |
| `教材协议` | `Txt` | 课本、训练资料、系统化经验文档这一类教材成果 |
| `执行闭环协议` | `D` | 推进、汇报、验收、遗留问题、复盘 |
| `工具链协议` | `T` | 后台助手、进度文件、技能调度 |
| `通用审查协议` | `A` | 交叉检查、全员确认、引用证据 |
| `流程执行协议` | `W` | 步骤顺序、不可跳步、不可合并、步骤标记 |

标记规则：
- `Y` = 当前技能显式引用
- `-` = 当前技能未显式引用

## 读表规则

### 一、谁最重

- `调研判断师` 现在是最重的最终成果技能，完整吃了“判断链”。
- `总转化器` 已经从“重入口”收窄成纯路由入口。
- `造工具总控`、`全栈编码师`、`部署上线师`、`代码维护师`、`通用任务引擎` 都是“执行链”重技能。

### 二、谁最轻

- `问题拆解师` 只吃 `R/C/F/D/A`，边界相对清楚。
- `方法提炼师` 和 `教材编写师` 已经不再完全同构，至少协议层边界已被拆开。

### 三、证据能力现在落在哪

- 当前显式吃 `证据协议` 的，已经扩展到 `答案澄清师`、`调研判断师`、`Skill编译器`、`循环定时任务总控`、`循环任务师`、`定时任务师`、`组合任务师`。
- 这意味着“信息检索”主要集中在需要查证、对比和选型的技能里，而不是铺满整个技能层。

## 当前发现

- `总转化器` 的主风险已经从“长胖”变成“会不会重新把最终成果判断吃回去”。后续应持续约束它只做路由，不做成果交付。
- `答案澄清师` 现在是新的轻量成果技能，但还需要观察它会不会重新滑向 `调研判断师`。
- `方法提炼师` 和 `教材编写师` 已经有协议层分界，但正文 workflow 仍然比较相似，后续还可以继续压。
- `循环定时任务总控` 现在是自动化唯一入口，后续要持续防止它滑向 `通用任务引擎`。
- `循环任务师`、`定时任务师`、`组合任务师` 的边界必须长期保持清楚，不能重新并回“大而全自动化技能”。
- `产品设计师`、`全栈编码师`、`部署上线师`、`代码维护师` 都不吃 `总转化协议`，说明它们已经不负责上层转化判断，定位是健康的。
- `通用任务引擎` 不吃 `总转化协议` 是合理的。它不是转化体系的一环，而是非专项任务兜底。

## 下一轮优先重构顺序

1. `答案澄清师` vs `调研判断师`
   现在答案和判断已经拆开，接下来要盯的是“答案成果会不会重新滑回判断成果”。
2. `循环定时任务总控` vs `通用任务引擎`
   自动化入口已经独立出来，接下来要盯的是“自动化总控会不会重新滑回万能兜底”。
3. `循环任务师` vs `定时任务师` vs `组合任务师`
   三者现在已经拆开，后续要防止边界重新糊掉。
4. `方法提炼师` vs `教材编写师`
   协议层已经拆开，但 workflow 仍然相似，后续还可以继续压差异。
5. `需求规划师` vs `造工具总控`
   这两个都吃 `F/J/D`，而总控额外再吃 `T`。后续要继续防止“总控抢需求规划的活”。

## 使用规则

- 新增或重构现役技能时，先更新本矩阵，再改 `SKILL.md`。
- 如果一个技能新增了某个协议引用，必须能一句话解释“为什么它非吃不可”。
- 如果两个技能协议引用完全一样，必须检查：它们的差异是不是只剩文案，没有结构性边界。
