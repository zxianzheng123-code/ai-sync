# 项目共享上下文 Playbook

每个业务项目固定一份 `context/project-context.md`，作为跨平台（Claude / Codex）传递上下文的单一真相源。不靠聊天传上下文。

---

## 1. 定位

| 项 | 说明 |
|----|------|
| 规则本体 | 放 ai-sync，作为共享协议 |
| context 文件 | 放在各业务项目自己的仓库里 |
| 启用方式 | 业务项目自己决定；启用就在 AGENTS.md / CLAUDE.md 加一行引用 |
| 不适用场景 | ai-sync 本身（工具库，无连续推进状态） |

## 2. 铁律

1. **接力棒，不是共享白板** — 同一时刻只有一个 agent 在写这份文件。交接时先读再写，不并行。
2. **读完先校验** — 读完后如果发现内容和实际状态（git log、文件现状）有矛盾，先修正 context 再开工，不带着错误上下文往下干。
3. **写入即精简** — 每次写入时，同时负责精简旧内容。正文部分（历史结论以上）不超过 30 行，超了就必须砍旧内容。

## 3. 读取规则

每次开始正式干活前，先读 `context/project-context.md`。不管是不是换了平台，都先看一眼。

## 4. 写入触发条件

用"动作锚定"判断，不靠主观感觉：

| 触发条件 | 示例 |
|----|------|
| 完成了一个阶段性产出（文件创建/修改/删除了） | "已建好数据库表结构" |
| 改变了原定计划（下一步和上次写的不一样了） | "原计划用方案 A，改为方案 B" |
| 遇到阻塞，需要另一方知道 | "接口还没就绪，等后端部署" |

不满足以上任何一条 → 不写。

## 5. 写入内容

四个字段，按固定模板写：

| 字段 | 写什么 | 必填 |
|----|------|------|
| 当前进展 | 做到哪了，一两句话 | 是 |
| 关键决策 | 本轮做了什么决策、为什么 | 有就写 |
| 下一步 | 接下来干什么 | 是 |
| 注意事项 | 卡住的原因、风险、约束 | 有就写 |

## 6. 文件模板

```markdown
## 当前进展

（一两句话）

## 关键决策

（有就写，没有留空）

## 下一步

（一两句话）

## 注意事项

（有就写，没有留空）

---

## 历史结论

（压缩后的旧内容，只留有用结论）
```

## 7. 排版规则

- 当前事情永远在模板的四个字段里，每次写入直接覆盖更新。
- 被替换下来的旧内容，提炼成一句结论压到"历史结论"区。
- 旧结论按时间倒序排，最新的在上面。
- 过程细节、调试日志、大段尝试记录一律不进这份文件。

## 8. 业务项目接入方式

在业务项目的 AGENTS.md 或 CLAUDE.md 末尾加一段：

```markdown
## 共享上下文

本项目启用跨平台共享上下文机制。

- 规则详见 ai-sync「临时资料/项目共享上下文Playbook.md」
- 上下文文件：`context/project-context.md`
- 每次开工前先读，有产出或变更时更新
```

## 9. 验证标准

- Claude 写完一轮后，Codex 只看 `context/project-context.md`，能接着往下干。
- Codex 写完一轮后，Claude 只看这份文件，也能知道当前状态和下一步。
- 连续五轮更新后，正文部分仍在 30 行以内，顶部一眼能看懂当前事情。
- 一次关键决策切换后，文件里能看出选了什么、为什么，不会被下一个 agent 翻回去。

## 10. 边界说明

| 机制 | 存什么 | 不存什么 |
|----|------|------|
| 本文件（project-context.md） | 任务进度、决策、下一步、阻塞点 | 用户偏好、长期规则、流水账 |
| .claude/memory/ | 用户偏好、反馈、跨对话持久信息 | 任务进度 |
| AGENTS.md / CLAUDE.md | 项目规则、开工顺序 | 临时状态 |
