# AI Tool Library

## 仓库定位

这是一个**个人自定义 AI 工具库**。

定位只有一句话：

**面向个人使用，持续沉淀可复用的 AI Skills、规则和方法资产，工具类型不设限制。**

这个仓库不是单一项目代码仓，也不是单一工作流产品。

它的核心职责是：

- 存放可复用的 Skills
- 存放全局规则
- 存放少量仓库级方法文档

## 使用方法

推荐使用方式只有一条：

**把 `AGENTS.md` 复制到任何项目根目录，即可把这套工具使用方式带进那个项目。**

这里的分工是：

- `AGENTS.md` 是对外使用入口
- 当前仓库是 `AGENTS.md`、规则和 Skills 的维护源头

## 根目录文件架构

根目录只表达**推荐结构**，不在这里枚举具体 Skill 目录：

```text
ai-sync/
├── README.md                         # 仓库总说明：定位、结构、维护原则
├── AGENTS.md                         # 可复制到任意项目的使用入口
├── rules/                            # 本项目构建者使用的内部规则
│   └── 通用文档写作规则.md
├── skills/                           # 全部 Skills，保持扁平
│   └── <skill-name>/
│       ├── SKILL.md
│       ├── references/
│       ├── scripts/
│       └── assets/
└── <repo-level-docs>.md              # 少量仓库级专题文档或讨论文档
```

## 各层职责

| 层   | 对象          | 职责                        | 边界                |
| --- | ----------- | ------------------------- | ----------------- |
| 1   | `README.md` | 解释仓库定位、推荐结构、维护原则          | 不写具体 Skill 正文     |
| 2   | `AGENTS.md` | 提供可复制到任意项目的 Skill 入口与触发规则 | 不替代 `SKILL.md` 正文 |
| 3   | `rules/`    | 存放本项目构建者使用的内部规则；当前写作规则文件是 `rules/通用文档写作规则.md` | 不作为项目使用者入口 |
| 4   | `skills/`   | 存放全部可复用 Skill             | 不混入无关的一次性项目资料     |

`skills/` 层补充规则：

1. 一项技能一个目录。
2. 尽量保持扁平，不做深层分类目录。
3. 技能内部只放和技能直接相关的内容。

## Skills 组织原则

这个仓库后续不按“项目类型”统一收口，而按“可复用能力”持续增加。

当前已经有四类能力：

- 流程与方法：`流水线指挥中心`、`PLAYBOOK编写师`、`Skill编译器`、`规则加载器`
- 内容与文档：`Obsidian Markdown编辑`、`Obsidian Bases编辑`、`JSON Canvas编辑`
- 工具与抓取：`Obsidian CLI`、`网页正文提取`
- 专题处理：`服装图片商品化`

后续新增 Skill 时，优先遵守这两个判断：

1. 这是可复用能力，还是一次性项目资料？
2. 这是独立 Skill，还是现有 Skill 的参考附件？

如果只是一次性项目资料，不进入 `skills/`。

## 维护规则

- 新 Skill 一律放在 `skills/` 根下，不单独再建二级分类目录。
- 是否能被触发，以 `AGENTS.md` 为准。
- Skill 正文写在 `SKILL.md`，长说明放进 `references/`。
- 本项目构建与维护时的写作规则，以 `rules/通用文档写作规则.md` 为准。
- 仓库级讨论、定位、迁移说明，优先写进根目录或后续单独归档，不混进 Skill 目录。

## 触发原则

现阶段这个仓库里的内容，统一按**主动触发**理解。

这里的主动触发，指的是：

- 由用户明确点名某个 Skill
- 或用户给出明确动作指令，等价于点名某个 Skill

这里不采用被动触发，指的是：

- 不因为语义“有点像”就自动启用某个 Skill
- 不因为仓库里存在某个 Skill 就默认接管任务

原因很简单：

这个工具库还不成熟，技能边界、文档口径和适用范围都还在收敛阶段。

所以当前规则应当保守：

**宁可少触发，也不要误触发。**

也就是说，后面的重点不再是继续堆实现，而是先把：

这三件事收稳。
