# 成题协议

> 定义所有技能在开始任何检索、判断、提炼、编写、执行之前，如何把原始输入压成“已成形的题”。本协议只管题目是否成形、成形到什么程度、缺什么；不管怎么找证据，也不管最后怎么下判断。

## 一、成题定义

“成题”指的是：当前请求已经被压成一个可继续处理的明确对象，而不是一团混在一起的意图。

一个题至少要回答清楚这 5 件事：

1. **题目对象**：这件事到底在说什么
2. **目标**：这次到底想拿到什么结果
3. **边界**：做什么 / 不做什么
4. **成功标志**：怎样算完成
5. **约束**：时间、范围、素材、限制条件

## 二、成题状态分级

| 状态 | 定义 | 处理方式 |
| --- | --- | --- |
| `未成题` | 目标混杂、对象不清、边界缺失 | 不进入下游，先拆解 |
| `半成题` | 目标大致清楚，但关键字段仍缺 | 补问缺口，补齐后再走 |
| `已成题` | 5 个关键字段已基本清楚 | 可进入对应终点 |
| `已锁题` | 关键字段已清楚且已获用户确认 | 可以执行，不再随意变更 |

## 三、缺口检查

开始处理前，按以下顺序检查：

```
1. 当前请求是不是只在说一件事？
   ├─ 否 → 未成题
   └─ 是 → 继续

2. 目标是不是唯一？
   ├─ 否 → 未成题
   └─ 是 → 继续

3. 边界、成功标志、约束是否足够支撑下游动作？
   ├─ 否 → 半成题
   └─ 是 → 已成题

4. 用户是否明确确认？
   ├─ 否 → 已成题
   └─ 是 → 已锁题
```

## 四、不同终点的成题最低要求

### 4.1 `答案`

至少要清楚：
- 用户到底在问什么
- 需要解释到什么程度

### 4.2 `判断`

至少要清楚：
- 要判断什么
- 比较对象或诊断对象是什么
- 重点维度是什么

### 4.3 `方法`

至少要清楚：
- 方法给谁用
- 用在什么场景
- 最终想稳定复现什么结果

### 4.4 `教材`

至少要清楚：
- 教材给谁看
- 看完之后要学会什么
- 这份教材讲的是哪套体系

### 4.5 `工具`

至少要清楚：
- 要解决什么业务问题
- 谁来用
- 第一版做到哪里为止

## 五、提问规则

补成题时，统一按以下规则问：

1. 一次只补 1 个关键缺口
2. 提问必须自带提案，不空问
3. 优先补高影响缺口
4. 用户已经说过的内容，不重复问

标准口径：

> 老板，我理解你现在想解决的是 `[题目对象]`，这次最想拿到的是 `[目标]`。我还差一个关键点：[缺口]。我建议先按 `[提案]` 锁住，行吗？

## 六、未成题退出规则

出现以下任一情况，直接判定“先拆解”：

- 一句话里同时要答案、判断、方法、教材、工具中的两个以上
- 用户自己明确表示“我也说不清”
- 比较对象、使用对象或目标结果一直漂移
- 补问 2 轮后仍无法锁住主目标

退出口径：

> 老板，这件事现在还没成形。我先不硬往下做，不然越做越乱。建议先拆开，锁住当前最重要的一题。

## 七、边界提醒

- 本协议只管“题有没有成形”，不管“证据够不够”
- 本协议只管“题该怎么锁”，不管“最后推荐哪个方案”
- `成题` 不是 `判断`
- `成题` 也不是 `执行`
