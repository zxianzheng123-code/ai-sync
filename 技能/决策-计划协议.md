# 决策-计划协议

> 定义所有技能在做决策和排计划时必须遵守的规范。核心问题：信息够不够做决定、怎么从多个选项中选最优、产出物达到什么标准。工具链相关规范见[工具链协议](工具链协议.md)。各 SKILL.md 引用本协议。

## 一、信息质量与处理深度

做决策之前，先判断手头的信息够不够：

### 1.1 信息充分性判断

```
做决策前检查：
  1. 关键信息是否齐全？（对照摸底维度清单）
     ├─ 齐全 → 可以做决策
     └─ 不齐全 → 先补信息，不带着缺口做决定

  2. 信息可靠性够不够？（对照信息-认知协议）
     ├─ 关键信息有"未验证"标注 → 先验证或告知用户风险
     └─ 全部可靠 → 可以做决策

  3. 这个决策值得多深入？
     ├─ 影响大、难逆转 → 深度调研 + 多方案对比 + 互撕
     └─ 影响小、可逆转 → 快速判断 + 推荐方案 + 用户确认
```

### 1.2 深度匹配原则

不是每个决策都需要深度调研。根据决策的影响范围匹配处理深度：

| 决策影响 | 处理深度 | 例子 |
|----------|----------|------|
| **高影响、难逆转** | 深度调研 + ≥3 方案对比 + 互撕 + 用户拍板 | 技术选型、服务商选择、架构设计 |
| **中影响、可调整** | 快速调研 + 推荐方案 + 理由 + 用户确认 | 页面布局、模块优先级、风格方向 |
| **低影响、易逆转** | 直接推荐 + 理由 + 用户确认 | 命名、措辞、小细节 |

## 二、决策框架与流程设计

### 2.1 多方案决策流程

涉及多个选项时，统一按以下流程走：

```
Step 1：发散——列出候选方案
  · 至少 3 个方案（避免锚定偏差）
  · 包含"不做/维持现状"选项（避免行动偏差）

Step 2：评估——统一维度打分
  · 确定评估维度（来自用户需求 + 场景约束）
  · 每个方案在每个维度上给出评价和依据
  · 对比必须公平，不偏袒任何一方

Step 3：互撕——反方辩手找漏洞
  · 按通用审查协议执行互撕
  · 重点找：推荐理由站不站得住脚、有没有忽略的风险

Step 4：推荐——给用户一个明确答案
  > 老板，我推荐 [方案X]，因为 [理由]。
  > 其他方案的问题是：[简述]。
  > 你觉得行吗？

  · 不给用户抛选择题（用户不是专家）
  · 但推荐理由要说透，让用户能判断
  · 用户有异议 → 展开讨论 → 调整推荐
```

### 2.2 单一推荐原则

面向用户时，不甩一堆选项让用户自己选。默认给一个推荐方案 + 理由。只有在以下情况才给多选：

- 多个方案确实各有优劣、无法客观分高下
- 选择涉及用户个人偏好（风格、品味）
- 用户主动要求看所有选项

## 三、输出质量

### 3.1 验收标准编制

每个产出物必须有明确的验收标准，验收标准由任务分配方编制：

```
验收标准要求：
  1. 每条标准必须可验证——能用"是/否"回答，不能是"差不多就行"
  2. 标准来源必须可追溯——来自用户需求、上游产物、或技能规则
  3. 必须包含"用户确认"这一条——产出物最终要用户点头
```

### 3.2 检查清单模板

所有阶段收尾时，必须跑检查清单：

```
FOR EACH 检查项 IN 验收标准:
  检查结果：✅ 达标 / ❌ 不达标
  证据：[具体引用产出物中的位置或可验证的事实]

IF 全部 ✅ → 验收通过
IF 有 ❌ → 列出不达标项 → 告知用户 → 商量补救方案
```
