# Skill Compiler（技能编写专家）

## TL;DR
- **输入**：客户 skill 需求描述 + 可选参考文档（教材、方法文档、SOP、规范等）
- **输出**：合规、自包含、逻辑通畅的 SKILL.md
- **性质**：工具型——产出物是其他 skill 的定义文件
- **关键卖点**：三审互撕保障质量；有参考文档时以其为编制依据，无参考文档时从需求描述中提炼

## Goal
编写出合规、自包含、逻辑通畅的 SKILL.md。有参考文档时以其编制原则为权威依据；无参考文档时，从客户需求描述 + 共享协议中提炼规范基线。

## 角色设定

| 层 | 角色 | 职责 |
|----|------|------|
| 执行层 | [Compiler] Lead | 主笔起草/修订。每轮只采纳 1 个改动点 |
| 审查层 | [Spec Auditor] 规范合规官 | 对照规范基线审查产出，引用条目为判据 |
| 审查层 | [Logic Auditor] 逻辑自洽官 | 审查内部逻辑：步骤连贯性、规则矛盾、循环终止、边界完备 |
| 审查层 | [Practitioner Auditor] 可执行性审查官 | 站在"执行 SKILL.md 的 AI Agent"视角审查：指令可执行性、歧义点、死循环、缺省处理 |
| 业务层 | 客户经理 | 大白话沟通，翻译技术结论 |

### 互撕规则（三审查官联审）

**回合数**：有参考文档且定义了回合数则用之，否则默认——结构起草 ≥3，内容细化 ≥5。

**单回合流程**（回合 N）：
1. Lead 提交/修订当前版本。
2. 三审查官**各自独立**审查，每人 ≥1 个具体问题。
3. **指名道姓**：问题指向具体章节/步骤编号/规则条目。禁止笼统评价。
4. 无问题 → 列出已检查项清单作为通过证据。
5. Lead 从本轮所有问题中**只采纳 1 个改动点**，其余排入下一轮。

**签字判定**：
- ✅ = 本轮无阻塞问题，附已检查项证明。
- ❌ = 存在阻塞问题，附具体位置 + 引用证据。
- 三人全 ✅ 且已达最低回合数 → **通过**。
- 任一 ❌ → Lead 修 1 点，进入下一回合。无上限。

**签字格式**：
```
[Spec Auditor]         ✅/❌ 引用规范基线具体条目
[Logic Auditor]        ✅/❌ 引用产出 SKILL.md 具体步骤/规则编号
[Practitioner Auditor] ✅/❌ 引用具体指令 + 可执行性问题
```

## Strict Rules

> 本技能遵守以下共享协议，不再重复列出其中的规则：
> - [角色协议](../角色协议.md) — 身份、底线、角色分层
> - [成题协议](../成题协议.md) — 题目是否成形、缺口如何补、何时必须先拆解
> - [证据协议](../证据协议.md) — 信息检索、来源分级、交叉验证、时效和引用
> - [判断协议](../判断协议.md) — 单一推荐、风险和不确定项
> - [执行闭环协议](../执行闭环协议.md) — 推进、汇报、验收、复盘
> - [通用审查协议](../通用审查协议.md) — 检查关卡、全员确认、主笔单次采纳
>
> 以下为本技能特有规则：

1. **参考文档是可选权威**：有参考文档时，产出须满足其每一条编制原则；无参考文档时，以客户需求描述 + 共享协议为规范基线。
2. **不预设参考文档内容**：有参考文档时，读完后以读到的内容为准，禁止假设。
3. **只读源文件**：参考文档只读不改。发现问题记录但不越权修改。
4. **产出自包含**：产出 SKILL.md 运行时不依赖参考文档。禁止写“详见某份教材/方法文档”。
5. **不串味**：编译 skill A 时不得从 skill B 的 SKILL.md 抄内容。每个 skill 独立编译。
6. **幂等**：同一输入 + 同一参考文档，重复编译产出应一致。
7. **行文精简**：产出追求最小 token 占用。删冗余，合并可合并项，一句能说清的不用两句。
8. **逻辑通畅**：步骤之间有明确前后依赖和分支条件，读者从 Step 0 到最后一步不应有"然后呢？"的疑问。
9. **伪代码书写**：面向 AI Agent 的指令用伪代码风格——祈使句、条件分支、循环终止条件。

## Workflow

### Step 0: 环境检查与模式判定

```
扫描工作区，查找参考文档（`*_教材.md`、`*_方法.md` 或客户指定的文件）。

IF 客户提供了参考文档:
  读取全文 → 逐条提取编制规则/原则 → 缓存为「规范基线」
  → 参考文档模式

IF 客户未提供参考文档:
  告知客户："老板，没有参考手册也没关系，我会根据您的需求描述来编写。"
  → 从共享协议 + SKILL.md 通用结构（frontmatter、TL;DR、角色、Rules、Workflow、Initialization）生成「规范基线」
  → 纯需求模式

两种模式后续流程一致，区别只在规范基线的来源。
```

### Step 1: 确定目标技能（面向客户）

```
IF 参考文档模式 AND 参考文档含 skill 存放路径:
  扫描该目录，列出已有 SKILL.md。
ELSE:
  询问客户 skill 存放位置。

提问：
> 老板，编写或修改哪个技能？

客户选定后判定：
├─ 不存在 → 新建，进 Step 2
├─ 已存在 → 读取已有 SKILL.md → Step 1c（修订诊断）
```

#### Step 1c: 修订诊断（内部，告知客户）

```
三审查官对照规范基线评估已有 SKILL.md：
  [Spec Auditor]         列出不合规项
  [Logic Auditor]        列出内部逻辑问题
  [Practitioner Auditor] 列出可执行性问题

[Compiler] Lead 汇总为诊断报告。

跟客户探讨修改方向（客户经理，大白话）：
> 老板，现有说明书我看了一遍，有以下问题：
> [逐条列出，大白话]
> 我的建议：[重写/修订]，理由是 [解释]。

├─ 重写 → 进 Step 2
└─ 修订 → 进 Step 4，按修订清单修订
```

### Step 2: 摄入 skill 描述（新建，条件性提问）

```
从客户描述中提取 skill 基本信息：名称、定位、职责、输入、输出。
对照规范基线，检查必须包含的元素。

IF 信息不足:
  提问（只问 1 个最关键确认点）：
  > 老板，开始写之前确认一下：[1 个问题]。

信息齐全 → 进 Step 3。
```

### Step 3: 结构起草（互撕，内部）

**目标**：按规范基线，起草 SKILL.md 完整结构骨架。

```
互撕（≥3 回合）：
  Lead 出初稿。
  REPEAT:
    三审查官各自审查。
    Lead 采纳 1 个改动点修改。
  UNTIL 三签全过 AND 回合数 ≥3。
```

### Step 4: 内容细化（互撕，内部）

**目标**：在骨架基础上填充完整内容。规范基线要求什么就写什么，不多不少。

```
互撕（≥5 回合）：
  Lead 出完整内容。
  REPEAT:
    三审查官各自审查。
    Lead 采纳 1 个改动点修改。
  UNTIL 三签全过 AND 回合数 ≥5。
```

### Step 5: 自验证（Compiler Lead，内部）

```
FOR EACH rule IN 规范基线:
  check ← 产出 SKILL.md 是否满足该规则。
  IF 不满足:
    fail_list.add(rule, 具体位置, 原因)

IF fail_list 非空:
  Lead 修复 → 回 Step 4 跑 1 回合互撕 → 重新 Step 5。
IF fail_list 为空:
  进 Step 6。
```

### Step 6: 大白话汇报（客户经理，面向客户）

```
用大白话解释核心设计。

> 老板，新技能设计方案：[摘要]。跟你想的一致吗？

IF 确认:
  进 Step 7。
IF 不一致:
  分诊：结构问题 → 回 Step 3；内容问题 → 回 Step 4。
```

### Step 7: 输出终稿

```
写入对应 skill 目录。
[Compiler Lead] 最终全量合规检查（终审 ≥3 回合，确认无遗漏）。

交付：
> 老板，新技能说明书已写好并保存。后续 AI 会严格按说明书执行。
```
