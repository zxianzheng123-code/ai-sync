# 总转化协议

> 定义所有技能共同在做的事情：把你的原始输入转成你要拿到的东西。本协议只管转化使命、成果分类、前置规则和交接口径；沟通方式、角色底线、信息判断等仍由其他共享协议负责。

## 一、总使命

所有技能最终都在服务同一件事：

- 把问题变成答案
- 把需要判断的事变成判断
- 把意图变成方法
- 把想法变成教材
- 把需求变成工具

## 二、五类最终成果

| 最终成果 | 定义 | 典型请求 | 什么时候算完成 |
| --- | --- | --- | --- |
| `答案` | 解释、澄清、认知补全 | "这是什么意思""常见原因有哪些" | 你拿到可直接理解的一句话结论或简明解释 |
| `判断` | 选型、对比、诊断、分析 | "哪个更合适""为什么会这样" | 给出明确推荐/判断 + 依据 + 风险 |
| `方法` | SOP、规则、流程、AI 执行规则 | "把这个固化成规则""以后按这个跑" | 文档可重复执行，步骤、条件、边界清楚 |
| `教材` | 课本、训练资料、系统化经验文档 | "写给新人学""整理成课本" | 文档可理解、可教学、可传承 |
| `工具` | 系统、页面、应用、自动化流程 | "做个工具""做个系统" | 进入造工具链并产出可执行工具需求 |

## 三、总前置规则

1. **题目还没说清先拆解**：题目是否说清的判断标准见[成题协议](成题协议.md)。未说清时不直接产出成果，先进入 `问题拆解师`。
2. **一次只锁一个主成果**：同一阶段只允许一个主成果。需要多个成果时，按顺序逐个完成，不混在一个技能里同时做。
3. **你要拿到的东西比动作优先**：先判断你当前最想拿到什么，再决定用什么动作。不能因为技能会"调研/写文档/拆解"就反推成果。
4. **未定的判断不沉淀**：核心结论没定之前，不进入 `方法` 或 `教材`。
5. **未定的问题不判断**：问题本身还乱的时候，不进入 `判断`。

## 四、成果判断顺序

按以下顺序判断当前请求应该转成什么：

1. 题目是否已经说清了？
   - 否 → `问题拆解师`
2. 你要的是解释、澄清、认知补全吗？
   - 是 → `答案`
3. 你要的是基于信息的选择、诊断、对比、分析吗？
   - 是 → `判断`
4. 你要的是以后能重复执行的方法、规则、流程吗？
   - 是 → `方法`
5. 你要的是以后能教给别人、传给新人、沉淀成体系的内容吗？
   - 是 → `教材`
6. 你要的是系统、页面、应用、自动化流程吗？
   - 是 → `工具`

## 五、成果到技能映射

| 最终成果 | 对应技能 |
| --- | --- |
| `答案` | `答案澄清师` |
| `判断` | `调研判断师` |
| `方法` | `方法提炼师` |
| `教材` | `教材编写师` |
| `工具` | `造工具总控` |

## 六、交接规则

| 上游 | 下游 | 必须交接什么 |
| --- | --- | --- |
| `问题拆解师` | `答案澄清师` | 题目定义、关键约束、当前瓶颈、推荐成果 |
| `问题拆解师` | `判断/方法/教材/工具` | 题目定义、关键约束、当前瓶颈、推荐成果 |
| `调研判断师` | `方法/教材` | 结论、依据、风险、待验证项 |
| `方法提炼师` | `工具` | 规则、流程、边界、异常处理口径 |
| `教材编写师` | `工具` | 背景、核心原理、场景、边界认知 |

## 七、边界提醒

- `答案` 不是缩水版 `判断`：不需要大量证据收集时，直接给答案。
- `判断` 不是缩水版 `方法`：做出判断不等于已经形成可执行方法。
- `方法` 不是缩水版 `教材`：方法重"照着做"，教材重"为什么这样做"。
- `教材` 不是更长的 `方法`：教材要帮助理解和传承，不只是堆步骤。
- `工具` 不是文档：只要目标是做成系统，就进入造工具链。
