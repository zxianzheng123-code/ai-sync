# 项目背景

## 给技术人员的提醒
- 以目前已经写好的程序为准；本文档只提供业务方向和限制条件。
- 保持简单：遇到说不清楚的地方，按"够用就行"的方式处理。
- 严格按照下面定好的产品样子和首版范围来做。

## 业务问题
- 要解决的问题：校服品牌的订货和零售全靠微信群对接学校、客服手工记账，效率极低、容易记错尺码和漏单，需要搬到线上解决
- 现在怎么凑合的：学校大货订单靠微信群+表格统计；家长零买靠加客服微信、发款式图报尺码转账、客服手工记录

## 使用者
- 主要使用者①：学校总务主任——代表学校统一提交大货采购订单
- 主要使用者②：学生家长——自助浏览本校校服、选尺码、下单付款
- 后台管理者：品牌方老板及客服——上架商品、设置学校专属尺码表和价格、处理订单发货、查看经营数据

## 产品形态
- 做在哪个端：微信小程序
- 首版范围：首版仅做微信小程序
- 视觉风格：简洁干净，蓝白配色，正式感
- 参考对象：有赞（取其商城能力，加入校服行业专属逻辑）

## 首版功能
- 模块①：学校绑定 — 让用户只能看到自己学校的校服 — 首次进入时选择并绑定所属学校
- 模块②：商品展示 — 让用户浏览本校可买的校服 — 查看本校校服的款式、图片、价格、专属尺码表
- 模块③：家长自助下单 — 家长自己完成购买全流程 — 选尺码→加入购物袋→提交订单→微信付款
- 模块④：学校大货订单 — 总务主任统一提交学校采购需求 — 按款式和尺码填写数量→提交订单（不走线上付款，线下对公转账后由品牌方在后台确认收款）
- 模块⑤：总务主任认证 — 确保只有真正的总务主任能下大货单 — 提交学校、姓名、职务信息，品牌方人工审核通过后开通大货下单权限
- 模块⑥：订单管理（买家端） — 让买家随时查看订单进展 — 查看订单列表、订单详情、物流状态
- 模块⑦：后台管理（卖家端） — 品牌方日常运营管理 — 上架和编辑商品、按学校设置专属尺码表和价格、处理订单和发货、查看总订单数和总金额（第一版只做最简统计）

## 业务死规矩
- 规矩①：学校之间数据严格隔离——A学校的家长绝对不能看到B学校的款式、价格和订单
- 规矩②：学校大货订单只有通过认证的总务主任才能下，其他人没有这个权限
- 规矩③：尺码必须使用每个学校的专属尺码表，不能使用通用尺码；不同学校的尺码表可以不同
