# 判断标准

> 定义调研判断工作流在做选型、对比、诊断、取舍和推荐时必须遵守的规则。本文件只管"如何从说清的题 + 已有证据走到判断"；不管怎么检索，也不管后续如何沉淀成方法或教材。

## 一、判断定义

「判断」指的是：

- 在多个方案里选一个
- 对一个现象给出原因判断
- 对一组数据给出解释和倾向
- 对一件事给出明确推荐

## 二、进入条件

进入判断前，至少满足：

1. 题目已说清
2. 判断对象明确
3. 关键证据已具备，或已明确哪些证据缺失

> 说明：如果证据收集退出时附带了"缺失清单"，本流程可以进入，但必须在风险项中逐条标注缺失维度的影响。如果缺失维度直接影响核心推荐，必须向你说明"这个推荐建立在不完整证据上"。

任一不满足：

- 题目不清 → 回 `成题` 阶段
- 证据不够 → 回 `证据收集` 阶段

## 三、判断类型

| 类型 | 典型问题 | 产出 |
| --- | --- | --- |
| `选项对比` | "A 和 B 选哪个" | 推荐方案 + 对比依据 |
| `问题诊断` | "为什么最近复购下滑" | 原因判断 + 可能性排序 |
| `方案选型` | "应该用什么方式做" | 方案推荐 + 风险 |
| `优先级排序` | "先做哪个" | 排序结果 + 理由 |

## 四、判断流程（6 步）

```
Step 1：列候选
  → 列出所有可能的方案/原因/选项

Step 2：锁评估维度
  → 确定从哪些角度来比较

Step 3：统一维度比较
  → 每个候选在同一套维度下打分/对比

Step 4：找不利的证据
  → 对当前倾向的方案，专门找反面的信息
  → 避免只看到好的一面

Step 5：给单一推荐
  → 综合以上，给出一个明确推荐

Step 6：补风险和不确定项
  → 标出核心风险、未验证的点、判断错误的代价
```

## 五、候选方案规则

1. 能比较时至少列 **3 个候选**
2. 必须包含 **"不做 / 维持现状"** 作为基线
3. 不准先锁一个方案再补理由

## 六、评估维度规则

评估维度必须来自：

- 你明确关心的点
- 当前场景的客观约束
- 下游执行真正会受影响的因素

禁止使用：

- 只因为容易写就选的维度
- 偷偷偏向某个方案的角度（比如故意选一个只有方案 A 强的维度来凑数）

## 七、单一推荐原则

面对你时，默认只给一个推荐。

标准口径：

> 老板，我推荐 `[方案X]`，因为 `[理由]`。其他方案的问题是 `[简述]`。

只有在以下情况才给并列多选：

- 方案确实打平
- 差异主要取决于个人偏好
- 你主动要求看全部方案

## 八、风险与不确定性

每个判断都要附带：

1. **核心风险** — 选了这个方案，最大的坑是什么
2. **仍未验证的点** — 哪些信息还没拿到、可能影响结论
3. **如果判断错了，代价有多大** — 错误成本评估

## 九、判断完成标准

判断完成必须同时满足：

- 推荐对象明确
- 推荐理由可追溯到证据
- 风险和不确定项已标出
- 你能看懂为什么这样判断
