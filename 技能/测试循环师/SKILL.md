# 测试循环师（Test Loop）

## TL;DR（一屏摘要）
- 输入：一个已有代码项目（任何技术栈）
- 输出：每轮 TEST-REPORT-RN.md + 修复代码 + 架构诊断，直到缺陷归零
- 性质：独立技能——可被造工具总控调度，也可单独触发
- 关键卖点：规划师动态生成测试计划，不依赖硬编码清单；代码+产品双视角全覆盖

## Goal
对一个已有项目进行全面体检：自动发现缺陷、追溯架构根因、修复重构、回归验证，螺旋迭代直到缺陷归零。规划师先理解项目全貌再输出动态计划，执行者拿着计划干活——不硬编码校验清单。

## 角色设定

> 角色行为规范见 [角色协议](../角色协议.md) 和各角色文件。

- **执行**：[工程主笔](../角色/工程主笔.md) — 本技能中称"测试主笔"，统筹测试循环全流程
  - 技能内细分执行角色：
    - [Planner] 测试规划师。读项目全貌、输出测试计划和 Agent 分工。使用 opus + Plan 模式。
    - [Architect] 架构诊断师。读综合报告，追溯 bug 的架构根因，输出重构方向。使用 opus。
    - [Reviewer] 代码审查员（N 个并行）。按规划师分工审查代码，输出分级问题清单。
    - [PM] 产品审查员（1 个，与代码审查并行）。以用户视角审查信息结构、操作动线、体验。
    - [Fixer] 修复工程师（N 个并行）。按模块分区在 worktree 隔离环境中修复代码。
- **审查**：[工程审查官](../角色/工程审查官.md) — 通用工程审查规则
  - 技能内细分审查角色：
    - [Verifier] 回归验证员。类型检查 + 业务规则复验 + 代码量对比。
- **沟通**：[技术翻译官](../角色/技术翻译官.md) — 所有面向客户的沟通，大白话、零专业黑话

## Strict Rules
> 本技能遵守以下共享协议，不再重复列出其中的规则：
> - [角色协议](../角色协议.md) — 身份、底线、角色分层
> - [工具链协议](../工具链协议.md) — 后台助手分派、进度文件管理
> - [执行闭环协议](../执行闭环协议.md) — 推进、汇报、验收、复盘
> - [通用审查协议](../通用审查协议.md) — 检查关卡、确认通过
> - [流程执行协议](../流程执行协议.md) — 步骤顺序、不可跳步、不可合并、步骤标记
>
> 以下为本技能特有规则：

1. **规划师自主读项目**：规划师必须先读项目文档和代码结构，自己发现该测什么。禁止使用预设清单或硬编码校验项。
2. **Agent 数量动态决定**：每轮 Agent 数量由规划师根据项目规模、复杂度和残留问题动态决定。首轮通常多（3-5 个），后续逐轮收敛。
3. **worktree 隔离修复**：所有修复 Agent 必须在 worktree 隔离环境中工作，避免并行冲突。
4. **改完即检**：每改完一个文件必须跑该项目的类型检查命令（tsc/mypy/cargo check 等，视技术栈而定）。
5. **缺陷不归零不停**：P0 缺陷未归零必须继续下一轮迭代，不能跳过或降级。
6. **信息以文件为准**：迭代间信息传递以产出文件为准，不依赖对话记忆。每轮产出物（测试报告、架构诊断、修复记录）作为下一轮输入。
7. **理解后重写**：修复时先读完涉及的全部代码，理解设计意图后再改。不打补丁、不绕路。同一区域的 bug + 架构问题一起解决。
8. **双视角全覆盖**：每轮同时进行代码审查和产品审查，代码质量和用户体验并重。

## Workflow

> 以下步骤受[流程执行协议](../流程执行协议.md)约束：不可跳步、不可合并、每步必须产出步骤标记。

### Step 0: 项目摸底（面向客户）

> 老板，我先摸一下项目的底，看看文档、代码结构和技术栈，然后告诉你大概情况。

```
读取顺序：
  1. 项目根目录文件（README.md、CLAUDE.md、AGENTS.md 等）
  2. 业务文档（如有：需求文档、数据说明、业务流程等）
  3. 代码结构概览（目录树、schema/模型定义、API 路由、页面/组件）
  4. 现有测试（如有：测试文件、测试配置）
  5. git log 最近 20 条提交，了解近期变动

产出：项目概况摘要（技术栈、代码规模、模块划分、用户画像）
```

向客户汇报摸底结果：
> 老板，项目情况摸清了：[技术栈]，[代码行数]，[N 个功能模块]。接下来我开始第一轮体检。

---

### Step 1 / S1: 测试规划（内部，Planner，opus + Plan）

```
输入：
  · 首轮：Step 0 的项目概况
  · 第 2 轮起：上轮架构诊断报告 + 修复记录 + 综合测试报告

规划师必读：
  · 项目文档（README、业务文档、开发指南）
  · 代码结构（schema、API、页面、核心 lib）
  · 迭代输入（第 2 轮起）

规划师输出 6 个交付物：
  ① 测试维度清单 — 这个项目应该从哪些角度测
  ② Agent 分工建议 — 分几个 Agent、每个负责什么
  ③ 每个 Agent 任务描述 — 具体要审查什么、读哪些文件
  ④ 浏览器测试指引 — 哪些页面需要实际操作验证
  ⑤ 风险地图 — 哪些区域最可能出问题
  ⑥ 架构回看（第 2 轮起）— 上轮修复是否引入新风险

拆分策略（规划师从以下选一种或混合）：
  · 按技术层：类型系统 / API 层 / 业务逻辑 / 安全
  · 按业务域：按项目的功能模块划分
  · 按风险等级：高风险集中审查 / 低风险快速扫描
  · 混合（推荐）：安全单独一个 + 其余按业务域拆
```

### Step 2 / S2: 代码审查（内部，N 个 Reviewer 并行）

```
每个 Reviewer 的 Prompt 结构：
  你是一个代码审查员。
  【你的任务领域】{ 规划师输出的该 Agent 任务描述 }
  【项目上下文】{ 规划师输出的风险地图中相关部分 }
  【工作方式】
    1. 先读规划师指定的上下文文件
    2. 基于理解去审查代码，主动发现问题
    3. 思考「如果我是用户，这段代码会在什么场景下出问题？」
    4. 对每个发现，评估严重度和影响范围
  【输出格式】按 P0/P1/P2 分组，每个问题含：
    · 文件路径 + 行号
    · 问题描述
    · 原因分析
    · 影响范围
    · 建议修复方向
```

### Step 3 / S3: 产品审查（内部，PM，与 S2 并行）

```
PM 的 Prompt 结构：
  你是一个产品经理，审查一个系统的用户体验。
  【用户画像】{ 从项目文档中提取的用户信息 }
  【审查维度】
    · 信息层级：导航深度是否合理？用户能否 ≤3 步抵达目标页？
    · 操作动线：高频操作是否顺畅？主动线是否无断点？
    · 数据展示：KPI 数据是否可信？各页面数据是否一致？
    · 空状态/边界：无数据时是否有引导？筛选无结果时是否友好？
    · 一致性：同类页面布局是否统一？按钮文案/颜色语义是否清晰？
    · 快捷操作：是否有直达链接？报表是否可钻入明细？
  【工作方式】
    1. 读功能需求文档理解业务意图
    2. 通读每个页面代码，模拟用户操作动线
    3. 对比需求 vs 实现，找出差异
    4. 从"小白用户"视角评估每个操作的可理解性
  【输出】按体验影响分级：P0(卡死) / P1(别扭) / P2(可优化)
```

### Step 4 / S4: 综合报告（面向客户）

```
合并 S2 代码审查 + S3 产品审查的发现：
  1. 去重（同一问题被多个 Agent 发现的合并）
  2. 统一分级（P0 致命 / P1 严重 / P2 改进）
  3. 排优先级（先修 P0，再修 P1，P2 留到最后）
  4. 输出 TEST-REPORT-RN.md（N = 轮次编号）

报告结构：
  · 总览统计表（P0/P1/P2 数量）
  · P0 问题明细（每个含文件路径、描述、分析、影响）
  · P1 问题明细
  · P2 问题明细
  · 修复优先级排序
  · 极端场景清单
```

> 老板，第 [N] 轮体检报告出来了。发现 [X] 个 P0、[Y] 个 P1、[Z] 个 P2。我把明细都整理好了，接下来开始诊断根因和修复。

### Step 5 / S5: 架构诊断（内部，Architect，opus）

```
架构诊断师读取综合报告，追溯 bug 的架构根因。

重点关注：
  · 数据是怎么流动的？（从创建到消费的完整链路）
  · 状态是怎么变化的？（关键业务状态机）
  · 有没有重复逻辑散落在多处？
  · 有没有缺失的统一层？（如校验层、计算层、权限层）

输出：
  ① 业务-代码映射：哪段业务逻辑对应哪些文件
  ② 架构问题假设：哪些 bug 是同一个架构缺陷导致的
  ③ 修复 Agent 分工：按模块分区，明确每个 Agent 负责什么
  ④ 重构方向建议：不只修 bug，还要消除根因
```

### Step 6 / S6: 修复重构（内部，N 个 Fixer 并行，worktree 隔离）

```
每个 Fixer 在 worktree 隔离环境中工作。

修复原则：
  · 先读完涉及的全部代码，理解设计意图
  · 理解后重写，不打补丁
  · 同一区域的 bug + 架构问题一起解决
  · 每改完一个文件跑类型检查
  · 如果修改涉及数据模型，必须同步更新相关文档

Fixer 回写格式（按工具链协议）：
  · 执行状态：已完成 / 需返工
  · 产出文件：实际修改的文件路径列表
  · 执行摘要：改了什么、为什么这么改
  · 遗留问题：无则写"无"
```

### Step 7 / S7: 回归验证（面向客户）

```
验证项：
  ① 类型检查（sonnet）— 编译零错误
  ② 业务规则复验（opus）— 只验证本轮修改涉及的业务规则
  ③ 代码量对比（sonnet）— 确认未膨胀，统计引入/删除行数
  ④ 浏览器复测（主对话）— 按风险地图优先测高风险区

判定：
  IF P0 == 0 AND P1 == 0:
    → 体检完成，向客户汇报总结
  ELSE:
    → 回到 S1，下一轮计划吸收本轮架构诊断
```

> 老板，第 [N] 轮修复完成。[修复概况]。[是否需要下一轮]。

## 迭代间信息传递

| 产出物 | 由谁产出 | 谁在下轮消费 | 消费方式 |
|--------|---------|-------------|---------|
| 架构诊断报告 | Architect | Planner | 识别残留风险 → 生成针对性测试 |
| 修复记录 | Fixer | Planner | 了解改了什么 → 回归测试覆盖 |
| 综合测试报告 | 主对话 | Architect | 新 bug → 追溯架构根因 |
| 回归验证结果 | Verifier | 主对话 | 决策是否需要下一轮 |
| 产品体验报告 | PM | Fixer | 体验问题与代码问题合并修复 |

## Agent 选型参考

| 任务类型 | 模型 | 理由 |
|---------|------|------|
| 规划类（全局视野+深度理解） | opus + Plan | 需要综合判断，输出动态计划 |
| 代码审查（按计划审查代码） | opus / sonnet | 业务复杂用 opus，模式匹配用 sonnet |
| 产品审查（用户体验） | opus | 需要模拟用户思维，理解业务意图 |
| 修复类（改代码） | opus + worktree | 改代码必须用最强模型，worktree 隔离 |
| 回归验证（类型检查等） | sonnet | 快速验证，不需要深度理解 |
