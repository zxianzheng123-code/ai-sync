## 执行模式
Master Agent 负责调度和验收，不亲自干活。每个步骤启动 SubAgent 执行。

## 工作人员
- 路由调度员（→ 技能/人设库/路由调度员/人设.md）

## Workflow

### Step 0：加载上下文（Master Agent）
执行 脚本/加载上下文.sh

### Step 1：任务分析
- 执行者：路由调度员（SubAgent）
- 目标：判定任务形态——循环任务（围绕同一目标持续推进直到退出条件）、定时任务（按时间触发一次运行）、还是组合任务（两者同时成立）
- 成功标准：任务形态判定明确；如用户混说则先用大白话解释三者差异再给出判定
- 输入：用户自动化任务需求
- 输出：任务形态判定 + 判定理由

### Step 2：路由（循环/定时/组合）
- 执行者：路由调度员（SubAgent）
- 目标：补齐公共规格字段（任务目标、平台与触发方式、输入来源、输出物、状态文件/状态源、单次运行边界、停止条件、防重策略、权限边界、验证方式、通知方式），进行平台选型
- 成功标准：公共规格字段全部补齐；平台主推荐已选定（只给1条）
- 输入：Step 1 的任务形态判定 + 用户需求
- 输出：规格卡草稿 + 平台推荐 + 路由目标（循环任务/定时任务/组合任务）

### Step 3：分发
- 执行者：Master Agent
- 目标：将规格卡和上下文派发给对应下游技能
- 成功标准：下游技能接收到完整的交接信息（任务目标、当前平台、权限边界、状态来源、风险点、用户偏好）
- 输入：Step 2 的路由结果
- 输出：
  - 循环任务 → 转给 循环任务 技能
  - 定时任务 → 转给 定时任务 技能
  - 组合任务 → 转给 组合任务 技能
  - 如发现实质是代码测试修复 → 转给 测试循环 技能

### Step 4：验证
- 如有 脚本/验证/ 下的脚本 → 自动执行
- 如有 作业指引/交付自查清单.md → 逐项检查
- 额外检查项：
  - [ ] 任务形态判定正确
  - [ ] 公共规格字段全部补齐
  - [ ] 平台选型合理
  - [ ] 下游回传的方案包完整（规格卡+主推荐+备选+风险+prompt/命令草稿）
- 全部通过才进入交付

### Step 5：交付审核（Master Agent）
汇总下游回传结果，统一输出交付包（任务规格卡 + 主推荐方案 + 备选方案对比 + 可直接运行的prompt/命令草稿 + 推荐理由），等用户确认。
