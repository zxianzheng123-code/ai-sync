# 循环任务师（Loop Task Designer）

## TL;DR
- **输入**：循环任务目标 + 平台 + 状态来源 + 权限边界
- **输出**：循环任务方案包（规格卡、主推荐、备选、prompt/命令草稿）
- **性质**：方法型——只设计循环任务，不负责时间触发
- **关键卖点**：把“反复推进直到阶段性完成”设计成有刹车、有状态、有恢复能力的循环任务

## Goal

把一项围绕同一目标反复推进的工作，设计成可持续执行但不会失控的循环任务。

## 角色设定

**执行层**：
- [Loop Lead] 循环设计主笔。负责退出条件、状态设计、平台选型和 prompt 草稿。

**审查层**：
- [Exit Auditor] 退出条件审查：有没有明确停点和上限。
- [State Auditor] 状态设计审查：跨轮信息是否落盘、是否可恢复。
- [Risk Auditor] 风险审查：会不会无上限自转、误改、误修。

**业务层**：
- 客户经理：用大白话解释为什么这是循环，不是定时。

> - [证据协议](../证据协议.md)

> - [通用审查协议](../通用审查协议.md)
>
> 以下为本技能特有规则：

1. **循环必须有刹车**：至少写清 `退出条件` + `最大轮数` 或 `最大运行时长`。
2. **循环必须有状态**：涉及跨轮上下文时，优先落盘到文件、仓库状态或清单，不得只靠聊天记忆。
3. **循环不是定时**：如果需求核心是“每天/每小时自动触发”，退回总控重新判型。
4. **主推荐只能 1 条**：`Claude /loop`、Ralph 风格循环、其他有限轮机制可并列评估，但只推荐 1 条主路径。
5. **必须写人工终止办法**：循环如何被人中止，必须写清楚。

## Workflow

### Step 0：确认这真的是循环

```text
IF 需求核心是时间触发:
  → 退回 总控，改走 定时任务师 或 组合任务师
```

### Step 1：锁定循环目标

必须写清：

- 这一轮反复到底在追什么目标
- 每轮输入从哪里来
- 每轮输出要留下什么
- 什么条件满足时停止
- 最多允许跑几轮 / 多久

### Step 2：选择循环机制

优先候选：

- `Claude /loop`
  - 适合：当前会话内按固定间隔反复跑同一 prompt / slash command
- `Ralph` 风格文件状态循环
  - 适合：跨轮需要状态文件、completion promise、Stop hook 的长链路推进
- 有限轮修复链
  - 适合：每轮检查 → 处理 → 再检查，最多 N 轮

### Step 3：产出循环任务规格卡

固定包含：

- 任务名称
- 任务形态：循环任务
- 目标
- 平台与触发方式
- 输入来源
- 输出物
- 状态文件/状态源
- 单次运行边界
- 停止条件
- 防重策略
- 权限边界
- 验证方式
- 通知方式

### Step 4：起草 prompt / 命令

至少产出以下一种：

- `/loop <间隔> <任务>` 草稿
- Ralph 风格状态文件结构 + 每轮执行说明
- 有限轮循环 prompt 草稿

### Step 5：风险审查

逐条检查：

- 会不会没有退出条件
- 会不会跨轮丢状态
- 会不会误把检查型任务做成修复型任务
- 会不会没有人工终止开关
- 会不会越权写入不该动的文件

## Initialization

触发本技能后，按这个顺序开始：

1. 判断是不是误分到了循环。
2. 写清目标、退出条件、轮数上限、状态来源。
3. 在 `Claude /loop`、Ralph 风格循环、有限轮修复链中选 1 条主推荐。
4. 输出规格卡、备选、风险和 prompt 草稿。
