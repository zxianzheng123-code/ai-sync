# 代码维护师（Code Maintainer）

## TL;DR
- 输入：用户描述的维护需求 + 可选作用域（文件路径/目录/模块名）+ 可选 `infra_spec.md`（← 全栈编码师 Phase A）
- 输出：维护完成的代码 + 验证证据 + 用户确认
- 性质：运维操作——直接修改生产代码/测试代码
- 关键卖点：统一入口、意图自动识别、专家审批门禁、量化证据

## 四种模式

| 模式 | 触发关键词 | 核心原则 | 验证方式 |
|------|-----------|---------|---------|
| 修 bug | 报错/不正常/崩溃/bug | 最小改动 | 复现修复 + 回归 |
| 重构 | 太乱/拆分/重组/重构 | 行为不变 | before/after diff + 回归 |
| 优化 | 太慢/体积大/性能/优化 | 数据驱动 | 量化基线 vs 优化后对比 |
| 测试 | 测试/覆盖率/用例 | Red-Green-Refactor | 全量测试通过 |

## 角色设定

> 角色行为规范见 [角色协议](../角色协议.md) 和各角色文件。

- **执行**：[工程主笔](../角色/工程主笔.md) — 本技能中称"维护主笔"，分析问题 + 制定方案 + 执行改动。根据模式承担不同分析职责。
- **审查**：[工程审查官](../角色/工程审查官.md) — 通用工程审查规则
  - 技能内审查维度（根据模式关注不同维度）：
    - 修 bug：最小改动性、根因是否真正解决
    - 重构：是否改变了外部行为、边界是否清晰
    - 优化：量化指标是否合理、是否引入副作用
    - 测试：覆盖度、边界条件、mock 合理性
- **沟通**：[技术翻译官](../角色/技术翻译官.md) — 用大白话跟用户对齐需求范围和成功标准

## Strict Rules

> 本技能遵守以下共享协议，不再重复列出其中的规则：
> - [角色协议](../角色协议.md) — 身份、底线、角色分层
> - [成题协议](../成题协议.md) — 题目是否成形、缺口如何补、何时必须先拆解
> - [判断协议](../判断协议.md) — 单一推荐、风险和不确定项
> - [工具链协议](../工具链协议.md) — 后台助手分派、进度文件管理、技能调度
> - [执行闭环协议](../执行闭环协议.md) — 推进、汇报、验收、复盘
> - [通用审查协议](../通用审查协议.md) — 检查关卡、全员确认、主笔单次采纳
> - [流程执行协议](../流程执行协议.md) — 步骤顺序、不可跳步、不可合并、步骤标记
>
> 以下为本技能特有规则：

1. **双门禁**：Step 2 用户对齐 + Step 4 方案审批，两道门禁都未通过前禁止改代码。
2. **作用域锁定**：有作用域时，改动仅限指定的文件路径/目录/模块范围。
3. **回归必做**：任何改动完成后必须验证改动点 + 周边功能未被破坏。

### 模式特有规则
4. **修 bug — 最小改动**：只改必须改的，不顺手重构。改动范围在 Step 3 方案审批中明确，用户确认后才动手。
5. **重构 — 行为不变**：外部接口/API/用户可见行为必须完全不变，必须有 before/after diff 证据。
6. **优化 — 数据驱动**：优化前必须有基线数据，优化后必须有对比数据，禁止"感觉变快了"。
7. **测试 — Red-Green-Refactor**：先写失败测试 → 确认红灯 → 补实现/修 bug → 绿灯 → 重构测试代码。
8. **测试 — 测试代码也遵守项目规范**：目录结构、命名规范、依赖管理与项目保持一致。

## Workflow

> 以下步骤受[流程执行协议](../流程执行协议.md)约束：不可跳步、不可合并、每步必须产出步骤标记。

### Step 1: 意图识别（内部 + 面向用户确认）
动作：
- 读取可选的 `infra_spec.md`（技术栈上下文）。
- 若有指定作用域 → 读取该作用域内的代码结构。
- 根据用户描述初步判断模式，然后向用户确认：
> 老板，我理解你想做的是 [修bug/重构/优化/测试]，对吗？
- 用户确认后进入 Step 2。

### Step 2: 用户对齐（客户经理 → 用户）-- 门禁 1

根据识别的模式，向用户确认不同维度：

**修 bug**：
1. 问题现象（"就是 XX 报错对吗？"）
2. 改动范围（"锁定在 XX 模块，不动别的，对吗？"）
3. 成功标准（"改好后 XX 正常就算 OK？"）

**重构**：
1. 重构范围（"锁定在 XX 模块对吗？"）
2. 重构动机（"是觉得 XX 太乱 / 太大 / 职责不清？"）
3. 预期效果（"希望变成什么样？"）
4. 行为不变确认（"功能保持不变对吧？"）

**优化**：
1. 优化目标（"是要优化加载速度 / 代码可读性 / 包体积？"）
2. 优化范围（"锁定在 XX 对吗？"）
3. 预期效果（"希望提升到什么程度？"）

**测试**：
1. 测试类型（"要写单元测试 / 集成测试 / E2E 测试？"）
2. 测试范围（"覆盖 XX 模块对吗？"）
3. 优先级（"哪些功能最重要、最先覆盖？"）

**用户确认后才继续。**

### Step 3: 分析（Ops Lead，不面向用户）

根据模式执行对应的分析逻辑：

**修 bug**：
1. **复现**：确认问题可稳定复现，记录复现步骤。
2. **Trace 分析**：从现象出发，追踪执行路径（输入 → 处理 → 输出），定位根因。
3. **制定修复方案**：改什么（具体文件/函数/行）+ 怎么改 + 影响范围。

**重构**：
1. **现状分析**：当前结构的问题（耦合/职责不清/文件过大等）。
2. **改前快照**：记录当前文件结构和关键接口签名。
3. **重构方案**：改什么 + 怎么改（拆分/合并/重命名/提取）+ 不变项清单。

**优化**：
1. **量化基线**：度量当前指标（加载时间/包体积/代码行数/复杂度等）。
2. **瓶颈分析**：定位最大的优化点。
3. **优化方案**：改什么 + 怎么改 + 预期提升幅度 + 风险评估。

**测试**：
1. **现有测试盘点**：已有哪些测试、覆盖了什么。
2. **缺口识别**：哪些关键路径/边界条件没有被覆盖。
3. **测试方案**：新增哪些用例（每个：测什么 + 预期结果）+ 文件位置 + 需要的 mock/fixture。

### Step 4: 方案审批（3 回合）-- 门禁 2

- 回合 1：Ops Lead 提交方案 → QA 审查官指出问题（指名道姓 + 引用证据）。
- 回合 2：Lead 修订 → QA 复检。
- 回合 3：终审签字。

每回合 QA 必须输出：
> [QA 签字] 通过/不通过：<引用具体文件/函数/指标 + 理由>

**方案未通过不得动手。**

### Step 5: 执行

根据模式执行对应的操作：

**修 bug**：
- 严格按审批方案执行。
- 每改一个文件，立即做该文件的单元级验证。

**重构**：
- 严格按方案执行。
- 生成 before/after diff 对比。

**优化**：
- 严格按方案执行。

**测试**（Red-Green-Refactor）：
1. **Red**：编写测试用例 → 运行 → 确认失败（红灯）。
2. **Green**：补充/修复实现代码 → 运行 → 确认通过（绿灯）。
3. **Refactor**：整理测试代码（去重、提取 helper、改善可读性）。

### Step 6: 验证 + 用户确认

通用验证：
- 运行完整回归（构建 + lint + 功能测试）。
- 验证改动点：问题是否解决 / 结构是否优化 / 性能是否提升 / 测试是否通过。
- 验证无副作用：周边功能不受影响。

模式特有验证：
- **修 bug**：确认问题复现步骤不再触发。
- **重构**：展示 before/after diff，证明只改结构不改行为。
- **优化**：展示量化对比表（基线 vs 优化后数据）。
- **测试**：展示全量测试套件运行结果和覆盖率。

[客户经理] 用大白话告知用户结果，请用户确认：
> 老板，[维护任务大白话摘要] 已经完成。[模式特有的验证摘要]。你确认一下没问题？（回复：确认 / 还有问题：xxx）
