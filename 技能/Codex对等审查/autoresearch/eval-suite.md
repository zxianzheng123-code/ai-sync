# Codex对等审查 — 评估套件

## 测试输入

### 场景 1：代码方案审查
> 请用 Codex对等审查 技能审查以下方案：我计划把项目中所有的 shell 脚本从 bash 迁移到 zsh，理由是团队都用 macOS，zsh 是默认 shell。迁移方式是直接把 shebang 从 #!/usr/bin/env bash 改成 #!/usr/bin/env zsh，其余不动。

### 场景 2：架构设计审查
> 请用 Codex对等审查 技能审查以下方案：我打算把所有技能的 SKILL.md 合并成一个大的 ALL_SKILLS.md 文件，这样 Claude 只需要加载一个文件就能获取所有技能定义，减少文件读取次数，提高效率。

### 场景 3：SKILL.md 审查
> 请用 Codex对等审查 技能审查以下方案：我新写了一个"自动日报"技能，流程是每天下午 5 点自动扫描 git log，生成当日工作摘要，直接发到用户的飞书群里，不需要用户确认。

## 评估标准

EVAL 1: Codex 实际调用
问题: 产出中是否包含真实的 codex exec 命令执行及其返回结果？
通过: 能看到至少一次 codex exec 的命令和 Codex 的实际回复内容
失败: 没有 codex exec 命令，或只是模拟/假装了 Codex 的回复

EVAL 2: 每轮独立对话
问题: 每次调用 Codex 时是否都是全新的独立调用（没有复用上下文）？
通过: 每次 codex exec 都是完整的 heredoc 调用，包含所有必要上下文，不依赖之前的对话
失败: 有任何一次调用试图引用或依赖之前 Codex 对话的上下文

EVAL 3: 对等讨论质量
问题: Claude 对 Codex 的意见是否有明确的同意/不同意标记和具体理由，而非全盘接受？
通过: 对 Codex 的每条意见都有明确的 ✅ 或 ❌ 标记，不同意的附有具体技术理由
失败: 全盘接受 Codex 所有意见，或没有逐条回应，或不同意但没给理由

EVAL 4: 变更说明完整
问题: 最终交付是否包含完整的变更说明？
通过: 明确标注了哪些地方因 Codex 反馈修改、哪些地方坚持了原方案及理由、讨论轮次统计
失败: 缺少变更说明，或变更说明不完整（缺修改项、缺坚持项、缺统计）

EVAL 5: 流程完整性
问题: Step 1-6 是否都有执行，没有跳步？
通过: 能看到每个 Step 的执行痕迹（进度显示、输出内容），流程完整走完
失败: 跳过了任何一个 Step，或某个 Step 只有标题没有实际执行
