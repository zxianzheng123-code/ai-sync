# 协议引用矩阵

> 本文件只统计 **现役技能**。现役范围以 `README.md` 和 `正哥技能召唤书.md` 当前公开结构为准，不包含历史残留技能目录。

## 协议图例

| 协议 | 缩写 | 管什么 |
| --- | --- | --- |
| `角色协议` | `R` | 身份、底线、角色分层 |
| `总转化协议` | `C` | 输入如何转成答案、判断、方法、教材、工具 |
| `成题协议` | `F` | 题目是否成形、缺口如何补、何时必须先拆解 |
| `答案协议` | `Ans` | 解释、澄清、认知补全这一类答案终点 |
| `证据协议` | `E` | 信息检索、来源分级、交叉验证、时效和引用 |
| `判断协议` | `J` | 对比、诊断、取舍、单一推荐 |
| `方法协议` | `Met` | SOP、规则、流程、AI 执行规则这一类方法终点 |
| `教材协议` | `Txt` | 课本、训练资料、系统化经验文档这一类教材终点 |
| `执行闭环协议` | `D` | 推进、汇报、验收、遗留问题、复盘 |
| `工具链协议` | `T` | SubAgent、进度文件、技能调度 |
| `通用审查协议` | `A` | 互撕、签字、引用证据 |

标记规则：
- `Y` = 当前技能显式引用
- `-` = 当前技能未显式引用

## 读表规则

### 一、谁最重

- `调研判断师` 现在是最重的终点技能，完整吃了“判断链”。
- `总转化器` 已经从“重入口”收窄成纯路由入口。
- `造工具总控`、`全栈编码师`、`部署上线师`、`代码维护师`、`通用任务引擎` 都是“执行链”重技能。

### 二、谁最轻

- `问题拆解师` 只吃 `R/C/F/D/A`，边界相对清楚。
- `方法提炼师` 和 `教材编写师` 已经不再完全同构，至少协议层边界已被拆开。

### 三、证据能力现在落在哪

- 当前只有 `答案澄清师`、`调研判断师`、`Skill编译器` 显式吃 `证据协议`。
- 这意味着“信息检索”在官方现役技能里已经比较集中，没有分散污染其他技能。

## 当前发现

- `总转化器` 的主风险已经从“长胖”变成“会不会重新把终点吃回去”。后续应持续约束它只做路由，不做终点交付。
- `答案澄清师` 现在是新的轻量终点，但还需要观察它会不会重新滑向 `调研判断师`。
- `方法提炼师` 和 `教材编写师` 已经有协议层分界，但正文 workflow 仍然比较相似，后续还可以继续压。
- `产品设计师`、`全栈编码师`、`部署上线师`、`代码维护师` 都不吃 `总转化协议`，说明它们已经不负责上层转化判断，定位是健康的。
- `通用任务引擎` 不吃 `总转化协议` 是合理的。它不是转化体系的一环，而是非专项任务兜底。

## 下一轮优先重构顺序

1. `答案澄清师` vs `调研判断师`
   现在答案和判断已经拆开，接下来要盯的是“答案终点会不会重新滑回判断终点”。
2. `方法提炼师` vs `教材编写师`
   协议层已经拆开，但 workflow 仍然相似，后续还可以继续压差异。
3. `需求规划师` vs `造工具总控`
   这两个都吃 `F/J/D`，而总控额外再吃 `T`。后续要继续防止“总控抢需求规划的活”。

## 使用规则

- 新增或重构现役技能时，先更新本矩阵，再改 `SKILL.md`。
- 如果一个技能新增了某个协议引用，必须能一句话解释“为什么它非吃不可”。
- 如果两个技能协议引用完全一样，必须检查：它们的差异是不是只剩文案，没有结构性边界。
