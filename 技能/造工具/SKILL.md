## 执行模式
Master Agent 负责调度和验收，不亲自干活。每个步骤启动 SubAgent 执行。

## 工作人员
- 路由调度员（→ 技能/人设库/路由调度员/人设.md）

## 下游技能链

```
需求规划 → 产品逻辑 → 产品设计 → 全栈编码 → 部署上线
                                                ↑
                                           代码维护（后续维护时触发）
```

| 阶段 | 技能 | 核心产出 |
|------|------|----------|
| ① 搞清楚要做什么 | 需求规划 | `project_context.md` |
| ② 理清怎么跑通 | 产品逻辑 | `product_logic.md` |
| ③ 画出来长什么样 | 产品设计 | 交互原型 + 设计说明 + 交互地图 |
| ④ 写代码 | 全栈编码 | 可运行的代码 |
| ⑤ 放到网上能用 | 部署上线 | 线上可访问的产品 |

## 进度文件

进度文件是总控的核心管控工具，存放在项目目录下（`design/progress.md`）。

### 进度文件结构

```markdown
# 项目进度：[项目名称]

创建时间：[日期]
当前阶段：[阶段编号和名称]
总控状态：[规划中 / 执行中 / 已完成]

---
```

## Workflow

### Step 0：加载上下文（Master Agent）
执行 脚本/加载上下文.sh，强制读取所有引用内容

### Step 1：接活
- 执行者：路由调度员（SubAgent）
- 目标：判断项目状态，建立初始上下文
- 成功标准：明确用户要做什么，已有上游产物已摄入
- 输入：用户的工具/系统需求（任意形式：一句话想法、业务痛点、教材/方法文档、问题拆解清单）
- 输出：项目意图确认 + 上下文摄入完成
- 执行逻辑：
  1. 检查是否已有进度文件 → 有则汇报当前状态，问用户继续还是重来
  2. 检查是否有教材/方法文档或问题拆解清单 → 有则提取要点，向用户确认理解
  3. 无上游输入 → 听用户说想做什么，大白话复述确认

### Step 2：出执行计划
- 执行者：路由调度员（SubAgent）
- 目标：产出完整执行计划，用户确认后锁定
- 成功标准：计划覆盖所有必要阶段，每阶段有验收标准，用户已确认
- 输入：Step 1 确认的项目意图
- 输出：锁定的执行计划 + 创建进度文件
- 执行逻辑：
  1. 审查需求，拆解阶段（默认 5 个，特殊情况裁剪）
  2. 每阶段写明任务说明、预期产出、验收标准、优化建议
  3. 向用户展示计划，等确认
  4. 确认后创建进度文件

### Step 3：逐阶段调度执行
- 执行者：路由调度员（SubAgent）
- 目标：按计划逐阶段派发任务到下游技能，验收每阶段结果
- 成功标准：当前阶段验收全部达标，进度文件已更新
- 输入：进度文件 + 上游阶段产物
- 输出：当前阶段产物 + 更新后的进度文件
- 执行逻辑（每阶段循环）：
  1. 展示进度条，告知用户当前阶段
  2. 按对应技能 SKILL.md 执行：面向用户的步骤由 Master Agent 做，后台工作派 SubAgent
  3. 阶段完成后逐条验收标准对照执行结果
  4. 全部达标 → 向用户汇报成果，问是否继续
  5. 有不达标项 → 告知用户 + 提出补救方案
  6. 执行中发现计划需调整 → 暂停，告知用户，确认后更新进度文件

### Step 4：验证
- 如有 脚本/验证/ 下的脚本 → 自动执行
- 如有 作业指引/交付自查清单.md → 逐项检查
- 全部通过才进入交付

### Step 5：交付审核（Master Agent）
输出成品，等用户确认

全部阶段完成后：
> 全部完成了！你的工具已经上线了：[访问地址]。后面要改东西或修问题，跟我说就行。

### 后续维护
用户反馈问题或新需求时，Master Agent 先判断性质：
- 修 bug / 小改动 → 调度「代码维护」技能
- 新功能（小）→ 调度「代码维护」技能
- 新功能（大）→ 重新走对应阶段的流程
