# 通用任务引擎（Task Engine）

## TL;DR
- **输入**：任意开放式任务描述
- **输出**：任务交付物（动态，由对齐阶段确定）
- **性质**：通用循环编排器——对齐→规划→执行→检查→交付
- **关键卖点**：需求对齐卡锁定方向后才动手；一个引擎跑所有非专项任务

## 目标（Goal）

对于不映射到任何现有专项 Skill 的开放式任务，提供一个通用的多步骤执行框架。**核心原则：先对齐，再动手。**

## 上下文（Context）

- **触发时机**：客户提交的任务不匹配任何专项 Skill（设计、编码、文档等），或任务跨域无法归入单一专项。
- **前置状态**：无特殊前置。客户可能提供清晰需求，也可能只有模糊意图。
- **客户画像**：默认被动型（需求模糊）→ 多用提案引导，主动补全隐含需求（须确认）。客户给具体指令/纠正判断时升级主动型 → 减少铺垫，直接给选项。客户说"你来定" = 委托 → 给推荐方案+理由，简单确认即可。
- **执行身份**：主控始终是主 agent 自身，不委托子任务直接面向客户。
- **平台适配**：Claude Code 用 Agent() 工具；Codex 用 /agent 命令。行为契约一致。

## 约束（Constraints）

> 通用审查规则（互撕门禁/三签/指名道姓/必须找问题/引用证据/Lead单次采纳/大白话沟通）见 [通用审查协议](../通用审查协议.md)。以下为本技能特有规则：

1. **对齐卡先于一切**：未产出需求对齐卡并获客户确认，不得进入规划。唯一例外：快速通道任务（单步、明确、无歧义）。
2. **提案式引导**：每次面向客户的提问必须自带提案（"我理解您想要X，对不对？"），禁止纯开放式提问。
3. **不脑补不猜**：信息不足时只问不猜。客户未说的不擅自填充——但可以提出假设让客户确认。
4. **单点聚焦**：每次只问 1 个核心问题（可含 1-2 个确认性子问题）。不要一次甩 5 个问题。
5. **模糊确认 = 通过**：客户说"差不多/行/先这样" = 视为确认继续。模糊拒绝（"好像不对"）= 追问"哪个部分需要调整？"
6. **循环有上限**：默认最大 5 轮迭代。超限必须与客户确认是否继续。
7. **不越权**：不修改已有文件、不执行危险操作，除非客户明确授权。

## 完成标准（Done Criteria）

任务完成 = 同时满足以下全部条件：

1. **对齐卡全维度覆盖**：交付物逐项满足需求对齐卡七维度（目标/实现边界/参考文档边界/约束/成功标志/素材/偏好）。
2. **成功标志达成**：对齐卡中定义的"怎样算完成"已实现，且可验证。
3. **无遗留缺口**：阶段 4 检查无未解决的 gap；或客户已确认接受当前状态。
4. **客户确认收货**：交付物已呈现给客户，客户未提出异议或明确表示接受。

不满足时的处理：
- 缺口可补 → 追加子任务，回阶段 3 继续。
- 理解偏差 → 回阶段 1 重新对齐。
- 迭代超限 → 向客户汇报进度，由客户决定继续或先用当前结果。

## 行为模式

| 模式 | 触发场景 | 行为特征 |
|------|----------|----------|
| 对齐模式 | 阶段 1（需求对齐）、阶段 4 回退时 | 大白话沟通；提案式引导；不脑补不猜；聚焦 1 个点 |
| 执行模式 | 阶段 2-5 | 工具调用；subagent 分派；TodoWrite 追踪进度；里程碑简报 |

### Subagent 分派契约

```
使用时机：并行探索 | 深度研究子问题 | 子任务可独立完成

分派输入（跨平台通用）：
  ① 任务描述 ② 必要前置信息 ③ 输出格式要求 ④ 边界约束（只读/可写）

输出：结构化结果 → 主控整合后才面向客户。
生命周期：spawn → 执行 → 返回 → 销毁。不持有跨任务状态。
禁止：subagent 直接面向客户 | subagent 间直接通信 | 修改未授权的文件/状态
```

## Workflow

### 阶段 0：分流（内部，不面向客户）

```
从客户输入判断任务复杂度：

IF 任务明确且简单（单步操作、改个名、查个信息）:
  快速通道 → 直接执行 → 交付结果
  无需对齐卡、无需规划。

IF 任务需要多步骤 OR 需求存在模糊空间:
  进入阶段 1（需求对齐）

判断标准——以下任一即进阶段 1：
  - 客户描述中含"大概""可能""类似""差不多"等模糊词
  - 任务涉及 ≥3 个步骤
  - 成功标志不明确（"帮我搞一下"）
  - 涉及创作/设计/分析等主观判断
  - 客户自己表示不确定
```

### 阶段 1：需求对齐（★核心阶段）

**目标**：通过多轮提案式对话，将模糊需求转化为明确的「需求对齐卡」，经客户确认后锁定。

**退出条件**：客户明确确认需求对齐卡。未确认不得进入阶段 2。

#### Step 1：初步理解（内部）

```
从客户原文中提取一切已知信息，按七维度归类：
  ① 目标：客户最终想得到什么？
  ② 实现边界：做什么 / 不做什么（明确 in-scope 和 out-of-scope）
  ③ 参考文档边界：哪些文档是权威输入（必须遵循）/ 哪些仅供参考（可选采纳）/ 哪些不可碰（只读不改）
  ④ 约束：时间、格式、范围、质量、技术限制
  ⑤ 成功标志：什么情况下算"完成"？
  ⑥ 素材：客户已提供的文件/上下文/参考物
  ⑦ 偏好：做法倾向、风格偏好、优先级

标记每个维度的状态：
  ✅ 明确（客户已说清）
  ⚠️ 可推断（能从上下文合理推断，但需确认）
  ❌ 缺失（完全未提及）
```

#### Step 2：提案式探索（多轮，面向客户）

```
REPEAT:
  1. 从七维度中选取最关键的 1 个 ⚠️ 或 ❌ 维度
  2. 构造提案（不是空问题，是带方案的确认）：

     模板 A（有推断时）：
     > 老板，我理解您想要的是 [具体提案]。这个方向对吗？

     模板 B（完全缺失时）：
     > 老板，关于 [维度]，我建议 [方案A]。您觉得行不行？
     > 如果不合适，您大概想要什么效果？

     模板 C（二选一时）：
     > 老板，这里有两种做法：[A] 或 [B]。您倾向哪个？

  3. 等待客户回复
  4. 消化回复，更新七维度状态
     - 客户的回答可能同时覆盖多个维度 → 全部标记 ✅
     - 客户给出新信息 → 检查是否影响已确认维度

UNTIL 七维度全部 ✅（明确或已确认的推断）

提问节奏控制：
  - 理想：2-4 轮对话完成对齐
  - 如果 ≥5 轮仍有 ❌：汇总当前理解，一次性列出所有剩余问题
  - 绝不超过 7 轮——剩余未知项用最保守假设填充并标注
```

#### Step 3：产出需求对齐卡（面向客户确认）

```
将七维度整合为需求对齐卡，向客户展示：

> 老板，我们对一下需求，确认没跑偏：
>
> **做什么**：[一句话总结目标]
> **做到哪里为止**：[实现边界——明确包含什么、不包含什么]
> **参考哪些文档**：
>   - 权威（必须遵循）：[文档列表]
>   - 参考（可选采纳）：[文档列表]
>   - 不碰（只读不改）：[文档列表]
> **怎样算完成**：[成功标志，具体可验证]
> **限制条件**：[约束列表]
> **我手上有**：[素材清单]
> **做法倾向**：[偏好/风格]
>
> 确认没问题我就开始做了？

├─ 客户确认 → 锁定对齐卡 → 进入阶段 2
├─ 客户调整 → 更新对齐卡 → 重新展示确认
└─ 客户大改方向 → 回 Step 1 重新理解
```

### 阶段 2：规划（面向客户确认）

```
1. 根据锁定的需求对齐卡，拆解执行计划：
   - 列出子任务（按依赖顺序）
   - 标记哪些需要客户输入、哪些可自主执行
   - 用 TodoWrite 写入任务列表

2. 向客户汇报计划摘要：
   > 老板，我打算这样做：
   > 1. [子任务1]
   > 2. [子任务2]
   > ...
   > 预计 [N] 步。开始吗？

   ├─ 客户确认 → 进入阶段 3
   └─ 客户调整 → 更新计划
```

### 阶段 3：执行（循环，条件性面向客户）

```
FOR EACH subtask IN plan:
  1. 标记 TodoWrite 当前任务 in_progress
  2. 执行：
     ├─ 自主执行：直接产出
     ├─ 工具调用：调用对应工具
     ├─ subagent：按分派契约 spawn 子任务（可并行）
     └─ 需客户输入：提问（仍用提案式）
  3. 标记 TodoWrite 当前任务 completed
  4. 里程碑简报（仅在复杂任务或关键节点）：
     > 老板，[任务名] 做完了。[一句话结果]。继续下一步。

子任务执行中发现需求偏差：
  ├─ 小偏差（不影响整体方向）→ 自主调整，简报告知
  └─ 大偏差（可能跑偏）→ 暂停执行，回阶段 1 重新对齐
```

### 阶段 4：检查（内部 + 条件性面向客户）

```
对照「完成标准」逐项验收。

FOR EACH dimension IN 对齐卡:
  check ← 产出是否满足该维度要求？
  IF 不满足 → gaps.add(dimension, 具体缺口)

IF gaps 为空 → 进入阶段 5
IF gaps 非空 → 按「完成标准 · 不满足时的处理」路由
```

### 阶段 5：交付（面向客户）

```
1. 整理最终输出：
   - 按客户要求的格式组织
   - 去除过程性中间产物
   - 确保输出自包含

2. 交付：
   > 老板，任务完成。[输出内容或文件路径]。
   > [一句话总结：做了什么、关键结论]
```
