# 定时任务师（Scheduled Task Designer）

## TL;DR
- **输入**：定时任务目标 + cadence + 平台 + 权限边界
- **输出**：定时任务方案包（规格卡、主推荐、备选、prompt/命令草稿）
- **性质**：方法型——只设计时间触发任务，不负责内部多轮自转
- **关键卖点**：把“每天/每小时自动跑一次”设计成不重叠、可告警、可审计的定时任务

## Goal

把一项按时间或节奏触发的自动任务，设计成稳定、可恢复、不会重复踩人的定时任务。

## 角色设定

> 角色行为规范见 [角色协议](../角色协议.md) 和各角色文件。

- **执行**：[设计主笔](../角色/设计主笔.md) — 本技能中称”定时设计师”，负责 cadence、平台选型、防重策略和通知设计
- **审查**：[逻辑审查官](../角色/逻辑审查官.md) — 检查步骤连贯、规则无矛盾
- **沟通**：[业务翻译官](../角色/业务翻译官.md) — 把”什么时候触发”和”为什么每次只跑一轮”讲清楚

**技能内审查维度**：
- 节奏审查：频率是否合理，是否会过度触发
- 防重审查：上次没跑完时怎么办
- 告警审查：失败、异常、完成后是否能被看见

>
> 以下为本技能特有规则：

1. **定时任务默认一触发只跑一轮**：如果单次运行内部还要多轮推进，退回总控改走 `组合任务师`。
2. **必须写 cadence**：分钟、小时、每天、每周，不能写成“定期”这种空词。
3. **必须写防重策略**：`跳过 / 排队 / 终止旧任务 / 允许并发` 必须选一个。
4. **必须区分巡检和自动修复**：巡检型允许只读；自动修复型必须明确写权限和告警。
5. **官方与社区可并列评估**：`Codex Automations`、`claude -p + cron/CI/GitHub Actions` 都可以进候选，但最终只给 1 条主推荐。

## Workflow

> 以下步骤受[流程执行协议](../流程执行协议.md)约束：不可跳步、不可合并、每步必须产出步骤标记。

### Step 0：确认这真的是定时

```text
IF 需求核心是“直到完成前反复推进”:
  → 退回 总控，改走 循环任务师 或 组合任务师
```

### Step 1：锁定触发节奏

必须写清：

- 什么时候触发
- 触发频率
- 每次运行预期做什么
- 如果错过一次怎么办

### Step 2：选择平台

优先候选：

- `Codex Automations`
  - 适合：Codex 生态内的后台 recurring task
- `claude -p + cron / CI / GitHub Actions`
  - 适合：Claude 环境下离开会话后仍要定时跑

### Step 3：产出定时任务规格卡

固定包含：

- 任务名称
- 任务形态：定时任务
- 目标
- 平台与触发方式
- 输入来源
- 输出物
- 状态文件/状态源
- 单次运行边界
- 停止条件
- 防重策略
- 权限边界
- 验证方式
- 通知方式

### Step 4：起草 prompt / 命令

至少产出以下一种：

- Codex Automation prompt 草稿
- `claude -p` 命令草稿 + cron/CI 调度示意

### Step 5：风险审查

逐条检查：

- cadence 会不会太密
- 上次未完成时会不会重叠冲突
- 失败后会不会没人知道
- 巡检型会不会误变成自动修复
- 自动修复型权限是不是越界

## Initialization

触发本技能后，按这个顺序开始：

1. 判断是不是误分到了定时。
2. 写清 cadence、防重策略、通知方式。
3. 在 `Codex Automations` 和 `claude -p + cron/CI/GitHub Actions` 中选 1 条主推荐。
4. 输出规格卡、备选、风险和 prompt/命令草稿。
