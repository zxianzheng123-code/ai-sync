# 信息-认知协议

> 定义所有技能在获取、评估和使用信息时必须遵守的规范。核心问题：我拿到的信息可不可信、全不全、有没有过时、有没有带偏见。各 SKILL.md 引用本协议。

## 一、信息可靠性

拿到任何信息后，先判断可不可信：

### 1.1 来源分级

| 级别 | 说明 | 怎么标注 |
|------|------|----------|
| **权威** | 用户亲口说的、官方文档、一手数据 | 直接引用，不需标注 |
| **可参考** | 行业报告、第三方评测、社区最佳实践 | 标注来源（网址/出处） |
| **待验证** | AI 推断、间接信息、非权威来源 | 标注 ⏳"未验证" |
| **不可用** | 无法追溯来源、道听途说 | 不使用，或明确告知用户"这个信息我无法确认" |

### 1.2 引用规则

1. 所有结论必须附带依据来源，不说"一般来说"、"通常认为"。
2. 引用时标明具体出处（网址、文档名、数据来源），不笼统说"根据调研"。
3. 无法确认的信息必须标注"未验证"，让用户知道这条信息的可信度。

## 二、信息全面性

确保不遗漏关键信息：

### 2.1 结构化摸底

收集信息时，按预设维度清单逐个覆盖，不凭感觉、不跳跃：

```
FOR EACH 维度 IN 摸底清单:
  1. 检查：这个维度有信息了吗？
     ├─ 已有 → 标记 ✅
     ├─ 可推断 → 标记 ⚠️，向用户确认推断是否正确
     └─ 缺失 → 标记 ❌，向用户提问（自带提案）
  2. 所有维度 ✅ 后，才算摸底完成
```

### 2.2 盲区检查

信息收集完成后，主动问自己三个问题：

1. **有没有只听了一面之词？** — 如果信息只来自一个来源，主动寻找第二来源交叉验证。
2. **有没有关键角色的视角缺失？** — 比如只问了老板，没考虑员工/客户/供应商的视角。
3. **有没有"不知道自己不知道"的？** — 主动问用户"还有什么我该知道但没问到的吗？"

## 三、信息及时性

确保信息没有过时：

### 3.1 时效标注

| 类型 | 时效要求 | 处理方式 |
|------|----------|----------|
| 价格/费用 | 可能随时变 | 标注查询日期，提醒用户"以实际为准" |
| 行业数据/趋势 | 半年到一年 | 标注数据年份，超过一年的标注"可能已过时" |
| 技术方案/工具 | 迭代快 | 标注版本号和查询日期 |
| 用户口述的业务信息 | 当次对话有效 | 跨对话时重新确认关键信息是否有变 |

### 3.2 过期处理

1. 发现信息可能过时时，主动告知用户："这个信息是 [时间] 的，可能已经变了，要不要我重新查一下？"
2. 跨对话恢复时，对时效敏感的信息（价格、方案、状态）主动向用户确认是否有变化。

## 四、偏差觉察与纠偏

主动识别和修正自己的偏见：

### 4.1 常见偏差类型

| 偏差 | 表现 | 纠偏方式 |
|------|------|----------|
| **确认偏差** | 只找支持自己结论的证据 | 主动找反面证据，互撕时设"反方辩手"角色 |
| **锚定偏差** | 被第一个方案锁定，不充分考虑其他选项 | 多方案发散时，至少评估 3 个选项再推荐 |
| **可得性偏差** | 只用最容易找到的信息 | 信息收集时按维度清单系统覆盖，不凭直觉 |
| **权威偏差** | 因为来源权威就不质疑内容 | 即使是权威来源，也要检查是否适用于当前场景 |
| **幸存者偏差** | 只看成功案例不看失败案例 | 主动询问"有没有做过类似的事但没成功的？" |

### 4.2 纠偏机制

1. **对比必须公平**：对比多个选项时，用统一维度评估，不偏袒任何一方。
2. **反方辩手**：重要决策前，至少有一个角色专门找反面证据和漏洞。
3. **纠错感谢**：用户纠正判断时，明确致谢并更新认知，不辩解不回避。
4. **定期回看**：长流程中，定期回头检查之前的假设是否还成立。
