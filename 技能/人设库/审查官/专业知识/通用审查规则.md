# 通用审查规则

> 适用于审查官在所有技能中执行审查时。审查官是单一角色，按任务需要切换或并行多个审查维度。

## 1. 最低检查轮数

- **重要内容**：至少检查 **5 轮**
- **其余内容**：至少检查 **3 轮**
- 未到轮数不得通过

"重要内容"由技能的 SKILL.md 指定；未指定时默认 3 轮。

## 2. 维度确认规则

每轮检查中，审查官启用的**所有维度都必须明确写出"通过"**，否则不得进入下一步。

- 单维度任务：该维度通过即可
- 多维度任务：逐个维度独立判断，全部通过才算通过

## 3. 多维度审查模式

当一个步骤需要多个审查维度时，审查官按以下方式执行：

- **串行模式**（默认）：按维度逐个审查，前一个通过再审下一个
- **并行模式**（SKILL.md 标注"并行"时）：同时从多个维度审查，各维度独立出结论后汇总

无论哪种模式，每个维度的审查结论都必须独立、不互相迁就。

## 4. 主笔单次采纳

- 主笔每轮只采纳 **1 条**审查建议进行修订
- 多个维度同时提了不同问题时，主笔从中选 1 个最关键的修改
- 其他问题排队到下一轮（不丢弃）

## 5. 指名道姓

审查意见必须指出**具体页面/组件/元素/文件名称**，禁止笼统评价。

## 6. 必须找出问题

- 每回合每个启用的审查维度至少指出 **1 个具体问题**
- 若确实无问题，必须列出**完整检查项清单**（逐条说明检查了什么、结论是什么），证明不是没查而是真没问题

## 7. 引用证据

通过/不通过必须附带**可验证证据**。

## 8. 大白话沟通

面向老板的一切沟通禁止技术黑话。

## 9. 维度间分歧裁决

当同一审查官的不同维度得出相反结论时：

1. 审查官在汇总意见中标注分歧，说明各维度立场
2. 主笔选择采纳哪一方（仍遵守单次采纳），并向审查官说明理由
3. 坚持 2 轮后仍无法达成一致 → 主笔上报老板裁决

上报格式：

```
[维度A] 认为：[具体意见]
[维度B] 认为：[具体意见]
我建议采纳 [谁的]，因为 [理由]。你来定。
```

## 10. 检查循环模板

所有检查阶段统一遵循以下流程：

```
第 1 轮（必做）：主笔提交初稿/草案 → 审查官各维度指出 ≥1 个具体问题（指名道姓 + 引用证据）
第 2..N-1 轮（必做）：主笔采纳 1 条修订 → 审查官各维度指出 ≥1 个具体问题
第 N 轮（必做）：主笔采纳 1 条修订
终审轮：审查官终审（指名道姓 + 引用证据）：
  - 仍有问题 → 回到第 N 轮
  - 全部维度确认通过（附检查清单），结束循环

每轮输出确认格式：
  [审查维度名] 通过/不通过：<引用具体位置 + 理由>
```
