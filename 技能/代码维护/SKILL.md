## 执行模式
Master Agent 负责调度和验收，不亲自干活。每个步骤启动 SubAgent 执行。

## 工作人员
- 全栈开发（→ 技能/人设库/全栈开发/人设.md）
- 审查官（→ 技能/人设库/审查官/人设.md）
- 测试工程师（→ 技能/人设库/测试工程师/人设.md）

## 四种模式

| 模式 | 触发关键词 | 核心原则 | 验证方式 |
|------|-----------|---------|---------|
| 修 bug | 报错/不正常/崩溃/bug | 最小改动 | 复现修复 + 回归 |
| 重构 | 太乱/拆分/重组/重构 | 行为不变 | before/after diff + 回归 |
| 优化 | 太慢/体积大/性能/优化 | 数据驱动 | 量化基线 vs 优化后对比 |
| 测试 | 测试/覆盖率/用例 | Red-Green-Refactor | 全量测试通过 |

## Workflow

### Step 0：加载上下文（Master Agent）
执行 脚本/加载上下文.sh，强制读取所有引用内容

### Step 1：问题定位
- 执行者：全栈开发（SubAgent）
- 目标：识别维护意图，读取代码上下文，向用户确认模式
- 成功标准：维护模式已确认，问题/需求已明确
- 输入：用户描述的维护需求 + 可选作用域（文件路径/目录/模块名）+ 可选 `infra_spec.md`
- 输出：意图识别结果 + 代码上下文摘要
- 执行逻辑：
  1. 读取 infra_spec.md（如有）获取技术栈上下文
  2. 若有指定作用域 → 读取该范围内的代码结构
  3. 根据用户描述判断模式（修bug/重构/优化/测试）
  4. 向用户确认模式判断是否正确

### Step 2：方案确认
- 执行者：全栈开发（SubAgent）
- 目标：与用户对齐需求范围和成功标准（门禁 1），分析问题并制定方案
- 成功标准：用户已确认需求范围，方案已通过内部审批（门禁 2）
- 输入：Step 1 意图识别结果 + 代码
- 输出：审批通过的执行方案
- 用户对齐（按模式不同）：
  - 修 bug：确认问题现象、改动范围、成功标准
  - 重构：确认重构范围、动机、预期效果、行为不变
  - 优化：确认优化目标、范围、预期效果
  - 测试：确认测试类型、范围、优先级
- 分析（按模式不同）：
  - 修 bug：复现 → Trace 分析 → 定位根因 → 修复方案（改什么+怎么改+影响范围）
  - 重构：现状分析 → 改前快照 → 重构方案（改什么+怎么改+不变项清单）
  - 优化：量化基线 → 瓶颈分析 → 优化方案（改什么+预期提升+风险）
  - 测试：现有测试盘点 → 缺口识别 → 测试方案（新增用例+文件位置+mock需求）
- 方案审批（门禁 2）：方案未通过不得动手
- 关键原则：
  - 双门禁：用户对齐 + 方案审批都未通过前禁止改代码
  - 作用域锁定：有作用域时改动仅限指定范围

### Step 3：修改实现
- 执行者：全栈开发（SubAgent）
- 目标：严格按审批方案执行改动
- 成功标准：改动符合方案，无超出范围的修改
- 输入：Step 2 审批通过的方案
- 输出：修改后的代码
- 执行逻辑（按模式不同）：
  - 修 bug：严格按方案执行，每改一个文件立即单元级验证
  - 重构：严格按方案执行，生成 before/after diff 对比
  - 优化：严格按方案执行
  - 测试（Red-Green-Refactor）：写测试用例→确认红灯→补实现→确认绿灯→整理测试代码
- 关键原则：
  - 修 bug 最小改动：只改必须改的，不顺手重构
  - 重构行为不变：外部接口/API/用户可见行为完全不变
  - 优化数据驱动：禁止"感觉变快了"
  - 测试代码也遵守项目规范

### Step 4：审查（工程审查维度）
- 执行者：审查官（SubAgent）
- 目标：审查改动质量，确保符合模式要求
- 成功标准：改动符合模式特有要求，无副作用
- 输入：Step 3 修改后的代码 + Step 2 方案
- 输出：审查结论
- 审查维度（按模式不同）：
  - 修 bug：最小改动性、根因是否真正解决
  - 重构：是否改变了外部行为、边界是否清晰
  - 优化：量化指标是否合理、是否引入副作用
  - 测试：覆盖度、边界条件、mock 合理性

### Step 5：测试
- 执行者：测试工程师（SubAgent）
- 目标：运行完整回归，验证改动点 + 无副作用
- 成功标准：改动点验证通过，周边功能未被破坏
- 输入：Step 3 修改后的代码
- 输出：测试报告
- 通用验证：
  - 运行完整回归（构建 + lint + 功能测试）
  - 验证改动点：问题解决/结构优化/性能提升/测试通过
  - 验证无副作用：周边功能不受影响
- 模式特有验证：
  - 修 bug：确认复现步骤不再触发
  - 重构：展示 before/after diff，证明只改结构不改行为
  - 优化：展示量化对比表（基线 vs 优化后）
  - 测试：展示全量测试套件运行结果和覆盖率

### Step 6：验证
- 如有 脚本/验证/ 下的脚本 → 自动执行
- 如有 作业指引/交付自查清单.md → 逐项检查
- 全部通过才进入交付

### Step 7：交付审核（Master Agent）
输出成品，等用户确认

用大白话告知用户结果：
> [维护任务大白话摘要] 已经完成。[模式特有的验证摘要]。你确认一下没问题？
