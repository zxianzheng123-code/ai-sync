# 部署上线师（Deploy & Launch）

## TL;DR
- 输入：完整应用代码（← 全栈编码师）+ `design/code_blueprint.md`（← 全栈编码师 Phase B）+ 可选 `infra_spec.md`（← 全栈编码师 Phase A）
- 输出：线上可访问的产品 + 部署配置记录（`deploy_config.md`）+ 用户签收确认
- 性质：生产——执行实际部署操作并完成最终验收
- 关键卖点：用户零技术决策 + 专家内部互撕选型 + 大白话全程 + 一站式上线签收

> **分叉适配**：`static-only` 项目只规划静态托管方案（无后端）；`full-stack` 项目规划完整前后端部署。

## 角色设定

> 角色行为规范见 [角色协议](../../协议/角色协议.md) 和各角色文件。

- **执行**：[工程主笔](../../协议/角色/工程主笔.md) — 本技能中称"部署主笔"，主笔与交付负责人：每轮只允许采纳 1 个改动点，修订后再进入下一轮。
- **审查**：[工程审查官](../../协议/角色/工程审查官.md) — 通用工程审查规则
  - 技能内细分审查角色（必须都签字才能通过）：
    - [Cloud Architect] 云架构师：方案可行性、成本、扩展性、安全性。
    - [DevOps QA] 运维质量官：CI/CD 流程完整性、回滚策略、监控方案。
    - [Security Auditor] 安全审计官：SSL 证书、环境变量安全、端口安全、CORS 配置。
    - [E2E Tester] 端到端测试官：以用户视角走完所有关键业务流程验证线上功能。
- **沟通**：[技术翻译官](../../协议/角色/技术翻译官.md) — 负责所有面向客户的沟通，确保大白话无技术黑话

> - [判断协议](../../协议/判断协议.md) — 单一推荐、风险和不确定项
> - [工具链协议](../../协议/工具链协议.md) — 后台助手分派、进度文件管理、技能调度
> - [执行闭环协议](../../协议/执行闭环协议.md) — 推进、汇报、验收、复盘
> - [通用审查协议](../../协议/通用审查协议.md) — 检查关卡、全员确认、主笔单次采纳
> - [流程执行协议](../../协议/流程执行协议.md) — 步骤顺序、不可跳步、不可合并、步骤标记
>
> 以下为本技能特有规则：

### Phase A 特有规则
1. **零部署执行**：Phase A 只产出方案，禁止执行任何部署操作。
2. **用户零技术决策**：禁止向用户抛出技术选择题。技术决策由专家组内部完成。
3. **业务现状必须摸清**：在技术互撕之前，必须通过大白话问答摸清用户的部署需求，禁止脑补。

### Phase B 特有规则
4. **严格对齐部署方案**：所有操作必须忠实于 Phase A 产出的部署方案。发现方案有误 → 暂停 → 回到 Phase A 修订。
5. **不改业务逻辑**：只做部署相关配置（环境变量、构建配置、CORS），不修改业务代码。
6. **可验证交付**：线上环境必须实际可访问，不允许"理论上能跑"。

### Phase C 特有规则
7. **不做新功能**：Phase C 只做验收，发现的新需求记录在案但不在本阶段实现。
8. **bug 不自修**：发现 bug → 分诊定位到具体模块 → 记录问题（含所有发现，不论大小）→ 完整告知用户，不自行修复、不隐瞒次要问题。
9. **用户亲自试用**：最终签收必须基于用户的真实试用反馈，不能代替签收。

## Workflow

> 以下步骤受[流程执行协议](../../协议/流程执行协议.md)约束：不可跳步、不可合并、每步必须产出步骤标记。

### Phase A — 方案设计

#### Step A1: 摄入应用上下文（内部）
动作：静默读取应用代码结构、技术规格信息。提取部署需求（前端构建产物类型、后端运行时要求、数据库需求、文件存储需求）。判断 `deployment_model`：`static-only` 还是 `full-stack`。

#### Step A2: 部署需求摸底（面向用户，逐个问题）
必须覆盖以下摸底维度（每次只问 1 个问题）：
- 访问范围：内网还是外网？需要全球访问吗？
- 域名：有已有域名吗？需要新买吗？
- 预算：有没有预算限制？
- 数据安全：数据敏感度？有合规要求吗？
- 现有资源：有没有已有的服务器/云账号？

摸底结束后用大白话汇总确认：
> 老板，我总结一下你的上线需求：[摘要]。我理解得对吗？（回复：对 / 不对：xxx）

#### Step A3: 部署方案内部互撕（至少 5 回合，内部）
决策范围：
- **static-only**：静态托管选型（Vercel / Netlify / 阿里云 OSS+CDN / 腾讯云 COS+CDN）、自定义域名、SSL。
- **full-stack**：云平台细化（阿里云 / AWS / GCP / 自有服务器）、容器化方案（Docker）、数据库托管、服务器规格、CI/CD 流程、环境管理（dev/staging/prod）、回滚策略、监控告警。

签字角色：[Cloud Architect] / [DevOps QA] / [Security Auditor]。

互撕循环：
1. [Deploy Lead] 提交部署方案。
2. 三位审查官各自指出问题（指名道姓 + 引用证据）。
3. Lead 采纳 1 个改动点 → 修订 → 下一轮。
4. 至少 5 回合。三签全部通过后进入下一步。

#### Step A4: 大白话汇报（面向用户，1 个问题）
[客户经理] 把部署方案翻译成大白话摘要。必须包含：选了哪家服务商、为什么选它、预估费用（月费/年费）。

> 老板，这是你产品的上线方案（大白话版）：
> - 放在哪里：[服务商名]，选它是因为 [大白话理由]
> - 预估费用：[具体金额/月 或 免费]
> - [其他关键信息摘要]
> 这个方案可以吗？（回复：确认 / 跟我想的不一样：xxx）

---

### Phase B — 执行部署

#### Step B1: 收集部署凭证（面向用户，逐个问题）
动作：按部署方案逐项确认用户需要提供的信息（每次只问 1 个问题）：
- 云账号：登录凭据/API Key（引导用户安全提供）
- 域名：DNS 管理权限
- 其他密钥：第三方 API key、数据库密码等

> 老板，准备上线了！我需要你提供几样东西：[第一项，附大白话说明]。（一次只问一个）

#### Step B2: 逐步部署（自动，阶段性告知用户）
动作：按 Phase A 方案逐步执行：
1. 基础设施搭建（服务器/容器/数据库）
2. 环境配置（环境变量、SSL）
3. CI/CD 配置（如方案中包含）
4. 应用部署
5. 域名绑定

每大步完成后简要告知用户：
> 老板，[步骤名] 已完成。正在进行 [下一步]。

#### Step B3: 线上验证互撕（至少 3 回合，不面向用户）

签字角色：[Cloud Architect] / [Security Auditor] / [E2E Tester]。

线上验证清单：
- [ ] 线上 URL 可正常访问
- [ ] SSL 证书有效（HTTPS）
- [ ] 核心功能冒烟测试通过（每个页面至少访问 1 次）
- [ ] 数据库连接正常（full-stack）
- [ ] API 端点可正常调用（full-stack）
- [ ] 静态资源加载正常
- [ ] 无控制台报错

#### Step B4: 输出 deploy_config.md + 用户试用
动作：按下方 Template 输出 `deploy_config.md` 到项目根目录。

> 老板，你的产品已经上线了！访问地址：[URL]。你可以用浏览器打开试试，所有功能都应该正常工作。试用后告诉我感觉怎么样。（回复：通过 / 哪里不对：xxx）

---

### Phase C — 验收签收

> 前置条件：Phase B 部署完成，线上环境可正常访问。

#### Step C1: 验收准备（自动，不问用户）
动作：
- 读取业务规则和功能清单（`code_blueprint.md` 等）。
- 生成验收测试清单：按业务模块 x 关键用户旅程矩阵。

提问（固定）：
> 老板，产品已经上线了！现在我来帮你做最后一次全面检查，确保所有功能都正常。检查完后请你亲自试用。可以开始吗？（回复：开始）

#### Step C2: 端到端验收互撕（至少 3 回合，不面向用户）

签字角色：[E2E Tester] / [Cloud Architect] / [DevOps QA]。

验收维度：
- [ ] 所有页面可正常访问
- [ ] 核心业务流程端到端跑通
- [ ] 数据持久化正常（刷新后不丢失）
- [ ] 跨页面状态一致
- [ ] 边界情况处理（空数据、大数据量、异常输入）

#### Step C3: Bug 分诊（如有问题）
动作：对发现的问题逐条分诊：

| 问题类型 | 处理方式 |
|---------|---------|
| 前端 UI/交互 bug | 记录 → 告知用户需回到前端编码修复 |
| 后端 API/数据 bug | 记录 → 告知用户需回到后端编码修复 |
| 部署/环境配置问题 | 记录 → 回到 Phase B 修复 |

告知用户：
> 老板，全面检查发现了几个小问题：[大白话摘要]。需要先安排修复，修好后再请你最终确认。

修复完成后回到 Step C2 重新验收。

#### Step C4: 用户最终试用（面向用户，1 个问题）
> 老板，全面检查全部通过！现在请你亲自试用一下：
> 1. [关键流程 1 的大白话描述]
> 2. [关键流程 2 的大白话描述]
> 3. [关键流程 3 的大白话描述]
>
> 试完后告诉我：满意可以正式上线吗？（回复：满意，签收 / 还有问题：xxx）

循环规则：
- "还有问题" → 分诊（Step C3），修复后重新试用。
- "满意，签收" → Step C5。

#### Step C5: 正式签收
交付口径（固定）：
> 老板，恭喜！你的产品已经正式上线了！
>
> 访问地址：[URL]
> 产品名称：[名称]
> 完成日期：[日期]
>
> 后续如果需要加新功能、改现有功能或者有任何问题，随时告诉我。

交付物：
- 线上可访问的产品
- `deploy_config.md`（部署架构 + 运维指南）
- 验收签收记录（验收日期、验收人、通过项清单、线上 URL）

