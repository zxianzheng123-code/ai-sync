## 执行模式
Master Agent 负责调度和验收，不亲自干活。每个步骤启动 SubAgent 执行。

## 工作人员
- 工程（→ 技能/人设库/工程/人设.md）

## Workflow

### Step 0：加载上下文（Master Agent）
执行 脚本/加载上下文.sh，强制读取所有引用内容

### Step 1：环境准备
- 执行者：工程（SubAgent）
- 目标：设计部署方案，摸清用户部署需求，完成方案选型
- 成功标准：部署方案完整（含服务商、费用、回滚策略），用户已确认
- 输入：完整应用代码 + `design/code_blueprint.md` + `infra_spec.md`
- 输出：锁定的部署方案
- 执行逻辑：
  1. 摄入应用上下文：读取代码结构和技术规格，判断 deployment_model（static-only / full-stack）
  2. 部署需求摸底（面向用户，每次只问 1 个问题）：
     - 访问范围（内网/外网/全球）
     - 域名（已有/新买）
     - 预算限制
     - 数据安全与合规要求
     - 现有服务器/云账号
  3. 部署方案内部决策：
     - static-only：静态托管选型（Vercel/Netlify/阿里云OSS+CDN等）、域名、SSL
     - full-stack：云平台细化、容器化、数据库托管、CI/CD、回滚策略、监控
  4. 大白话向用户汇报方案（必须包含：选了哪家、为什么、预估费用）
- 关键原则：
  - 用户零技术决策
  - 零部署执行：本步骤只产出方案，不执行部署操作

### Step 2：部署执行
- 执行者：工程（SubAgent）
- 目标：按方案执行部署，线上环境可正常访问
- 成功标准：线上URL可访问、SSL有效、核心功能冒烟通过
- 输入：Step 1 部署方案 + 用户提供的凭证
- 输出：线上可访问的产品 + `deploy_config.md`
- 执行逻辑：
  1. 收集部署凭证（面向用户，每次只问 1 个）：云账号/域名DNS/第三方密钥
  2. 逐步执行部署：基础设施搭建 → 环境配置 → CI/CD配置 → 应用部署 → 域名绑定
  3. 每大步完成后简要告知用户进度
  4. 线上验证：URL可访问、SSL证书有效、核心功能冒烟测试、数据库连接（full-stack）、API端点可调（full-stack）、静态资源正常、无控制台报错
  5. 输出 deploy_config.md
- 关键原则：
  - 严格对齐 Step 1 方案，方案有误则暂停回到 Step 1
  - 不改业务逻辑：只做部署相关配置
  - 可验证交付：线上必须实际可访问

### Step 3：冒烟测试
- 执行者：工程（SubAgent）
- 目标：端到端验收，确保线上产品功能完整
- 成功标准：所有页面可访问、核心业务流程跑通、数据持久化正常
- 输入：线上环境 + 业务规则和功能清单
- 输出：验收测试报告
- 执行逻辑：
  1. 生成验收测试清单（按业务模块 x 关键用户旅程矩阵）
  2. 端到端验收：所有页面可访问、核心流程跑通、数据持久化、跨页状态一致、边界情况
  3. 发现问题 → Bug 分诊：
     - 前端 UI/交互 bug → 记录，需回到全栈编码修复
     - 后端 API/数据 bug → 记录，需回到全栈编码修复
     - 部署/环境配置问题 → 回到 Step 2 修复
  4. 用户最终试用：提供关键流程的大白话操作指引
- 关键原则：
  - 不做新功能：只做验收
  - bug 不自修：发现问题完整记录并告知用户，不自行修复不隐瞒
  - 用户亲自试用：最终签收基于用户真实反馈

### Step 4：验证
- 如有 脚本/验证/ 下的脚本 → 自动执行
- 如有 作业指引/交付自查清单.md → 逐项检查
- 全部通过才进入交付

### Step 5：交付审核（Master Agent）
输出成品，等用户确认

交付物：
- 线上可访问的产品
- `deploy_config.md`（部署架构 + 运维指南）
- 验收签收记录（验收日期、验收人、通过项清单、线上URL）

正式签收口径：
> 恭喜！你的产品已经正式上线了！
> 访问地址：[URL]
> 产品名称：[名称]
> 完成日期：[日期]
> 后续需要加新功能、改现有功能或有任何问题，随时告诉我。

```markdown
- CDN/Storage: [方案]（如有）

## Environment Variables (Redacted)

|-----|---------|-----------|

## CI/CD
- Trigger: [push to main / manual]
- Pipeline: [build → test → deploy]
- Build Command: [具体命令]

## Rollback
1. [回滚操作步骤]

## Troubleshooting
| 症状 | 可能原因 | 排查步骤 |
|------|---------|---------|

```
