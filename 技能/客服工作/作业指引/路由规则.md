# 路由规则

> 本指引供客服工作 Step 3（路由）使用。目标：把成题后的需求匹配到正确的工作流程，或进入计划模式。

## 一、路由表

| 工作流 | 触发条件 |
| --- | --- |
| 问题拆解 | 当用户的需求还没说清楚、混杂多个意图时 |
| 答案澄清 | 当用户需要解释、澄清、认知补全时 |
| 调研判断 | 当用户需要选型、对比、诊断、分析时 |
| 方法提炼 | 当用户需要 SOP、规则、流程、可重复执行的方法时 |
| 教材编写 | 当用户需要课本、训练资料、系统化经验文档时 |
| 造工具 | 当用户要做系统、页面、应用、自动化流程时 |
| 循环定时任务 | 当用户要设置循环或定时执行的自动化任务时 |
| Skill编译 | 当用户要创建或修改一个新技能时 |
| 测试循环 | 当用户要对已有代码做测试循环时 |
| 通用任务 | 当以上工作流都不匹配时的兜底 |

## 二、五类最终成果定义

所有工作流最终都在服务同一件事——把用户的原始需求转成用户要拿到的东西：

| 最终成果 | 定义 | 典型请求 | 什么时候算完成 |
| --- | --- | --- | --- |
| `答案` | 解释、澄清、认知补全 | "这是什么意思""常见原因有哪些" | 用户拿到可直接理解的一句话结论或简明解释 |
| `判断` | 选型、对比、诊断、分析 | "哪个更合适""为什么会这样" | 给出明确推荐/判断 + 依据 + 风险 |
| `方法` | SOP、规则、流程、AI 执行规则 | "把这个固化成规则""以后按这个跑" | 文档可重复执行，步骤、条件、边界清楚 |
| `教材` | 课本、训练资料、系统化经验文档 | "写给新人学""整理成课本" | 文档可理解、可教学、可传承 |
| `工具` | 系统、页面、应用、自动化流程 | "做个工具""做个系统" | 进入造工具链并产出可执行工具 |

## 三、成果判断顺序

路由时按以下顺序判断当前请求应该转成什么：

1. 题目是否已经说清了？
   - 否 → 问题拆解
2. 用户要的是解释、澄清、认知补全吗？
   - 是 → 答案澄清
3. 用户要的是基于信息的选择、诊断、对比、分析吗？
   - 是 → 调研判断
4. 用户要的是以后能重复执行的方法、规则、流程吗？
   - 是 → 方法提炼
5. 用户要的是以后能教给别人、传给新人、沉淀成体系的内容吗？
   - 是 → 教材编写
6. 用户要的是系统、页面、应用、自动化流程吗？
   - 是 → 造工具
7. 用户要的是循环或定时执行的自动化任务吗？
   - 是 → 循环定时任务
8. 用户要创建或修改技能吗？
   - 是 → Skill编译
9. 用户要对已有代码做测试循环吗？
   - 是 → 测试循环
10. 以上都不匹配？
    - → 通用任务（兜底）
11. 通用任务也不合适？
    - → 计划模式（临时编排）

## 四、交接规则

路由完成后，必须向下游工作流传递以下信息：

| 字段 | 说明 |
| --- | --- |
| 题目对象 | 这件事到底在说什么 |
| 目标 | 用户这次想拿到什么 |
| 边界 | 做什么 / 不做什么 |
| 成功标志 | 怎样算完成 |
| 约束 | 时间、范围、素材、限制条件 |
| 路由依据 | 为什么选了这个工作流 |

## 五、边界提醒

- 答案不是缩水版判断：不需要大量证据收集时，直接给答案
- 判断不是缩水版方法：做出判断不等于已经形成可执行方法
- 方法不是缩水版教材：方法重"照着做"，教材重"为什么这样做"
- 教材不是更长的方法：教材要帮助理解和传承，不只是堆步骤
- 工具不是文档：只要目标是做成系统，就进入造工具
- 一次只锁一个主成果：同一阶段只允许一个主成果，需要多个成果时按顺序逐个完成
