# 转化路由指引

> 本指引定义路由前的总前置规则和成果到技能的映射关系。供客服工作在路由决策时参考。

## 一、总前置规则

在路由到任何工作流之前，必须遵守以下规则：

1. **题目还没说清先拆解**：题目是否说清的判断标准见 成题指引.md 第二、三节。确定成果类型后，该成果的成题最低要求见成题指引第四节。未说清时不直接产出成果，先进入问题拆解工作流。
2. **一次只锁一个主成果**：同一阶段只允许一个主成果。需要多个成果时，按顺序逐个完成，不混在一个工作流里同时做。
3. **用户要拿到的东西比动作优先**：先判断用户当前最想拿到什么，再决定用什么动作。不能因为工作流会"调研/写文档/拆解"就反推成果。
4. **未定的判断不沉淀**：核心结论没定之前，不进入方法提炼或教材编写。
5. **未定的问题不判断**：问题本身还乱的时候，不进入调研判断。

## 二、成果到工作流映射

| 最终成果 | 对应工作流 | 触发条件 |
| --- | --- | --- |
| 拆解 | 问题拆解 | 当需求还没说清楚、混杂多个意图时 |
| 答案 | 答案澄清 | 当用户需要解释、澄清、认知补全时 |
| 判断 | 调研判断 | 当用户需要选型、对比、诊断、分析时 |
| 方法 | 方法提炼 | 当用户需要 SOP、规则、流程、可重复执行的方法时 |
| 教材 | 教材编写 | 当用户需要课本、训练资料、系统化经验文档时 |
| 工具 | 造工具 | 当用户要做系统、页面、应用、自动化流程时 |
| 循环任务 | 循环定时任务 | 当用户要设置循环或定时执行的自动化任务时 |
| 新技能 | Skill编译 | 当用户要创建或修改一个新技能时 |
| 测试 | 测试循环 | 当用户要对已有代码做测试循环时 |
| 其他 | 通用任务 | 当以上工作流都不匹配时的兜底 |
| 无法匹配 | 计划模式 | 当通用任务也无法覆盖时，临时编排 |

## 三、交接规则

| 上游 | 下游 | 必须交接什么 |
| --- | --- | --- |
| 客服工作 | 问题拆解 | 原始需求、当前卡在哪里、已知信息 |
| 客服工作 | 答案澄清 | 题目定义、关键约束、需要解释到什么程度 |
| 客服工作 | 调研判断 | 题目定义、比较对象、重点维度、关键约束 |
| 客服工作 | 方法提炼 | 题目定义、方法给谁用、用在什么场景、关键约束 |
| 客服工作 | 教材编写 | 题目定义、教材给谁看、学会什么、关键约束 |
| 客服工作 | 造工具 | 题目定义、业务问题、谁来用、第一版边界 |
| 调研判断 | 方法提炼 / 教材编写 | 结论、依据、风险、待验证项 |
| 方法提炼 | 造工具 | 规则、流程、边界、异常处理口径 |
| 教材编写 | 造工具 | 背景、核心原理、场景、边界认知 |

## 四、成果区分标准

当不确定该路由到哪个工作流时，按以下标准区分：

- **答案 vs 判断**：答案是"这是什么"，判断是"该选哪个"。不需要大量证据收集时，直接走答案。
- **判断 vs 方法**：判断是"做出选择"，方法是"把选择变成可重复执行的流程"。做出判断不等于已经形成可执行方法。
- **方法 vs 教材**：方法重"照着做"，教材重"为什么这样做"。方法面向执行者，教材面向学习者。
- **教材 vs 方法**：教材要帮助理解和传承，不只是堆步骤。如果目标是"以后新人能学会"，走教材。
- **工具 vs 文档**：只要目标是做成系统、页面、应用，就走造工具，不要停在写文档。

## 五、计划模式触发条件

当以下情况同时满足时，进入计划模式：

1. 路由表中所有工作流都无法覆盖用户的需求
2. 通用任务工作流也不合适
3. 需求本身是清晰的（已通过成题）

计划模式执行步骤：

1. 从人设库中挑选合适的工作人员
2. 临时定义步骤、交付物、检查点
3. 按步骤执行
4. 执行完毕后询问用户：这个流程要不要沉淀为正式技能？
   - 要 → 交给 Skill编译 正式编排为新技能
   - 不要 → 一次性任务结束
