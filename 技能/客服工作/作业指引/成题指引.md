# 成题指引

> 本指引供客服工作 Step 2（成题）使用。目标：判断用户需求是否说清楚，没说清就补问，说清了就放行。

## 一、成题定义

"成题"指的是：用户的需求已经被压成一个可继续处理的明确对象，而不是一团混在一起的意图。

一个题至少要回答清楚这 5 件事：

1. **题目对象**：这件事到底在说什么
2. **目标**：这次到底想拿到什么结果
3. **边界**：做什么 / 不做什么
4. **成功标志**：怎样算完成
5. **约束**：时间、范围、素材、限制条件

## 二、成题状态分级

| 状态 | 定义 | 处理方式 |
| --- | --- | --- |
| `还没说清` | 目标混杂、对象不清、边界缺失 | 不进入下游，先在 Step 2 循环补问 |
| `差一点` | 目标大致清楚，但关键字段仍缺 | 补问缺口，补齐后再进入 Step 3 |
| `说清了` | 5 个关键字段已基本清楚 | 可进入 Step 3 路由 |
| `确认锁定` | 关键字段已清楚且已获用户确认 | 可以执行，不再随意变更。标记格式：`[成题锁定] 题目对象 / 目标 / 成功标志 — 已获用户确认`。变更已锁定的题目需要用户再次确认 |

## 三、缺口检查流程

开始处理前，按以下顺序检查：

```
1. 当前请求是不是只在说一件事？
   ├─ 否 → 还没说清
   └─ 是 → 继续

2. 目标是不是唯一？
   ├─ 否 → 还没说清
   └─ 是 → 继续

3. 边界、成功标志、约束是否足够支撑下游动作？
   ├─ 否 → 差一点
   └─ 是 → 说清了

4. 用户是否明确确认？
   ├─ 否 → 说清了
   └─ 是 → 确认锁定
```

## 四、不同工作流的成题最低要求

一旦确定了成果类型，这道题至少要补齐以下信息才算"说清了"：

### 答案类（答案澄清）
- 到底在问什么
- 需要解释到什么程度

### 判断类（调研判断）
- 要判断什么
- 比较对象或诊断对象是什么
- 重点维度是什么

### 方法类（方法提炼）
- 方法给谁用
- 用在什么场景
- 最终想稳定复现什么结果

### 教材类（教材编写）
- 教材给谁看
- 看完之后要学会什么
- 这份教材讲的是哪套体系

### 工具类（造工具）
- 要解决什么业务问题
- 谁来用
- 第一版做到哪里为止

## 五、提问规则

补题时，统一按以下规则问：

1. **一次只补 1 个关键缺口**：不要一口气问一堆
2. **提问必须自带提案**：不空问，给出你的判断让用户确认
3. **优先补高影响缺口**：先补影响方向的，再补影响细节的
4. **已经说过的不重复问**：用户说过的内容直接采纳

## 六、标准口径

补问时使用以下口径：

> 老板，我理解你现在想解决的是 `[题目对象]`，这次最想拿到的是 `[目标]`。我还差一个关键点：[缺口]。我建议先按 `[提案]` 锁住，行吗？

## 七、还没说清时的退出规则

出现以下任一情况，判定"还没说清"，进入问题拆解工作流：

- 一句话里同时包含多个混杂的意图
- 用户明确表示"我也说不清"
- 比较对象、使用对象或目标结果一直在漂
- 补问 2 轮后仍无法锁住主目标

退出口径：

> 老板，这件事现在还没说清。我先不硬往下做，不然越做越乱。建议先拆开，锁住当前最重要的一题。

## 八、边界提醒

- 本指引只管"题有没有说清"，不管"怎么找证据"
- 本指引只管"题该怎么锁"，不管"最后推荐哪个方案"
- 成题不是判断，成题也不是执行
