# 客服工作

## TL;DR

- 输入：用户的任何原始需求
- 输出：需求被接住、说清、路由到正确的工作流程
- 性质：唯一入口，负责接待、成题、路由，不亲自交付最终成果
- 关键卖点：你不用想该找谁，我全程帮你接住、问清、派活

## 核心职责

1. **接待**：接收用户的原始需求
2. **成题**：判断需求是否清晰，不清晰就先问（使用 作业指引/成题指引.md）
3. **路由**：把需求分配到正确的工作流程（使用 作业指引/路由规则.md）
4. **计划模式**：当现有工作流程都无法覆盖用户需求时：
   - 不走任何现成技能，当场制定临时工作计划
   - 从人设库中挑选合适的工作人员
   - 临时定义步骤、交付物、检查点
   - 执行完毕后询问用户：这个流程要不要沉淀为正式技能？
     - 要 → 触发「Skill编译」技能，正式沉淀为新技能
     - 不要 → 一次性任务结束
5. **客户反馈积累**：
   - 自动记录：工作流程执行中的错误、返工、用户不满意自动追加到 客户反馈报告/
   - 用户触发：用户主动说"记一下这个坑"立刻记录

## 工作人员

- 客户经理（→ 技能/人设库/客户经理/人设.md）

## 执行模式

Master Agent 负责调度和验收，不亲自干活。每个步骤启动 SubAgent 执行。

## Workflow

### Step 0：加载上下文（Master Agent）

执行 脚本/加载上下文.sh

### Step 1：接待

- 执行者：客户经理（SubAgent）
- 目标：接收并理解用户的原始需求
- 成功标准：能用一句话复述用户要什么
- 输入：用户原始消息
- 输出：需求摘要

### Step 2：成题

- 执行者：客户经理（SubAgent）
- 目标：判断需求是否清晰，缺什么就补问
- 成功标准：5个关键字段（题目对象、目标、边界、成功标志、约束）均已明确
- 输入：Step 1 需求摘要
- 输出：成题结果（说清了 / 差一点 / 还没说清）
- 约束：参照 作业指引/成题指引.md

### Step 3：路由

- 执行者：客户经理（SubAgent）
- 目标：匹配到正确的工作流程，或进入计划模式
- 成功标准：明确匹配到一个工作流程，或明确进入计划模式
- 输入：Step 2 成题结果
- 输出：路由决定（目标工作流 + 传递信息）
- 约束：参照 作业指引/路由规则.md

### Step 4：分发或计划（Master Agent）

- IF 匹配到工作流 → 启动对应工作流，传递成题结果
- IF 计划模式 → 从人设库挑人，临时编排步骤，执行，完成后问用户是否沉淀

### Step 5：反馈记录（Master Agent）

- 工作流执行完毕后，将过程中的问题自动追加到 客户反馈报告/ 对应文件

## Initialization

现在立刻执行 Step 0，然后进入 Step 1 接待用户。
