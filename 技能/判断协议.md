# 判断协议

> 定义所有技能在做选型、对比、诊断、取舍和推荐时必须遵守的规则。本协议只管“如何从已成题 + 已有证据走到判断”；不管怎么检索，也不管后续如何沉淀成方法或教材。

## 一、判断定义

“判断”指的是：

- 在多个方案里选一个
- 对一个现象给出原因判断
- 对一组数据给出解释和倾向
- 对一件事给出明确推荐

## 二、进入条件

进入判断前，至少满足：

1. 题目已成形
2. 判断对象明确
3. 关键证据已具备，或已明确哪些证据缺失

任一不满足：

- 题目不清 → 回 `成题协议`
- 证据不够 → 回 `证据协议`

## 三、判断类型

| 类型 | 典型问题 | 产出 |
| --- | --- | --- |
| `选项对比` | “A 和 B 选哪个” | 推荐方案 + 对比依据 |
| `问题诊断` | “为什么最近复购下滑” | 原因判断 + 可能性排序 |
| `方案选型` | “应该用什么方式做” | 方案推荐 + 风险 |
| `优先级排序` | “先做哪个” | 排序结果 + 理由 |

## 四、判断流程

```
Step 1：列候选
Step 2：锁评估维度
Step 3：统一维度比较
Step 4：找反面证据
Step 5：给单一推荐
Step 6：补风险和不确定项
```

## 五、候选方案规则

1. 能比较时至少列 3 个候选
2. 必须包含“不做 / 维持现状”作为基线
3. 不准先锁一个方案再补理由

## 六、评估维度规则

评估维度必须来自：

- 用户明确关心的点
- 当前场景的客观约束
- 下游执行真正会受影响的因素

禁止使用：

- 只因为容易写就选的维度
- 对某个候选明显有利的私货维度

## 七、单一推荐原则

面对用户时，默认只给一个推荐。

标准口径：

> 老板，我推荐 `[方案X]`，因为 `[理由]`。其他方案的问题是 `[简述]`。

只有在以下情况才给并列多选：

- 方案确实打平
- 差异主要取决于个人偏好
- 用户主动要求看全部方案

## 八、风险与不确定性

每个判断都要附带：

1. **核心风险**
2. **仍未验证的点**
3. **如果判断错了，代价有多大**

## 九、判断完成标准

判断完成必须同时满足：

- 推荐对象明确
- 推荐理由可追溯到证据
- 风险和不确定项已标出
- 用户能看懂为什么这样判断

## 十、边界提醒

- 本协议不负责“怎么找证据”
- 本协议不负责“怎么把结论写成 SOP”
- 本协议也不负责“怎么把结论写成教材”
