# Product Logic

## Instructions For Design & Coding AI
- 本文档定义系统运转逻辑：场景、流程、联动、触发规则。
- 页面设计必须覆盖本文档中的所有场景和联动点。
- 编码时的数据流和事件处理必须与本文档的触发规则对齐。
- 发现逻辑矛盾或缺口时，以本文档为准，向上游反馈。
- 卖家定价不得超过原价的 70%
- 交易完成后 24 小时内双方可互评，超时默认好评
- 信用积分低于 60 分的用户限制发布和购买
- 同一本书最多同时发布 3 条出售信息（防刷单）

## Business Scenarios

### Scenario: 卖家发布二手教材
- Trigger: 卖家进入书籍发布模块时
- Flow:
  1. 卖家 在 书籍发布 填写书籍信息（课程/ISBN/书名）、定价、新旧程度并上传照片 → 系统校验定价不超过原价的 70%，将书籍发布草稿状态设为"待提交"
  2. 卖家 在 书籍发布 提交发布 → 系统将书籍出售信息状态设为"已发布"，写入发布时间
  3. 系统 在 消息通知 向订阅该课程的买家推送新书上架通知 → 消息通知状态设为"已推送"
- Success End: 书籍出售信息状态为"已发布"
- Exception Paths:
  - IF 定价超过原价的 70%: 系统将发布操作状态设为"定价超限"，向卖家展示"定价不得超过原价的 70%，请重新定价"，卖家可修改定价后重新提交。
  - IF 同一本书当前已有 3 条出售信息处于"已发布"状态: 系统将发布操作状态设为"已拦截"，向卖家展示"同一本书最多同时发布 3 条出售信息，请下架现有信息后再发布"，卖家可先下架已有信息再重新发布。
  - IF 卖家信用积分低于 60 分: 系统将发布操作状态设为"已限制"，向卖家展示"信用积分低于 60 分，限制发布和购买"，卖家可联系平台管理员申诉处理。

### Scenario: 买家搜索并下单购买教材
- Trigger: 买家进入搜索匹配模块时
- Flow:
  1. 买家 在 搜索匹配 输入课程/ISBN/书名进行搜索 → 系统返回匹配书籍列表，并展示同课程教材的智能推荐
  2. 买家 在 搜索匹配 查看书籍详情（书况、定价、卖家信用积分） → 系统展示该书籍完整信息
  3. 买家 在 交易管理 确认下单 → 系统将订单状态设为"待卖家确认"，写入下单时间
  4. 系统 在 消息通知 向卖家推送新订单通知 → 消息通知状态设为"已推送"
  5. 卖家 在 交易管理 确认订单 → 系统将订单状态设为"已确认"，写入确认时间
- Success End: 订单状态为"已确认"
- Exception Paths:
  - IF 买家信用积分低于 60 分: 系统将下单操作状态设为"已限制"，向买家展示"信用积分低于 60 分，限制发布和购买"，买家可联系平台管理员申诉处理。
  - IF 卖家在 24 小时内未确认订单: 系统将订单状态设为"已超时取消"，向买家展示"卖家未及时确认，订单已取消"，买家可重新搜索其他出售信息下单。

### Scenario: 买卖双方确认交易完成并互评
- Trigger: 买家进入交易管理模块，订单状态为"已确认"时
- Flow:
  1. 买家 在 交易管理 确认收到书籍并完成交易 → 系统将订单状态设为"已完成"，写入完成时间
  2. 系统 在 评价系统 为买卖双方生成评价任务 → 评价窗口状态设为"待互评"，写入 24 小时评价截止时间
  3. 买家 在 评价系统 对卖家提交评价 → 系统将买家评价状态设为"已提交"，更新卖家信用积分
  4. 卖家 在 评价系统 对买家提交评价 → 系统将卖家评价状态设为"已提交"，更新买家信用积分
- Success End: 订单状态为"已完成"，买家评价状态为"已提交"，卖家评价状态为"已提交"
- Exception Paths:
  - IF 评价截止时间到达且任一方未提交评价: 系统将未提交方评价状态设为"默认好评"，向未提交方展示"评价时间已过，系统已默认好评"，未提交方可查看评价记录。
  - IF 买家对差评内容提交申诉: 系统将申诉状态设为"审核中"，向平台管理员展示差评申诉申请，平台管理员可审核后决定是否撤销评价。

### Scenario: 买家举报违规书籍信息
- Trigger: 买家在搜索匹配模块查看书籍详情时
- Flow:
  1. 买家 在 搜索匹配 选择举报原因并提交举报 → 系统将书籍出售信息状态设为"被举报"，写入举报时间和举报原因
  2. 系统 在 消息通知 向平台管理员推送举报通知 → 消息通知状态设为"已推送"
- Success End: 书籍出售信息状态为"被举报"
- Exception Paths:
  - IF 买家未选择举报原因: 系统将举报操作状态设为"信息不完整"，向买家展示"请选择举报原因后再提交"，买家可选择原因后重新提交。
  - IF 同一买家对同一书籍已提交过举报: 系统将举报操作状态设为"已拦截"，向买家展示"你已举报过该信息，请等待平台处理"，买家可查看举报进度。

### Scenario: 平台管理员处理违规书籍信息
- Trigger: 平台管理员进入书籍发布模块，书籍出售信息处于"被举报"状态时
- Flow:
  1. 平台管理员 在 书籍发布 查看被举报的书籍信息 → 系统展示书籍详情及举报原因
  2. 平台管理员 在 书籍发布 确认违规并下架 → 系统将书籍出售信息状态设为"已下架"，写入下架时间和违规原因
  3. 平台管理员 在 评价系统 扣减违规用户信用积分 → 系统将违规用户信用积分减少，写入扣分记录
- Success End: 书籍出售信息状态为"已下架"
- Exception Paths:
  - IF 扣分后用户信用积分低于 60 分: 系统将该用户账户状态设为"已限制"，向用户展示"信用积分低于 60 分，限制发布和购买"，用户可联系平台管理员申诉。
  - IF 被举报内容经审核无违规: 系统将举报记录状态设为"举报无效"，向举报方展示"经核实，该内容未违规"，举报方可查看核实结果。

### Scenario: 买家通过消息联系卖家
- Trigger: 买家进入消息通知模块时
- Flow:
  1. 买家 在 消息通知 向卖家发送私信 → 系统将消息状态设为"已发送"，写入发送时间
  2. 系统 在 消息通知 向卖家推送消息提醒 → 消息提醒状态设为"已推送"
  3. 卖家 在 消息通知 查看并回复买家私信 → 系统将回复消息状态设为"已送达"，写入回复时间
- Success End: 回复消息状态为"已送达"
- Exception Paths:
  - IF 收件方账户状态为"已限制": 系统将私信操作状态设为"已限制"，向买家展示"该用户账户已被限制，无法私信"，买家可选择其他卖家。
  - IF 私信内容包含平台屏蔽词: 系统将消息状态设为"已拦截"，向买家展示"消息内容包含违规词汇，无法发送"，买家可修改内容后重新发送。

## Module Connections

| 上游模块 | 触发操作 | 下游模块 | 自动反应 |
|---------|---------|---------|---------|
| 书籍发布 | 书籍出售信息状态设为"已发布" | 搜索匹配 | 搜索匹配索引更新，新书在搜索结果和同课程推荐中可见 |
| 书籍发布 | 书籍出售信息状态设为"已发布" | 消息通知 | 消息通知向订阅该课程的买家推送新书上架通知，消息状态设为"已推送" |
| 搜索匹配 | 买家提交举报 | 书籍发布 | 书籍出售信息状态设为"被举报"，写入举报时间和举报原因 |
| 交易管理 | 订单状态设为"已完成" | 评价系统 | 评价系统为买卖双方生成评价任务，评价窗口状态设为"待互评"，写入 24 小时评价截止时间 |
| 评价系统 | 评价提交或默认好评生成 | 消息通知 | 消息通知向被评价方推送评价结果，消息状态设为"已推送" |
| 评价系统 | 用户信用积分低于 60 分 | 书籍发布 | 书籍发布拦截该用户的新增发布操作，已发布信息状态设为"已暂停" |
| 评价系统 | 用户信用积分低于 60 分 | 交易管理 | 交易管理拦截该用户的新增下单操作，展示"信用积分低于 60 分，限制发布和购买" |

## Trigger Rules

| 编号 | 触发条件 | 触发动作 | 通知对象 |
|------|---------|---------|---------|
| T1 | 评价截止时间到达且任一方评价状态不为"已提交" | 将未提交方评价状态设为"默认好评" | 未提交评价方 |
