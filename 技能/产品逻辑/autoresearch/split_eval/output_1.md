# Product Logic

## Instructions For Design & Coding AI
- 本文档定义系统运转逻辑：场景、流程、联动、触发规则。
- 页面设计必须覆盖本文档中的所有场景和联动点。
- 编码时的数据流和事件处理必须与本文档的触发规则对齐。
- 发现逻辑矛盾或缺口时，以本文档为准，向上游反馈。
- 校服按学校定制，不同学校款式不同，不能混发
- 大单（500 件以上）走对公转账，小单走在线支付
- 库存低于安全线时必须提醒采购补货
- 退货必须在发货后 7 天内申请，且校服未穿着

## Business Scenarios

### Scenario: 学校采购员提交订单
- Trigger: 学校采购员进入订单管理模块，选择商品并提交订单时触发
- Flow:
  1. 学校采购员 在 客户管理 确认所属学校档案存在 → 系统返回该学校已绑定的款式列表
  2. 学校采购员 在 商品管理 选择本校款式和尺码并填写数量 → 系统写入选中款式编号、尺码和数量
  3. 学校采购员 在 订单管理 提交订单 → 系统生成订单，订单状态设为"待付款"，写入下单时间
  4. 系统 在 支付结算 根据订单数量设定支付方式 → 订单数量 < 500 件时，支付方式设为"在线支付"；订单数量 ≥ 500 件时，支付方式设为"对公转账"
  5. 学校采购员 在 支付结算 完成支付 → 系统将订单状态设为"已付款"，写入支付时间
- Success End: 订单状态为"已付款"
- Exception Paths:
  - IF 学校采购员选择了非本校款式（校服按学校定制，不同学校款式不同，不能混发）: 系统将该商品选择状态设为"不可用"，向学校采购员展示"该款式不属于贵校，无法加入订单"，学校采购员可重新选择本校款式后继续下单。
  - IF 订单数量 ≥ 500 件但学校采购员尝试使用在线支付: 系统将支付方式状态设为"已拦截"，向学校采购员展示"500 件以上订单须使用对公转账"，学校采购员可切换为对公转账后重新提交。

### Scenario: 仓库管理员出库发货
- Trigger: 订单状态为"已付款"，仓库管理员进入库存管理模块执行出库操作时触发
- Flow:
  1. 仓库管理员 在 订单管理 查看待发货订单列表 → 系统展示所有状态为"已付款"的订单
  2. 仓库管理员 在 库存管理 按订单明细执行出库操作 → 系统扣减对应款式和尺码的库存数量，写入出库时间
  3. 仓库管理员 在 订单管理 提交发货确认 → 系统将订单状态设为"已发货"，写入发货时间
- Success End: 订单状态为"已发货"
- Exception Paths:
  - IF 库存数量不足以满足订单明细: 系统将该款式出库状态设为"库存不足"，向仓库管理员展示"[款式名称]库存不足，无法完成出库"，仓库管理员可暂停出库并联系校服零售商（老板）处理补货。
  - IF 出库后任一款式库存低于安全线（库存低于安全线时必须提醒采购补货）: 系统将该款式库存状态设为"预警"，向校服零售商（老板）展示"[款式名称]库存已低于安全线，请及时补货"，校服零售商（老板）可进入库存管理发起入库。

### Scenario: 学校采购员申请退货
- Trigger: 订单状态为"已发货"，学校采购员进入订单管理模块发起退货申请时触发
- Flow:
  1. 学校采购员 在 订单管理 选择目标订单并提交退货申请 → 系统记录申请时间，将退货申请状态设为"待审核"
  2. 校服零售商（老板） 在 订单管理 审核退货申请 → 系统展示订单发货时间、申请时间及备注信息
  3. 校服零售商（老板） 在 订单管理 确认通过退货 → 系统将退货申请状态设为"已批准"，订单状态设为"退货中"
  4. 仓库管理员 在 库存管理 确认收到退回校服 → 系统将对应款式和尺码库存数量加回，写入入库时间，订单状态设为"已退货"
  5. 校服零售商（老板） 在 支付结算 确认退款 → 系统将退款状态设为"已退款"，写入退款时间
- Success End: 订单状态为"已退货"，退款状态为"已退款"
- Exception Paths:
  - IF 退货申请提交时距发货时间已超过 7 天（退货必须在发货后 7 天内申请）: 系统将退货申请状态设为"已拒绝"，向学校采购员展示"退货申请已超过发货后 7 天有效期，无法受理"，学校采购员可联系校服零售商（老板）线下协商。
  - IF 校服已穿着（退货必须校服未穿着）: 系统将退货申请状态设为"已拒绝"，向学校采购员展示"校服已穿着，不符合退货条件"，学校采购员可联系校服零售商（老板）线下协商。

### Scenario: 仓库管理员库存入库
- Trigger: 仓库管理员进入库存管理模块，执行入库操作时触发
- Flow:
  1. 仓库管理员 在 库存管理 选择入库款式并录入到货数量 → 系统写入入库款式编号、数量和入库时间
  2. 仓库管理员 在 库存管理 提交入库确认 → 系统将对应款式账面库存数量增加，若原库存状态为"预警"则更新为"正常"
- Success End: 对应款式账面库存数量已更新，库存状态为"正常"
- Exception Paths:
  - IF 录入款式编号在商品管理中不存在: 系统将入库操作状态设为"无效"，向仓库管理员展示"该款式编号不存在，请先在商品管理中录入款式"，仓库管理员可联系校服零售商（老板）完成款式录入后重试。
  - IF 录入数量为 0 或负数: 系统将数量字段设为"无效值"，向仓库管理员展示"入库数量必须大于 0"，仓库管理员可修正数量后重新提交。

### Scenario: 仓库管理员库存盘点
- Trigger: 仓库管理员进入库存管理模块，发起盘点操作时触发
- Flow:
  1. 仓库管理员 在 库存管理 发起盘点 → 系统生成盘点任务，盘点状态设为"进行中"，写入盘点开始时间
  2. 仓库管理员 在 库存管理 逐款式录入实际库存数量 → 系统将录入值与账面库存对比，标记差异款式
  3. 仓库管理员 在 库存管理 提交盘点结果 → 系统以实际数量更新账面库存，盘点状态设为"已完成"，写入盘点完成时间
- Success End: 盘点状态为"已完成"
- Exception Paths:
  - IF 盘点结果与账面差异超出合理范围: 系统将差异款式标记状态设为"异常"，向校服零售商（老板）展示"[款式名称]盘点差异较大，请复核"，校服零售商（老板）可要求仓库管理员重新盘点该款式。
  - IF 仓库管理员提交盘点时部分款式未录入: 系统将未录入款式的盘点状态设为"未完成"，向仓库管理员展示"以下款式未录入实际数量，请补全后再提交"，仓库管理员可补录后重新提交。
  - IF 盘点完成后任一款式库存低于安全线（库存低于安全线时必须提醒采购补货）: 系统将该款式库存状态设为"预警"，向校服零售商（老板）展示"[款式名称]库存已低于安全线，请及时补货"，校服零售商（老板）可进入库存管理发起入库。

## Module Connections

| 上游模块 | 触发操作 | 下游模块 | 自动反应 |
|---------|---------|---------|---------|
| 客户管理 | 学校采购员确认所属学校档案 | 商品管理 | 系统将该学校已绑定款式列表传入商品选择界面，仅展示本校款式 |
| 商品管理 | 学校采购员选定款式、尺码和数量写入 | 订单管理 | 系统将选中款式编号、尺码和数量写入订单草稿 |
| 订单管理 | 订单提交，数量写入 | 支付结算 | 系统根据订单数量将支付方式字段设定（≥500件设为"对公转账"，<500件设为"在线支付"） |
| 支付结算 | 支付完成 | 订单管理 | 系统将订单状态从"待付款"更新为"已付款"，写入支付时间 |
| 订单管理 | 订单状态变为"已付款" | 库存管理 | 系统将该订单对应款式和尺码标记为"待出库"，显示在出库待处理列表 |
| 库存管理 | 出库确认，库存数量扣减 | 订单管理 | 系统将订单状态从"已付款"更新为"已发货"，写入发货时间 |
| 订单管理 | 退货申请状态设为"已批准" | 库存管理 | 系统将退回款式和尺码标记为"待入库"，等待仓库管理员确认收货 |
| 库存管理 | 退货入库确认完成，订单状态设为"已退货" | 支付结算 | 系统生成退款记录，将退款状态设为"待退款" |

## Trigger Rules

| 编号 | 触发条件 | 触发动作 | 通知对象 |
|------|---------|---------|---------|
| T1 | 系统定时检测：任一款式当前库存数量低于安全线 | 系统将该款式库存状态设为"预警"，向校服零售商（老板）推送"[款式名称]库存已低于安全线，请及时补货" | 校服零售商（老板） |

## Business Rules Traceability
- 校服按学校定制，不同学校款式不同，不能混发 → 体现在 Scenario "学校采购员提交订单" Exception Path（选择非本校款式被拦截）
- 大单（500 件以上）走对公转账，小单走在线支付 → 体现在 Scenario "学校采购员提交订单" Flow Step 4（系统根据数量锁定支付方式）及 Exception Path（大单尝试在线支付被拦截）
- 库存低于安全线时必须提醒采购补货 → 体现在 Scenario "仓库管理员出库发货" Exception Path（出库后低于安全线），Scenario "仓库管理员库存盘点" Exception Path（盘点后低于安全线），及 Trigger Rule T1（系统定时主动检测）
- 退货必须在发货后 7 天内申请，且校服未穿着 → 体现在 Scenario "学校采购员申请退货" Exception Path（超期拒绝、已穿着拒绝）
