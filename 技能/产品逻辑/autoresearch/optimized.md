## 执行模式
Master Agent 负责调度和验收，不亲自干活。每个步骤启动 SubAgent 执行。

## 工作人员

## Workflow

### Step 0：加载上下文（Master Agent）
执行 脚本/加载上下文.sh，强制读取所有引用内容

### Step 1：用户流程梳理
- 执行者：产品（SubAgent）
- 目标：从用户真实业务场景出发，走通所有主场景和补充场景
- 成功标准：主场景完整走通（有 Trigger 指明角色和条件、Success End、至少 2 条 Exception Paths），所有模块至少出现在 1 个场景中，所有业务规矩至少被 1 个场景体现，所有用户角色至少作为 1 个场景的操作者出现
- 输入：`design/project_context.md`（模块清单 + 业务规矩 + 用户画像）
- 输出：场景流程记录（每步格式：[操作者] 在 [模块名] 做 [操作] → 系统 [反应]）
- 执行逻辑：
  1. 读取 project_context.md，逐一提取：模块清单、业务规矩（逐条编号）、用户画像（逐个角色）
  2. 向用户确认模块清单是否仍然正确
  3. 找出主场景：问用户最常做的一件事是什么（用户沉默则基于模块推断 3 个候选）
  4. 逐步走通主场景：每步追问下一步做什么、在哪个模块操作、出问题怎么办
  5. 补充其他场景，循环直到所有模块和用户角色覆盖（每个场景应聚焦一个独立的核心用户目标，不与其他场景重复核心流程）
  6. 逐条核对业务规矩，确认每条都在某个场景的 Flow 或 Exception Path 中体现（特别注意数值阈值必须原样保留）
  7. **Flow 原子化检查**：逐一审查每个 Flow Step 的→右侧，数一下涉及几个模块名。如果 >1 个不同模块，必须拆成多步（每步只涉及一个模块的操作）
  8. **Success End 完整性检查**：对每个场景，拆解标题中的动作目标（如"确认交易并互评"→目标1:确认交易、目标2:互评），确认 Success End 为每个目标都写了对应的状态值
  9. **前置状态闭环检查**：列出所有场景 Trigger 中引用的前置状态，逐一在其他场景的 Flow 或 Success End 中找产生该状态的步骤。包括聚合状态（如"所有XX均已完成"）——须确认有场景逐个产生该状态。找不到来源的，必须补充缺失场景或调整 Trigger 措辞为可追溯的条件
- 关键原则：
  - 不讨论技术实现，只谈业务场景、操作顺序、触发条件
  - 不新增模块：只使用 project_context.md 中已有的模块
  - 每条流程必须有明确的起点、正常终点和至少 2 条异常出口
  - 每步只描述一个操作者的一个动作，不合并多步
  - 异常处理必须包含具体的系统状态变更或操作指令，不得用笼统的"提示错误"
  - **跨模块拆分原则**：→右侧涉及 2 个以上不同模块名时，必须拆成多步。例：`店员提交消费 → 系统写入消费记录，更新积分余额，检测等级阈值` 跨了消费记录+积分管理+会员管理 → 必须拆成 3 步

### Step 2：页面结构设计
- 执行者：产品（SubAgent）
- 目标：从 Flow 中提炼模块间的联动关系（Module Connections 是 Flow 的摘要视图，不是独立设计的"愿景"）
- 成功标准：模块联动无断头路，联动表上游≠下游（无自循环），每个模块至少出现一次，**每条联动都能指向 Flow 中的具体步骤**
- 输入：Step 1 场景流程记录
- 输出：模块联动关系表（模块A完成某操作 → 自动触发模块B的某反应）
- 执行逻辑：
  1. 逐一扫描 Step 1 所有场景的每个 Flow Step 和 Exception Path
  2. 找出跨模块的系统反应（→右侧的动作涉及与操作者所在模块不同的另一个模块）
  3. 将每个跨模块反应提取为联动表的一行
  4. **反向验证**：检查联动表中是否有任何一行在 Flow 中找不到对应步骤。找不到的行必须删除——禁止写"应该存在但 Flow 中没描述"的联动
- 质量要求：
  - 每行"自动反应"必须描述具体的数据传递或状态变更，不得用"触发流程"等泛泛表述
  - 上游模块和下游模块不得是同一个模块
  - 每个模块至少作为上游或下游出现一次
  - **联动表只做 Flow 的摘要提炼，不得包含 Flow 中不存在的联动**
- 向用户确认联动关系是否正确

### Step 3：交互逻辑定义
- 执行者：产品（SubAgent）
- 目标：梳理系统自动执行的行为（仅限无人操作触发的纯系统行为）
- 成功标准：每条触发规则有明确的触发条件、触发动作、通知对象；每条规则都能追溯到场景中的业务来源；数值阈值与 project_context.md 完全一致
- 输入：Step 1 场景流程 + Step 2 模块联动关系
- 输出：触发规则清单（触发条件 + 触发后动作 + 通知谁）
- 执行逻辑：
  1. 从 Flow 和 Exception Path 中识别"定时任务"和"阈值监控"类的系统自动行为
  2. 将这些提取为 Trigger Rules
  3. **去重检查**：逐条对比 Trigger Rule 与所有 Exception Path，如果某条 Trigger Rule 的逻辑已在 Exception Path 中完整描述（相同触发条件+相同动作），则删除该 Trigger Rule，不重复
  4. Trigger Rule 的定位：**只放定时任务、阈值检测等无人操作触发的纯系统自动行为**。用户操作触发的逻辑已在 Flow/Exception 中描述，不再写入 Trigger Rules
- 质量要求：
  - 每条 Trigger Rule 必须能在某个场景的 Flow 或 Exception Path 中找到业务来源
  - 不得有两条规则描述本质相同的触发逻辑
  - **不得与 Exception Path 或 Flow Step 内容重复**（如 Exception 已写"截止时间到达且未成团→系统取消并退款"，Trigger Rule 不再写同样的逻辑）
  - 数值阈值（金额、时间、数量等）必须与 project_context.md 中的业务规矩完全一致
- 向用户逐条确认

## 反模式（常见错误）

- ❌ Instructions 照搬模板通用文字，不加任何项目特有规则
- ❌ Instructions 对原文做语义推断或强化（如原文说"限制发布和购买"却写成"不得发布，也不得购买"——这增加了原文没有的"不得"措辞）→ 应直接摘抄原文
- ❌ Exception Path 只写"系统提示错误"、"系统拦截"、"系统处理" → 必须写明具体的系统状态变更（状态改为什么值）和后续动作（用户可以做什么）
- ❌ Flow 一步塞两个操作（"A做X并做Y"）→ 必须拆成两步
- ❌ Flow 一步包含两个操作者（"A和B一起做Z"或"A与B协商"）→ 必须拆成各自独立的步骤
- ❌ Flow 一步包含条件分支（"A做X或做Y"）→ 拆成两个独立步骤，或在主流程写一条、把另一条放 Exception Path
- ❌ Module Connections 上游=下游（自循环）→ 两个不同模块之间才算联动
- ❌ Module Connections "自动反应"写"触发流程" → 必须写具体传递了什么数据或改了什么状态
- ❌ Trigger Rule 凭空添加，在场景中找不到来源 → 每条规则都必须对应场景中的某个步骤或异常处理
- ❌ Trigger Rules 表完全为空（即使所有规则在 Exception Path 中已描述，也应保留定时/阈值类的系统自动行为作为 Trigger Rule）
- ❌ Module Connections 表中写了 Flow 里没有的联动（每条联动的触发操作都必须在某个场景的 Flow Step 或 Exception Path 中有对应步骤）
- ❌ Coverage Check 缩写场景名 → 必须与 Business Scenarios 标题一字不差
- ❌ 遗漏业务规矩中的数值阈值（金额、时间、数量）→ 必须原样保留，不得四舍五入或改写
- ❌ 给业务规矩添加原文中不存在的数值（如原文只说"12月31日"却写成"12月31日23:59"）→ 只写原文有的，不推断
- ❌ 把单向规则改写为双向规则（如原文说"大单走对公"不等于"小单禁止用对公"）→ 只转化原文明确说的，不添加原文没说的禁止
- ❌ 两个场景的核心目标重复（如"下单"和"提交订单"本质是一个场景）→ 每个场景必须有独立的核心用户目标
- ❌ 缺少 Business Rules Traceability 章节 → 每条业务规矩必须追溯到具体场景
- ❌ "用户"作为操作者（太泛）→ 必须用具体角色名称（如"学校采购员"而非"用户"）
- ❌ Trigger 写心理状态（如"有需求时"、"想要做某事时"）→ 必须写可观测的系统条件或用户操作（如"进入XX模块时"、"XX状态为YY时"）
- ❌ Trigger Rule 与某个 Exception Path 或 Flow Step 内容完全重复（如 Exception 已写"超7天系统拒绝"，Trigger Rule 又写"超7天→系统拒绝"）→ Trigger Rule 只放定时任务、阈值检测等纯系统自动行为，不重复 Flow/Exception 已描述的逻辑
- ❌ Success End 写"流程完成"/"已更新"/"已处理" → 必须写"XX状态为'YY'"格式的具体枚举值
- ❌ Success End 只覆盖场景标题的部分目标（如场景叫"确认交易并互评"但 Success End 只写订单状态不写评价状态）→ 场景标题涉及几个动作，Success End 就必须包含几个对应的状态值
- ❌ Business Rules Traceability 把不同规矩映射到同一个 Trigger Rule → 每条规矩的追溯必须指向确实对应的场景步骤或规则，不得混淆
- ❌ Business Rules Traceability 引用的 Trigger Rule 实际对应的是另一条规矩（如"照片规矩"引用了 T6 但 T6 实际是"逾期催交"规则）→ 引用前必须核实 Trigger Rule 的触发条件是否确实对应该条规矩
- ❌ Instructions 中指定了 scenario 原文未明确的对象（如原文说"提醒采购"，output 指定为"仓库管理员"）→ 只写原文说的，不具体化原文未指定的对象
- ❌ Module Connections 独立构造"应有的联动"而非从 Flow 提炼 → Module Connections 是 Flow 的摘要视图，每行必须对应 Flow 中已存在的跨模块步骤
- ❌ Flow 一步的→右侧跨了多个模块（如"写入消费记录，更新积分，检测等级"跨 3 个模块）→ 数→右侧的模块名数量，>1 个就拆成多步
- ❌ Flow 一步从模块A读取/检测，然后向模块B写入（如"检测宠物档案疫苗日期，通过则设预约状态"跨宠物档案+预约管理）→ 读取来源和写入目标必须是同一个模块，否则拆分
- ❌ Module Connections 把人工操作当自动联动（如 Flow 中仓库管理员手动执行入库，但联动表写成"审核通过→自动触发入库"）→ 联动表只放系统自动完成的跨模块反应
- ❌ Trigger Rule 重复 Exception Path 中的时间条件逻辑（如 Exception 已写"IF 超7天系统拒绝"，Trigger Rule 又写"超7天→系统拒绝"）→ 即使涉及时间条件，已在 Exception Path 完整描述的逻辑不再写入 Trigger Rule
- ❌ 场景 Trigger 引用的前置状态在整个文档中无产生来源 → 每个前置状态必须能在某个场景的 Flow/Success End 中找到产生点，找不到就补场景

## product_logic.md 模板

```markdown
# Product Logic

## Instructions For Design & Coding AI
- 本文档定义系统运转逻辑：场景、流程、联动、触发规则。
- 页面设计必须覆盖本文档中的所有场景和联动点。
- 编码时的数据流和事件处理必须与本文档的触发规则对齐。
- 发现逻辑矛盾或缺口时，以本文档为准，向上游反馈。
- [在此逐条摘抄 project_context.md 业务规矩章节的原文。严格要求：
  1. 一字不改地复制原文——不推断、不添加、不加前缀（如"不得违反："）、不做语气转换
  2. 数值原样：原文写什么就写什么，不改格式不加精度]

## Business Scenarios
（场景数量建议 3-7 个：太少覆盖不全，太多则拆分过细。按用户最常执行到最少执行的顺序排列）

### Scenario: [场景名——简短描述核心动作，如"学校采购员下单"而非"下单流程"]
- Trigger: [哪个角色 + 在什么可观测的系统状态/条件下 + 做什么操作时触发。禁止写心理状态如"有需求时"/"想要时"，只写可观测的条件如"进入XX模块时"]
- Flow:
  1. [一个操作者——具体角色名] 在 [模块名] 做 [一个操作] → 系统 [反应——包含状态变更或数据写入]
  2. ...
  规则：
  - 每步左侧只能有一个操作者做一件事（禁止两个操作者同步、禁止"做X或做Y"分支）
  - "系统"可作为操作者，但仅用于定时任务或阈值检测等无人工触发的自动动作（如"系统在12月31日执行积分清零"）。如果上一步人工操作触发了系统反应，该反应写在上一步的"→"右侧，不要拆成单独的"系统"步骤
  - 系统反应中"设状态+写时间戳"算同一操作（时间戳是状态变更的附属记录），但"设状态+推通知"必须拆分（推通知属于通知管理模块，是跨模块动作）。判断标准：→右侧的所有动作是否都发生在同一个模块内？是→可合并；否→必须拆分
  - **跨模块拆分判断法**：一步中涉及的所有模块名（包括读取来源和写入目标）必须是同一个模块。如果从模块A读取/检测，然后向模块B写入/更新，这算跨了两个模块，必须拆分。
    ✓ 正例：`店员在消费记录提交消费 → 系统写入消费记录`（读写都在消费记录，单模块）
    ✗ 反例1：`店员提交消费 → 系统写入消费记录，更新积分余额，检测等级阈值`（跨消费记录+积分管理+会员管理 3 个模块）→ 拆成 3 步
    ✗ 反例2：`系统检测宠物档案中疫苗日期，通过则将预约状态设为"待付定金"`（从宠物档案读取 + 向预约管理写入 = 跨 2 个模块）→ 拆成 2 步：
      1. 系统在宠物档案检测疫苗接种日期是否在30天内 → 系统记录检测结果
      2. 系统在预约管理根据检测结果将预约状态设为"待付定金" → 系统记录状态变更
    ✗ 反例3：`系统查询并展示会员等级、年度消费额、当前积分余额`（从会员管理+积分管理读取 = 跨 2 个模块）→ 拆成 2 步
```
