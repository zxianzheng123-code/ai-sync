# Product Logic

## Instructions For Design & Coding AI
- 本文档定义系统运转逻辑：场景、流程、联动、触发规则。
- 页面设计必须覆盖本文档中的所有场景和联动点。
- 编码时的数据流和事件处理必须与本文档的触发规则对齐。
- 发现逻辑矛盾或缺口时，以本文档为准，向上游反馈。
- 校服按学校定制，不同学校款式不同，不能混发
- 大单（500 件以上）走对公转账，小单走在线支付
- 库存低于安全线时必须提醒采购补货
- 退货必须在发货后 7 天内申请，且校服未穿着

## Business Scenarios

### Scenario: 校服零售商（老板）维护客户档案并绑定款式
- Trigger: 校服零售商（老板）进入客户管理模块，为新学校建档或更新已有学校信息时触发
- Flow:
  1. 校服零售商（老板）在客户管理录入学校信息（学校名称、联系人） → 系统在客户管理将客户档案状态设为"已建档"，记录建档时间
  2. 校服零售商（老板）在商品管理为该学校绑定定制款式 → 系统在商品管理将该学校与对应款式的关联状态设为"已绑定"
- Success End: 客户档案状态为"已建档"，学校与定制款式关联状态为"已绑定"
- Exception Paths:
  - IF 录入的学校名称已存在于客户管理：系统在客户管理将本次操作状态设为"已拦截"，向校服零售商（老板）展示"该学校已建档，请勿重复录入"，校服零售商（老板）可进入已有档案进行信息更新。
  - IF 为学校绑定的款式已被其他学校绑定：系统在商品管理将本次操作状态设为"已拦截"，向校服零售商（老板）展示"该款式已属于其他学校，不能混发"，校服零售商（老板）可选择其他款式后重新绑定。

### Scenario: 仓库管理员入库补货
- Trigger: 仓库管理员进入库存管理模块，对收到的货物执行入库登记时触发
- Flow:
  1. 仓库管理员在库存管理录入入库信息（款式、尺码、数量） → 系统在库存管理将对应款式可用库存数量增加，写入入库记录
  2. 系统在库存管理检查入库后库存是否恢复至安全线以上 → 系统在库存管理记录检查结果
  3. 系统在库存管理根据检查结果将该款式库存状态更新为"正常" → 系统记录状态变更时间
- Success End: 入库记录状态为"已写入"，库存状态为"正常"
- Exception Paths:
  - IF 录入的款式在商品管理中不存在：系统在库存管理将本次入库操作状态设为"已拦截"，向仓库管理员展示"款式不存在，请先联系商家在商品管理中录入该款式"，仓库管理员可通知校服零售商（老板）录入新款式后重新入库。
  - IF 入库后库存仍低于安全线：系统在库存管理将该款式库存状态保持为"低于安全线"，向校服零售商（老板）展示"[款式名]入库后仍低于安全线，建议继续补货"，校服零售商（老板）可再次安排补货。

### Scenario: 学校采购员下单并支付
- Trigger: 学校采购员进入商品管理模块，客户档案状态为"已建档"且学校与定制款式关联状态为"已绑定"，选定校服款式后发起下单操作时触发
- Flow:
  1. 学校采购员在商品管理查看本校对应款式及尺码表 → 系统在商品管理展示该学校已绑定款式和尺码信息
  2. 学校采购员在订单管理提交订单（含款式、数量、尺码） → 系统在订单管理将订单状态设为"待支付"，记录下单时间
  3. 系统在订单管理检查订单数量是否达到 500 件 → 系统在订单管理记录检查结果
  4. 系统在支付结算根据检查结果生成对应支付方式账单 → 系统在支付结算将账单状态设为"待支付"
  5. 学校采购员在支付结算完成在线支付或上传对公转账凭证 → 系统在支付结算将账单状态设为"已支付"，记录支付时间
  6. 系统在订单管理将订单状态更新为"待出库" → 系统在订单管理记录状态变更时间
- Success End: 订单状态为"待出库"，账单状态为"已支付"
- Exception Paths:
  - IF 学校采购员提交的款式不属于该学校已绑定款式：系统在商品管理将本次操作状态设为"已拦截"，向学校采购员展示"所选款式与学校不符，不能混发"，学校采购员可重新选择正确款式后再次提交。
  - IF 订单数量 ≥ 500 件但学校采购员选择在线支付：系统在支付结算将账单状态设为"支付方式错误"，向学校采购员展示"500 件以上订单须走对公转账"，学校采购员可切换为对公转账方式后重新提交。
  - IF 学校客户档案不存在：系统在客户管理将本次操作状态设为"已暂停"，向学校采购员展示"该学校尚未建档，请联系零售商"，学校采购员可等待校服零售商（老板）完成建档后重新下单。

### Scenario: 仓库管理员出库发货
- Trigger: 仓库管理员进入订单管理模块，发现存在状态为"待出库"的订单时触发
- Flow:
  1. 仓库管理员在订单管理查看待出库订单详情（学校、款式、数量、尺码） → 系统在订单管理展示订单详情
  2. 仓库管理员在库存管理提交出库操作（核对款式与订单一致后填写出库数量） → 系统在库存管理将对应款式可用库存数量减去出库数量，记录出库时间
  3. 系统在库存管理检查出库后库存是否低于安全线 → 系统在库存管理记录检查结果
  4. 系统在库存管理根据检查结果将该款式库存状态设为"低于安全线" → 系统在库存管理记录状态变更时间
  5. 系统在订单管理将订单状态更新为"已发货"，记录发货时间 → 系统在订单管理记录状态变更
- Success End: 订单状态为"已发货"，库存出库记录状态为"已写入"
- Exception Paths:
  - IF 库存可用数量不足以满足订单数量：系统在库存管理将出库申请状态设为"库存不足"，向仓库管理员展示"当前库存不足，无法完成出库"，仓库管理员可等待入库补货后重新提交出库。
  - IF 出库款式与订单要求款式不一致：系统在库存管理将出库操作状态设为"款式不匹配"，向仓库管理员展示"出库款式与订单不符，请核实后重新填写"，仓库管理员可修改出库信息后重新提交。

### Scenario: 学校采购员申请退货
- Trigger: 学校采购员进入订单管理模块，对状态为"已发货"的订单发起退货申请时触发
- Flow:
  1. 学校采购员在订单管理选择已发货订单并提交退货申请（含退货原因） → 系统在订单管理记录申请时间
  2. 系统在订单管理检查申请时间距发货时间是否在 7 天内 → 系统在订单管理记录检查结果
  3. 系统在订单管理根据检查结果将退货申请状态设为"待审核" → 系统在订单管理记录状态变更时间
  4. 校服零售商（老板）在订单管理审核退货申请（含核查校服是否未穿着） → 系统在订单管理将退货申请状态设为"审核通过"，记录审核时间
  5. 系统在支付结算生成退款账单 → 系统在支付结算将退款账单状态设为"待退款"
  6. 校服零售商（老板）在支付结算确认执行退款 → 系统在支付结算将退款账单状态设为"已退款"，记录退款时间
  7. 系统在订单管理将订单状态更新为"已退货" → 系统在订单管理记录状态变更时间
- Success End: 退货申请状态为"审核通过"，退款账单状态为"已退款"，订单状态为"已退货"
- Exception Paths:
  - IF 退货申请提交时距发货时间已超过 7 天：系统在订单管理将退货申请状态设为"已拒绝"，向学校采购员展示"退货申请已超过 7 天期限，无法受理"，学校采购员可联系校服零售商（老板）协商处理。
  - IF 校服零售商（老板）审核时标记校服已穿着：系统在订单管理将退货申请状态设为"已拒绝"，向学校采购员展示"校服已穿着，不符合退货条件"，学校采购员可查看拒绝原因后关闭申请。

### Scenario: 校服零售商（老板）对账查账
- Trigger: 校服零售商（老板）进入支付结算模块，查看指定时间段内账单情况时触发
- Flow:
  1. 校服零售商（老板）在支付结算设置查账时间范围并提交查询 → 系统在支付结算展示该时间段内所有账单列表（含支付方式、金额、状态）
  2. 校服零售商（老板）在支付结算选择单笔账单查看明细 → 系统在支付结算展示该账单关联的订单信息和支付记录
  3. 校服零售商（老板）在支付结算生成对账报告 → 系统在支付结算将对账报告状态设为"已生成"，记录生成时间
- Success End: 账单列表状态为"已展示"，对账报告状态为"已生成"
- Exception Paths:
  - IF 查询时间范围内无账单记录：系统在支付结算将查询结果状态设为"无记录"，向校服零售商（老板）展示"该时间段内暂无账单"，校服零售商（老板）可调整时间范围重新查询。
  - IF 查询结果中存在账单状态为"待支付"超过 30 天：系统在支付结算将该账单标记为"逾期未支付"，向校服零售商（老板）展示"存在逾期未支付账单，请跟进"，校服零售商（老板）可联系对应学校采购员催款。

## Module Connections

| 上游模块 | 触发操作 | 下游模块 | 自动反应 |
|---------|---------|---------|---------|
| 商品管理 | 校服零售商（老板）为学校绑定定制款式 | 客户管理 | 系统将该学校与对应款式的关联状态写入客户档案，关联状态设为"已绑定" |
| 订单管理 | 系统检查订单数量达到 500 件 | 支付结算 | 系统生成对公转账账单，账单支付方式标记设为"对公转账"，账单状态设为"待支付" |
| 订单管理 | 系统检查订单数量未达到 500 件 | 支付结算 | 系统生成在线支付账单，账单支付方式标记设为"在线支付"，账单状态设为"待支付" |
| 支付结算 | 学校采购员完成支付，账单状态设为"已支付" | 订单管理 | 系统将订单状态更新为"待出库" |
| 库存管理 | 仓库管理员提交出库操作 | 订单管理 | 系统将订单状态更新为"已发货"，写入发货时间戳 |
| 订单管理 | 校服零售商（老板）审核退货申请通过 | 支付结算 | 系统生成退款账单，退款账单状态设为"待退款" |
| 支付结算 | 校服零售商（老板）确认退款，退款账单状态设为"已退款" | 订单管理 | 系统将订单状态更新为"已退货" |

## Trigger Rules

| 编号 | 触发条件 | 触发动作 | 通知对象 |
|------|---------|---------|---------|
| T1 | 库存管理中任一款式出库后可用库存数量低于安全线 | 系统将该款式库存状态设为"低于安全线" | 校服零售商（老板） |

## Business Rules Traceability
- 校服按学校定制，不同学校款式不同，不能混发 → 体现在 Scenario "校服零售商（老板）维护客户档案并绑定款式" Exception Path（IF 款式已被其他学校绑定）；Scenario "学校采购员下单并支付" Exception Path（IF 款式不属于该学校已绑定款式）；Scenario "仓库管理员出库发货" Exception Path（IF 出库款式与订单要求款式不一致）
- 大单（500 件以上）走对公转账，小单走在线支付 → 体现在 Scenario "学校采购员下单并支付" Flow Step 3（系统检查订单数量是否达到 500 件）、Flow Step 4（生成对应支付方式账单）；Exception Path（IF 500 件以上但选择在线支付）
- 库存低于安全线时必须提醒采购补货 → 体现在 Scenario "仓库管理员出库发货" Flow Step 3-4（系统检测出库后库存是否低于安全线并设状态）；Exception Path（IF 出库后库存低于安全线）；Trigger Rule T1
- 退货必须在发货后 7 天内申请，且校服未穿着 → 体现在 Scenario "学校采购员申请退货" Flow Step 2（系统检查申请时间距发货时间是否在 7 天内）；Exception Path（IF 超过 7 天）；Exception Path（IF 校服已穿着）
