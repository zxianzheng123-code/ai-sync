# Product Logic

## Instructions For Design & Coding AI

- 页面设计必须覆盖本文档中的所有场景和联动点。
- 编码时的数据流和事件处理必须与本文档的触发规则对齐。
- 发现逻辑矛盾或缺口时，以本文档为准，向上游反馈。
- 预约必须提前至少 24 小时，节假日提前 72 小时
- 宠物必须有 30 天内的疫苗接种记录才能入住
- 取消预约提前 48 小时全额退定金，否则扣 50%
- 寄养期间每天至少上传 1 张宠物照片给主人

## Business Scenarios

### Scenario: 宠物主人提交寄养预约
- Trigger: 宠物主人进入预约管理模块，选择寄养日期和房间类型后点击提交预约时触发
- Flow:
  1. 宠物主人在预约管理选择寄养开始日期和结束日期 → 系统在预约管理检测所选开始日期距当前时间是否满足提前 24 小时（节假日 72 小时）要求，记录检测结果
  2. 宠物主人在预约管理选择房间类型 → 系统在预约管理将房间类型写入当前预约草稿
  3. 宠物主人在预约管理点击提交预约 → 系统在宠物档案检测该宠物是否存在 30 天内的疫苗接种记录，记录检测结果
  4. 系统在预约管理根据步骤 1 和步骤 3 的检测结果将预约状态设为"待付定金"，写入预约单号、宠物主人、日期、房间类型信息
  5. 系统在通知管理向宠物主人推送"预约待付定金"通知，通知状态设为"已发送"
- Success End: 预约状态为"待付定金"，通知状态为"已发送"
- Exception Paths:
  - IF 所选日期距当前时间不足 24 小时（非节假日）: 系统在预约管理将当前选择状态设为"日期不可用"，向宠物主人展示"预约须提前至少 24 小时，请重新选择日期"，宠物主人可重新选择日期。
  - IF 所选日期为节假日且距当前时间不足 72 小时: 系统在预约管理将当前选择状态设为"日期不可用"，向宠物主人展示"节假日须提前至少 72 小时预约，请重新选择日期"，宠物主人可重新选择日期。
  - IF 宠物档案中无 30 天内的疫苗接种记录: 系统在预约管理将预约状态设为"已拒绝"，向宠物主人展示"宠物需有 30 天内的疫苗接种记录方可预约，请先更新宠物档案"，宠物主人可前往宠物档案上传疫苗记录后重新提交预约。

### Scenario: 宠物主人支付定金确认预约
- Trigger: 宠物主人进入支付管理模块，当前存在预约状态为"待付定金"的预约时触发
- Flow:
  1. 宠物主人在支付管理选择目标预约并提交定金支付 → 系统在支付管理将定金支付状态设为"已付定金"，写入支付记录和支付时间
  2. 系统在预约管理将预约状态更新为"已确认"，写入状态变更时间戳
  3. 系统在通知管理向宠物主人推送"预约已确认"通知，通知状态设为"已发送"
- Success End: 定金支付状态为"已付定金"，预约状态为"已确认"，通知状态为"已发送"
- Exception Paths:
  - IF 支付失败（支付渠道返回失败）: 系统在支付管理将支付状态设为"支付失败"，向宠物主人展示"支付未成功，请检查支付方式后重试"，宠物主人可重新提交支付。
  - IF 定金支付超时（超出支付等待时限仍未完成）: 系统在支付管理将支付状态设为"已超时"，向宠物主人展示"定金支付已超时，请重新发起支付"，宠物主人可重新进入支付管理完成支付。

### Scenario: 宠物主人取消预约
- Trigger: 宠物主人进入预约管理模块，当前存在预约状态为"已确认"的预约，点击取消预约时触发
- Flow:
  1. 宠物主人在预约管理点击取消目标预约 → 系统在预约管理将预约状态设为"取消处理中"，记录取消请求时间
  2. 系统在支付管理检测取消请求时间距寄养开始日期是否超过 48 小时，记录检测结果及对应退款比例（≥48小时则100%，<48小时则50%）
  3. 系统在支付管理按检测结果计算退款金额，将退款记录状态设为"待退款"，写入退款金额
  4. 系统在预约管理将预约状态设为"已取消"，写入取消时间戳
  5. 系统在通知管理向宠物主人推送取消确认及退款金额通知，通知状态设为"已发送"
- Success End: 预约状态为"已取消"，退款记录状态为"待退款"，通知状态为"已发送"
- Exception Paths:
  - IF 取消时距寄养开始日期不足 48 小时: 系统在支付管理将退款金额设为定金的 50%，向宠物主人展示"取消时间不足 48 小时，仅退还定金的 50%"，宠物主人可确认后继续或撤回取消操作。
  - IF 退款处理失败（支付渠道返回失败）: 系统在支付管理将退款记录状态设为"退款失败"，向宠物主人展示"退款处理失败，请联系店长处理"，店长可在支付管理手动重新发起退款。

### Scenario: 寄养员上传每日寄养日志
- Trigger: 寄养员进入寄养日志模块，当日存在预约状态为"寄养中"的宠物且当日日志状态非"已提交"时触发
- Flow:
  1. 寄养员在寄养日志填写当日喂食记录 → 系统在寄养日志将喂食记录写入当日日志草稿
  2. 寄养员在寄养日志填写活动记录和健康观察内容 → 系统在寄养日志将活动记录和健康观察内容写入当日日志草稿
  3. 寄养员在寄养日志上传宠物照片（至少 1 张） → 系统在寄养日志保存照片并记录上传时间
  4. 寄养员在寄养日志点击提交当日日志 → 系统在寄养日志将当日日志状态设为"已提交"，记录提交时间
  5. 系统在通知管理向宠物主人推送每日寄养报告，内容包含喂食记录、活动记录、健康观察及照片，通知状态设为"已发送"
- Success End: 当日日志状态为"已提交"，通知状态为"已发送"
- Exception Paths:
  - IF 寄养员提交日志时当日照片上传数量为 0: 系统在寄养日志拒绝提交，将当日日志状态保持为"草稿"，向寄养员展示"请至少上传 1 张宠物照片后再提交"，寄养员可上传照片后重新提交。
  - IF 通知推送失败（通知管理返回失败）: 系统在通知管理将通知状态设为"发送失败"，向寄养员展示"报告推送失败，系统将稍后重试"，寄养日志状态保持"已提交"不受影响。

### Scenario: 店长确认宠物入住
- Trigger: 店长进入预约管理模块，存在预约状态为"已确认"且寄养开始日期为当天的预约时触发
- Flow:
  1. 店长在预约管理查看当天待入住的预约列表 → 系统在预约管理展示预约状态为"已确认"且寄养开始日期为当天的所有预约
  2. 店长在预约管理点击确认宠物已到店 → 系统在预约管理将预约状态设为"寄养中"，写入入住时间戳
  3. 系统在宠物档案读取该宠物的特殊需求备注 → 系统将特殊需求信息展示给寄养员
- Success End: 预约状态为"寄养中"，特殊需求备注状态为"已展示"
- Exception Paths:
  - IF 宠物主人未在寄养开始日期当天到店（当天店长未操作确认入住）: 系统在预约管理将预约状态设为"待处理"，向店长展示"该预约宠物未按时到店，请联系宠物主人确认"，店长可手动标记改期或取消。
  - IF 宠物到店后店长发现疫苗记录与档案不符: 系统在预约管理将预约状态设为"待核实"，向店长展示"疫苗记录存疑，请核实后再确认入住"，店长可要求宠物主人现场更新宠物档案后重新确认入住。

### Scenario: 宠物主人更新宠物档案疫苗记录
- Trigger: 宠物主人进入宠物档案模块，点击新增疫苗接种记录时触发
- Flow:
  1. 宠物主人在宠物档案填写疫苗名称、接种日期、接种机构 → 系统在宠物档案记录填写内容
  2. 宠物主人在宠物档案点击提交疫苗记录 → 系统在宠物档案将疫苗接种记录状态设为"已保存"，写入接种日期
- Success End: 疫苗接种记录状态为"已保存"
- Exception Paths:
  - IF 接种日期填写为未来日期: 系统在宠物档案拒绝保存，向宠物主人展示"接种日期不能为未来日期，请核实后重新填写"，宠物主人可修改日期后重新提交。
  - IF 必填字段（疫苗名称、接种日期）为空: 系统在宠物档案拒绝保存，向宠物主人展示"请填写完整的疫苗名称和接种日期"，宠物主人可补全信息后重新提交。

### Scenario: 宠物主人支付寄养尾款
- Trigger: 宠物主人进入支付管理模块，当前存在尾款支付状态为"待付尾款"的预约时触发
- Flow:
  1. 宠物主人在支付管理查看目标预约的尾款明细 → 系统在支付管理展示尾款金额和计费明细
  2. 宠物主人在支付管理提交尾款支付 → 系统在支付管理将尾款支付状态设为"已付尾款"，写入支付记录和支付时间
  3. 系统在预约管理将预约状态更新为"已完成"，写入完成时间戳
  4. 系统在通知管理向宠物主人推送寄养完成通知，通知状态设为"已发送"
- Success End: 尾款支付状态为"已付尾款"，预约状态为"已完成"，通知状态为"已发送"
- Exception Paths:
  - IF 尾款支付失败（支付渠道返回失败）: 系统在支付管理将尾款支付状态设为"支付失败"，向宠物主人展示"支付未成功，请检查支付方式后重试"，宠物主人可重新提交支付。
  - IF 宠物主人超时未支付尾款: 系统在支付管理将尾款支付状态设为"逾期未付"，向宠物主人展示"尾款支付逾期，请尽快完成支付或联系店长"，宠物主人可重新进入支付管理完成支付。
