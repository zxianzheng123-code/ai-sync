# Product Logic

## Instructions For Design & Coding AI
- 本文档定义系统运转逻辑：场景、流程、联动、触发规则。
- 页面设计必须覆盖本文档中的所有场景和联动点。
- 编码时的数据流和事件处理必须与本文档的触发规则对齐。
- 发现逻辑矛盾或缺口时，以本文档为准，向上游反馈。
- 不得在未达最低成团人数的情况下将团购状态设为"已成团"。
- 不得在配送状态不为"已收货"的情况下允许发起结算。
- 团购必须在截止时间前达到最低成团人数，否则自动取消并退款
- 团长佣金按实际成交金额的 10% 计算
- 供应商发货后 48 小时内团长必须确认收货
- 消费者自提超过 3 天未取，视为自动确认收货

## Business Scenarios

### Scenario: 团长发布团购
- Trigger: 团长进入团购发起模块，点击提交发布操作时触发
- Flow:
  1. 团长在团购发起选择商品 → 系统记录选品信息
  2. 团长在团购发起填写团购价、截止时间和最低成团人数 → 系统保存参数草稿
  3. 团长在团购发起点击发布 → 系统将团购状态设为"招募中"，写入发布时间戳
  4. 系统在拼单管理为该团购创建拼单记录，状态设为"招募中" → 系统写入最低成团人数和截止时间
- Success End: 团购状态为"招募中"；拼单记录状态为"招募中"
- Exception Paths:
  - IF 截止时间未填写：系统将发布操作拦截，向团长展示"截止时间不能为空，请补充后再发布"，团长可补填截止时间后重新提交。
  - IF 最低成团人数未填写：系统将发布操作拦截，向团长展示"请设置最低成团人数"，团长可补填后重新提交。

### Scenario: 消费者加入拼单
- Trigger: 消费者进入拼单管理，打开状态为"招募中"的团购并提交参团时触发
- Flow:
  1. 消费者在拼单管理选择规格并提交参团 → 系统在拼单管理创建拼单记录，将消费者参团状态设为"已加入"
  2. 系统在拼单管理检测当前已参团人数是否达到最低成团人数 → 系统记录检测结果
  3. 系统在拼单管理将团购状态设为"已成团" → 系统写入成团时间戳，锁定团购不再接受新拼单
- Success End: 消费者参团状态为"已加入"；团购状态为"已成团"
- Exception Paths:
  - IF 截止时间到达且已参团人数未达最低成团人数：系统将团购状态设为"已取消"，向所有已参团消费者展示"团购未成功，款项将退回原渠道"，系统在结算管理发起退款，将所有参团消费者退款状态设为"退款中"。
  - IF 消费者重复提交同一团购参团：系统将操作拦截，向消费者展示"您已参与该团购，不可重复加入"，消费者可查看已参团记录。

### Scenario: 供应商发货与团长确认收货
- Trigger: 团购状态为"已成团"，供应商进入配送管理点击发货操作时触发
- Flow:
  1. 供应商在配送管理录入物流单号并点击发货 → 系统将配送状态设为"已发货"，写入发货时间戳
  2. 系统在配送管理将收货截止时间设为发货时间 +48 小时 → 系统写入收货截止时间
  3. 团长在配送管理点击确认收货 → 系统将配送状态设为"已收货"，写入收货时间戳
  4. 系统在结算管理将该团购结算记录状态设为"可结算" → 系统写入可结算标记
- Success End: 配送状态为"已收货"；结算记录状态为"可结算"
- Exception Paths:
  - IF 供应商发货后满 48 小时团长仍未确认收货：系统将配送状态设为"收货超时"，向团长展示"收货已超时，请尽快确认收货"，团长可在配送管理点击确认收货。
  - IF 供应商录入的物流单号为空：系统将发货操作拦截，向供应商展示"物流单号不能为空，请填写后提交"，供应商可补填后重新提交。

### Scenario: 消费者自提
- Trigger: 配送状态为"已收货"，消费者进入配送管理完成自提时触发
- Flow:
  1. 消费者在配送管理确认已取货并提交 → 系统将消费者自提状态设为"已自提"，写入自提时间戳
- Success End: 消费者自提状态为"已自提"
- Exception Paths:
  - IF 消费者进入自提页面时配送状态不为"已收货"：系统在配送管理将页面状态设为"暂不可自提"，向消费者展示"当前订单尚未到达自提阶段"，消费者可返回拼单管理查看团购进度。
  - IF 消费者到达自提点时商品缺货：系统将消费者自提状态设为"异常"，向消费者展示"商品存在缺货情况，请联系团长处理"，消费者可联系团长协商解决。

### Scenario: 团长发起佣金与货款结算
- Trigger: 结算记录状态为"可结算"，团长进入结算管理点击发起结算操作时触发
- Flow:
  1. 团长在结算管理点击发起结算 → 系统将结算状态设为"结算申请中"，写入申请时间
  2. 系统在结算管理读取该团购实际成交金额 → 系统写入结算基础金额
  3. 系统在结算管理按实际成交金额的 10% 计算并写入团长佣金金额 → 系统记录团长应得佣金
  4. 系统在结算管理计算平台抽成并写入供应商货款金额 → 系统记录供应商应收货款
  5. 系统在结算管理将团长佣金状态设为"已结算" → 系统写入团长到账金额和结算时间戳
  6. 系统在结算管理将供应商货款状态设为"已结算" → 系统写入供应商到账金额和结算时间戳
- Success End: 团长佣金状态为"已结算"；供应商货款状态为"已结算"
- Exception Paths:
  - IF 结算记录状态不为"可结算"时团长尝试发起结算：系统将结算操作拦截，向团长展示"请确认收货后再发起结算"，团长可返回配送管理完成收货确认。
  - IF 打款失败（收款账号信息有误）：系统将结算状态设为"打款失败"，向团长和供应商展示"账号信息有误，请更新收款账号后重新发起结算"，团长和供应商可在用户管理更新账号后重新操作。

### Scenario: 供应商入驻申请
- Trigger: 供应商进入用户管理，点击提交入驻申请时触发
- Flow:
  1. 供应商在用户管理填写资质信息并提交入驻申请 → 系统将供应商账号状态设为"待审核"
  2. 系统在用户管理完成审核 → 系统将供应商账号状态设为"已认证"，写入认证通过时间
- Success End: 供应商账号状态为"已认证"
- Exception Paths:
  - IF 提交的资质信息不完整：系统将操作拦截，向供应商展示"请补充完整资质信息后提交"，供应商可补充后重新提交。
  - IF 审核不通过：系统将供应商账号状态设为"已拒绝"，向供应商展示"申请未通过，请根据反馈修改资质信息后重新提交"，供应商可修改后重新申请。

### Scenario: 团长认证申请
- Trigger: 消费者进入用户管理，点击提交团长认证申请时触发
- Flow:
  1. 消费者在用户管理填写社区信息并提交认证申请 → 系统将团长账号状态设为"待审核"
  2. 系统在用户管理完成审核 → 系统将团长账号状态设为"已认证"，写入认证通过时间
- Success End: 团长账号状态为"已认证"
- Exception Paths:
  - IF 提交的社区信息不完整：系统将操作拦截，向消费者展示"请补充完整社区信息后提交"，消费者可补充后重新提交。
  - IF 审核不通过：系统将团长账号状态设为"已拒绝"，向消费者展示"认证未通过，请根据反馈修改信息后重新提交"，消费者可修改后重新申请。

## Module Connections

| 上游模块 | 触发操作 | 下游模块 | 自动反应 |
|---------|---------|---------|---------|
| 团购发起 | 团长点击发布，团购状态设为"招募中" | 拼单管理 | 为该团购创建拼单记录，状态设为"招募中"，写入最低成团人数和截止时间 |
| 拼单管理 | 截止时间到达，团购状态设为"已取消" | 结算管理 | 为所有已参团消费者发起退款，将退款状态设为"退款中" |
| 配送管理 | 团长点击确认收货，配送状态设为"已收货" | 结算管理 | 将该团购结算记录状态设为"可结算"，写入可结算标记 |
