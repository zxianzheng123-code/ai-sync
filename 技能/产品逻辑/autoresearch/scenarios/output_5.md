# Product Logic

## Instructions For Design & Coding AI
- 本文档定义系统运转逻辑：场景、流程、联动、触发规则。
- 页面设计必须覆盖本文档中的所有场景和联动点。
- 编码时的数据流和事件处理必须与本文档的触发规则对齐。
- 发现逻辑矛盾或缺口时，以本文档为准，向上游反馈。
- 卖家定价不得超过原价的 70%
- 交易完成后 24 小时内双方可互评，超时默认好评
- 信用积分低于 60 分的用户限制发布和购买
- 同一本书最多同时发布 3 条出售信息（防刷单）

## Business Scenarios

### Scenario: 卖家发布书籍出售信息
- Trigger: 卖家（毕业生）进入书籍发布模块，发起新建出售信息操作时触发
- Flow:
  1. 卖家（毕业生）在书籍发布填写书籍信息（书名、ISBN、原价、定价、新旧程度）并上传照片 → 系统暂存填写内容
  2. 卖家（毕业生）在书籍发布点击"提交发布" → 系统在评价系统读取该卖家当前信用积分
  3. 系统在评价系统检测该卖家信用积分是否低于 60 分 → 系统记录检测结果
  4. 系统在书籍发布根据检测结果检查该卖家对同一本书当前有效发布条数是否已达 3 条 → 系统记录检查结果
  5. 系统在书籍发布根据检查结果检验定价是否超过原价的 70% → 系统记录检验结果
  6. 系统在书籍发布根据检验结果将出售信息状态设为"已发布"，记录发布时间 → 系统完成发布
  7. 系统在消息通知向关注该课程的买家（新生）推送新书上架通知 → 系统记录通知发送时间
- Success End: 出售信息状态为"已发布"
- Exception Paths:
  - IF 卖家信用积分低于 60 分: 系统将本次操作状态设为"已拦截"，向卖家（毕业生）展示"信用积分低于60分，已限制发布"，卖家（毕业生）可关闭操作或查看积分详情。
  - IF 同一本书当前有效发布条数已达 3 条: 系统将本次操作状态设为"已拦截"，向卖家（毕业生）展示"同一本书最多同时发布3条出售信息"，卖家（毕业生）可先下架已有信息后再重新发布。
  - IF 填写的定价超过原价的 70%: 系统将本次操作状态设为"已拦截"，向卖家（毕业生）展示"定价不得超过原价的70%"，卖家（毕业生）可修改定价后重新提交。

### Scenario: 买家搜索并查看书籍详情
- Trigger: 买家（新生）进入搜索匹配模块，输入课程/ISBN/书名关键词后提交搜索时触发
- Flow:
  1. 买家（新生）在搜索匹配输入关键词并提交搜索 → 系统在搜索匹配返回匹配书籍列表（含书况、定价、卖家信用积分）
  2. 系统在搜索匹配将同课程其他教材作为推荐附加到结果末尾 → 系统完成列表组装
  3. 买家（新生）在搜索匹配点击某条出售信息 → 系统在书籍发布加载该出售信息完整详情（定价、照片、新旧程度、卖家信息）
- Success End: 搜索结果状态为"已展示"，出售信息详情状态为"已展示"
- Exception Paths:
  - IF 搜索关键词无匹配结果: 系统将搜索结果状态设为"无结果"，向买家（新生）展示"未找到相关书籍，试试其他关键词"，买家（新生）可修改关键词重新搜索或浏览推荐的同课程教材。
  - IF 买家点击的出售信息在点击前已被下架: 系统将详情加载状态设为"不可用"，向买家（新生）展示"该书籍已下架"，买家（新生）可返回搜索结果查看其他出售信息。

### Scenario: 买家下单与卖家确认交易
- Trigger: 买家（新生）在出售信息详情页点击"立即购买"，出售信息状态为"已发布"时触发
- Flow:
  1. 买家（新生）在交易管理确认下单 → 系统在评价系统读取该买家当前信用积分
  2. 系统在评价系统检测该买家信用积分是否低于 60 分 → 系统记录检测结果
  3. 系统在交易管理根据检测结果将订单状态设为"待卖家确认"，记录下单时间 → 系统完成下单
  4. 系统在消息通知向卖家（毕业生）推送"有新订单待确认"提醒 → 系统记录通知发送时间
  5. 卖家（毕业生）在交易管理点击"确认交易" → 系统将订单状态设为"待交付"，记录确认时间
  6. 系统在消息通知向买家（新生）推送"卖家已确认，请安排交付"提醒 → 系统记录通知发送时间
- Success End: 订单状态为"待交付"
- Exception Paths:
  - IF 买家信用积分低于 60 分: 系统将本次操作状态设为"已拦截"，向买家（新生）展示"信用积分低于60分，已限制购买"，买家（新生）可关闭操作或查看积分详情。
  - IF 卖家点击"拒绝交易": 系统将订单状态设为"已取消"，向买家（新生）展示"卖家已拒绝本次交易"，买家（新生）可重新搜索其他出售信息。
  - IF 卖家超过 48 小时未响应: 系统将订单状态设为"已取消"，向买家（新生）展示"卖家未响应，订单已自动取消"，买家（新生）可重新搜索其他出售信息。

### Scenario: 买卖双方确认收货并互评
- Trigger: 买家（新生）在交易管理中对状态为"待交付"的订单点击"确认收货"时触发
- Flow:
  1. 买家（新生）在交易管理点击"确认收货" → 系统将订单状态设为"待评价"，记录完成时间
  2. 系统在消息通知向卖家（毕业生）推送"交易已完成，24小时内可互评"提醒 → 系统记录通知发送时间
  3. 系统在消息通知向买家（新生）推送"交易已完成，24小时内可互评"提醒 → 系统记录通知发送时间
  4. 买家（新生）在评价系统提交对卖家的评价（评分+文字） → 系统将买家评价状态设为"已提交"，记录提交时间
  5. 系统在评价系统根据买家提交的评分更新卖家（毕业生）信用积分 → 系统记录积分变更
  6. 卖家（毕业生）在评价系统提交对买家的评价（评分+文字） → 系统将卖家评价状态设为"已提交"，记录提交时间
  7. 系统在评价系统根据卖家提交的评分更新买家（新生）信用积分 → 系统记录积分变更
  8. 系统在交易管理将订单状态设为"已完成" → 系统完成交易闭环
- Success End: 订单状态为"已完成"，买家评价状态为"已提交"，卖家评价状态为"已提交"
- Exception Paths:
  - IF 买家反映书籍与描述严重不符: 系统将订单状态设为"纠纷处理中"，向平台管理员展示"有纠纷待处理"，平台管理员可在交易管理介入裁决。
  - IF 卖家对差评发起申诉: 系统将该差评申诉状态设为"申诉中"，向平台管理员展示"有差评申诉待处理"，平台管理员可在评价系统审核并维持或撤销差评。

### Scenario: 买卖双方私信沟通
- Trigger: 买家（新生）在出售信息详情页点击"联系卖家"，出售信息状态为"已发布"时触发
- Flow:
  1. 买家（新生）在消息通知发起私信并发送消息内容 → 系统创建会话，将会话状态设为"进行中"，将消息状态设为"已发送"
  2. 系统在消息通知向卖家（毕业生）推送新私信提醒 → 系统记录通知发送时间
  3. 卖家（毕业生）在消息通知查看并回复消息 → 系统将消息状态设为"已回复"，记录回复时间
- Success End: 会话状态为"进行中"，消息状态为"已回复"
- Exception Paths:
  - IF 消息含违规内容: 系统将消息状态设为"已拦截"，向买家（新生）展示"消息含违规内容，已被拦截"，买家（新生）可修改内容后重新发送。
  - IF 卖家超过 48 小时未回复: 系统将会话状态保持为"进行中"，向买家（新生）展示"卖家暂未回复，可继续等待或重新搜索其他书籍"，买家（新生）可选择继续等待或返回搜索结果。

### Scenario: 平台管理员处理违规内容与纠纷
- Trigger: 平台管理员进入书籍发布模块，发现存在疑似违规的出售信息或存在状态为"纠纷处理中"的订单时触发
- Flow:
  1. 平台管理员在书籍发布查看疑似违规出售信息详情 → 系统展示该出售信息完整内容及发布者信息
  2. 平台管理员在书籍发布点击"下架处理" → 系统将出售信息状态设为"已下架"，记录下架时间
  3. 系统在消息通知向对应卖家（毕业生）推送"出售信息因违规已被下架"提醒 → 系统记录通知发送时间
  4. 平台管理员在交易管理查看纠纷订单详情 → 系统展示纠纷订单信息及双方沟通记录
  5. 平台管理员在交易管理裁定纠纷结果 → 系统将订单状态按裁定结果设为"已完成"或"已取消"，记录裁定时间
  6. 系统在消息通知向买卖双方推送裁定结果通知 → 系统记录通知发送时间
- Success End: 出售信息状态为"已下架"，纠纷订单状态为"已完成"或"已取消"
- Exception Paths:
  - IF 卖家对下架决定提出异议: 系统将异议状态设为"处理中"，向平台管理员展示异议内容，平台管理员可在书籍发布重新审核后维持或恢复下架决定。
  - IF 被下架的出售信息关联有状态为"待卖家确认"或"待交付"的订单: 系统将该关联订单状态设为"已取消"，向买家（新生）展示"出售信息已下架，订单已取消"，买家（新生）可重新搜索其他出售信息。
