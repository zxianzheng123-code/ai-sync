# 产品逻辑技能优化 — 继续攻克剩余缺陷

## 背景

我对「技能/产品逻辑/SKILL.md」做了 50 轮 autoresearch 自动化优化实验。优化后的副本在 `技能/产品逻辑/autoresearch/optimized.md`。

优化过程中解决了 6 个评估维度（异常路径三要素、联动表质量、数值精确、Instructions忠实原文、全角色覆盖、联动行数），但仍有 6 个维度未达标。

## 评估方法

采用「拆分评估」消除自评偏差：
- **生产者 agent**：按 SKILL.md 为测试场景生成 product_logic.md，只产出不评价
- **评估者 agent**：独立的找茬审查员，读取产出和原始需求，逐条判定 pass/fail

测试用 5 个模拟场景（校服零售商城、社区团购、宠物寄养预约、线下门店会员、二手教材交易），位于 `技能/产品逻辑/autoresearch/scenarios/scenario_1~5.md`。

## 当前 6 个未达标的评估维度

### 1. 联动与 Flow 一致（通过率 1/5）— 最难

**问题**：Module Connections 表里写了"应该发生的联动"，但没有任何场景的 Flow Step 实现它。比如写了"订单已付款 → 库存管理标记待出库"，但没有 Flow 步骤写入"待出库"状态。

**根因**：agent 把 Module Connections 当"设计愿景"写，而非从 Flow 中提炼实际已发生的联动。

**修复方向**：Module Connections 应该是 Flow 的"摘要视图"——先写 Flow，再从 Flow 中提炼联动关系，而不是独立构造。

### 2. Flow 原子化（通过率 2/5）

**问题**：系统反应跨模块合并。典型例子：`店员提交消费 → 系统写入消费记录，更新积分余额，检测等级阈值`——一步跨了消费记录、积分管理、会员管理三个模块。

**根因**：模板说"同模块内可合并，跨模块须拆分"，但 agent 对"模块边界"判断模糊，常把紧密相关的操作视为"同一件事"。

**修复方向**：给出更明确的判断标准——如果→右侧涉及两个不同模块名，就必须拆成两步。可以加正例/反例。

### 3. Traceability 准确（通过率 3/5）

**问题**：Business Rules Traceability 引用了错误的 Trigger Rule。典型：照片规矩（每天至少上传1张）映射到逾期催交的 Trigger Rule（22:00未提交日志→催交），但那条 Trigger 对应的是"日志整体未提交"而非"照片缺失"。降级规矩映射到升级检测步骤。

**根因**：agent 找到一个"看起来相关"的 Trigger Rule 就填上去，没有核实该 Rule 的触发条件是否确实对应该条规矩。

**修复方向**：在 Traceability 写法中加入强制核实步骤——写完引用后，回读被引用的 Trigger Rule 触发条件，确认它和当前规矩是同一件事。

### 4. Success End 覆盖全目标（通过率 3/5）

**问题**：场景标题含多个动作目标（如"确认交易并互评"），但 Success End 只写了其中一个（订单状态），遗漏了另一个（评价状态）。

**根因**：agent 习惯用一个主状态代表整个场景成功，没有把标题拆解为多个目标逐一覆盖。

**修复方向**：在模板中增加"先分解场景标题为 N 个目标动词，再为每个目标写一个状态值"的步骤。

### 5. Trigger 无冗余（通过率 3/5）

**问题**：Trigger Rule 与某个 Exception Path 内容完全重复。比如 Exception Path 已写"截止时间到达且未成团 → 系统取消团购并退款"，Trigger Rule 又写了一遍同样的逻辑。

**根因**：agent 把 Exception Path 中描述的"如果X则Y"逻辑当作 Trigger Rule 重新描述了一遍，导致同一逻辑出现在两处。

**修复方向**：明确 Trigger Rules 的定位——只放"没有人操作触发的纯系统自动行为"（定时任务、阈值监控），Flow 和 Exception 中已描述的逻辑不再重复写入 Trigger Rules。

### 6. 前置状态可追溯（通过率 3/5）

**问题**：场景 Trigger 引用了某个状态（如"订单状态为已发货"），但整个文档中没有任何场景的 Flow 产生过该状态。还有聚合状态问题（如"所有消费者均已完成"无法从单个消费者场景追溯）。

**根因**：agent 创建场景时没有做全局状态链检查，导致有些中间状态缺少产生场景。

**修复方向**：加入一个"状态链闭环检查"步骤——列出所有 Trigger 中引用的前置状态，逐一在其他场景的 Flow/Success End 中找产生点，找不到就补充场景。

## 任务

请读取 `技能/产品逻辑/autoresearch/optimized.md`，针对上述 6 个缺陷逐一修改 SKILL.md 的工作流定义、模板格式、反模式列表和验证清单。

修改完成后，用拆分评估验证：
1. 为 `技能/产品逻辑/autoresearch/scenarios/` 下的 5 个场景各生成一份 product_logic.md（5 个独立生产者 agent）
2. 再用 5 个独立评估者 agent 审查产出，按上述 6 条标准逐条判定

目标：6 个维度的通过率均提升到 ≥ 4/5。
