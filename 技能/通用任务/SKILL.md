## 执行模式
Master Agent 负责调度和验收，不亲自干活。每个步骤启动 SubAgent 执行。

## 工作人员
- 从人设库中按任务需要动态选取（→ 技能/人设库/）

## Workflow

### Step 0：加载上下文（Master Agent）
执行 脚本/加载上下文.sh

### Step 1：任务分析
- 执行者：Master Agent
- 目标：判断任务复杂度——简单任务走快速通道（先告知用户再直接执行），复杂任务进入需求对齐
- 成功标准：复杂度判定明确；简单任务已获用户确认后执行；复杂任务进入Step 2
- 输入：用户任务描述
- 输出：复杂度判定 + 快速通道结果（如适用）

### Step 2：临时编排
- 执行者：Master Agent
- 目标：通过提案式对话将模糊需求转化为需求对齐卡（七维度：目标、实现边界、参考文档边界、约束、成功标志、素材、偏好），经用户确认后锁定；拆解执行计划
- 成功标准：七维度全部覆盖；用户已确认对齐卡；执行计划已拆解（子任务按依赖排序）
- 输入：用户任务描述
- 输出：需求对齐卡 + 执行计划

### Step 3：执行
- 执行者：从人设库动态选取的SubAgent（可并行）
- 目标：按执行计划逐项完成子任务
- 成功标准：每个子任务产出符合对齐卡要求；发现需求偏差时暂停确认（小偏差告知调整，大偏差回Step 2重新对齐）
- 输入：Step 2 的执行计划 + 对齐卡
- 输出：各子任务产出物

### Step 4：验证
- 如有 脚本/验证/ 下的脚本 → 自动执行
- 如有 作业指引/交付自查清单.md → 逐项检查
- 对照需求对齐卡逐项验收：
  - [ ] 对齐卡全维度覆盖
  - [ ] 成功标志已达成且可验证
  - [ ] 无遗留缺口（或用户已确认接受）
- 缺口可补 → 追加子任务回Step 3
- 理解偏差 → 回Step 2重新对齐
- 迭代超5轮 → 向用户汇报进度，由用户决定
- 全部通过才进入交付

### Step 5：交付审核（Master Agent）
输出成品，等用户确认。
