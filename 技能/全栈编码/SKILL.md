## 执行模式
Master Agent 负责调度和验收，不亲自干活。每个步骤启动 SubAgent 执行。

## 工作人员
- 全栈开发（→ 技能/人设库/全栈开发/人设.md）
- 工程审查（→ 技能/人设库/工程审查/人设.md）
- 测试工程师（→ 技能/人设库/测试工程师/人设.md）

## 五阶段概览

| 阶段 | 名称 | 核心产出 | 条件 |
|------|------|----------|------|
| Phase A | 技术选型与项目搭建 | `infra_spec.md` + 干净项目骨架 | 必做 |
| Phase B | 代码架构规划 | `design/code_blueprint.md` | 必做 |
| Phase C | 前端编码 | 生产级前端代码 | 必做 |
| Phase D | 后端编码 | 生产级后端代码 | 仅 full-stack |
| Phase E | AI 能力集成 | LLM 集成 + Prompt 工程 | 仅含 AI 功能时 |

## Workflow

### Step 0：加载上下文（Master Agent）
执行 脚本/加载上下文.sh，强制读取所有引用内容

### Step 1：技术方案
- 执行者：全栈开发（SubAgent）
- 目标：完成技术选型、项目搭建和代码架构规划（Phase A + Phase B）
- 成功标准：项目骨架可运行（dev server启动、浏览器可访问、无lint报错），代码蓝图完整
- 输入：`design/project_context.md` + `design/product_logic.md` + `design/design_spec.md` + `design/interaction_map.md`
- 输出：`infra_spec.md` + `design/code_blueprint.md` + 干净项目骨架
- Phase A 执行逻辑：
  1. 摄入业务上下文，提取关键约束
  2. 业务现状摸底（面向用户，每次只问1个问题）：数据现状/使用方式/运行环境/现有工具
  3. 技术选型内部决策：框架/语言/样式方案/数据层/包管理器（用户零技术决策）
  4. 目录结构设计
  5. 大白话向用户汇报技术方案（禁止技术黑话）
  6. 脚手架执行与自验证
  7. 输出 infra_spec.md
- Phase B 执行逻辑：
  1. 摄入所有上游产物
  2. 全局架构设计（共享组件 + 状态管理 + 路由表）
  3. full-stack 时：API 契约 + DB Schema 设计
  4. 逐页代码拆解（组件树 + Props/State + 事件映射 + 数据结构）
  5. 大白话向用户汇报
  6. 输出 code_blueprint.md
- 关键原则：
  - 用户零技术决策：禁止向用户抛技术选择题
  - AI 友好优先：多方案同等可行时选 AI 最容易维护的
  - 干净骨架：初始项目不含任何业务代码
  - Phase B 零代码产出：只产出文档

### Step 2：编码实现
- 执行者：全栈开发（SubAgent）
- 目标：逐页编码前端 + 全局集成 + 后端编码（Phase C + Phase D）
- 成功标准：所有页面可正常访问和交互，前后端联调通过（如适用）
- 输入：Step 1 产出的所有产物
- 输出：生产级前端代码 + 生产级后端代码（如适用）
- Phase C 前端编码：
  1. 逐页编码：每次只写 1 个页面，从零编写（禁止复制原型代码）
  2. 每页完成后自验证（dev server/视觉还原截图对比/状态切换/无lint报错）
  3. 每页用户试用确认后才进入下一页
  4. 所有页面完成后全局集成（路由拼装/导航/全局状态/跨页联动）
  5. E2E 验证 + 用户全局试用
- Phase D 后端编码（仅 full-stack）：
  1. 后端环境搭建
  2. 逐模块编码（一次只写 1 个 API 模块）
  3. mock→真实 API 替换前先告知用户确认
  4. 用户试用确认
  5. 全栈 E2E 验证
- Phase E AI 集成（仅含 AI 功能时）：
  1. 盘点 AI 模块
  2. AI 架构方案（模型选择/集成模式/Prompt策略/成本估算）
  3. 逐模块编码（Prompt模板 + API调用层 + 降级逻辑）
  4. 用户试用 AI 功能
- 关键原则：
  - 从零写代码：禁止从原型复制
  - CSS Variables Only：禁止硬编码色值
  - 逐页/逐模块推进：未通过不得开始下一个
  - Big Rewrite 原则：方向不对时先告知用户再推倒
  - API Key 绝不硬编码

### Step 3：代码审查
- 执行者：工程审查（SubAgent）
- 目标：审查代码质量、设计合规性、工程规范
- 成功标准：代码质量达标，设计还原准确，无工程规范违规
- 输入：Step 2 编码产物 + 上游设计文档
- 输出：审查报告（问题清单 + 改进建议）
- 审查维度：
  - 代码质量：可读性、可维护性、错误处理
  - 视觉还原：与 design_spec.md 截图对比
  - 设计合规：与 code_blueprint.md 一致性
  - 工程规范：目录结构、命名规范、依赖管理

### Step 4：测试
- 执行者：测试工程师（SubAgent）
- 目标：验证功能完整性和稳定性
- 成功标准：核心业务流程端到端跑通，无阻断性bug
- 输入：Step 2 编码产物
- 输出：测试报告（测试结果 + 发现的问题）
- 测试范围：
  - 核心功能冒烟测试
  - 跨页面状态一致性
  - 边界情况处理（空数据/大数据量/异常输入）
  - 数据持久化验证

### Step 5：验证
- 如有 脚本/验证/ 下的脚本 → 自动执行
- 如有 作业指引/交付自查清单.md → 逐项检查
- 全部通过才进入交付

### Step 6：交付审核（Master Agent）
输出成品，等用户确认

交付物：
- `infra_spec.md`（技术说明书）
- `design/code_blueprint.md`（代码蓝图）
- 生产级前端代码
- 生产级后端代码（仅 full-stack）
- 发现清单（如有，存入 `design/_discovery/`）
